# AGENTS.md — 開発規約

このリポジトリで作業する AI エージェント / 貢献者向けの規約。README（使い方）とは別に、
**コードの書き方・コメントの残し方・ビルド/テスト/セキュリティの約束事**をまとめる。

---

## コメントの残し方（意図中心）

コメントは「**何をしているか（what）**」ではなく「**なぜそうしているか（why＝意図・根拠）**」を書く。

- **仕様・決定・標準を参照する**: `spec D9`・`decisions D3`・`RFC 4777`・`SC30-3533`・`tn5250`・`ACS` 等、
  判断の出所を明記する（例: `// 証明書検証は既定 ON（spec: rejectUnauthorized:false は明示オプトイン）`）。
- **落とし穴・非自明な判断を明文化する**（例）:
  - `// Vue は未指定の Boolean prop を false にキャストするため withDefaults で true を明示する`
  - `// 先に状態を戻す: 応答 send が再入的に次データを流しても誤読しないため`
  - `// codec サブパスからブラウザ安全に import（root は pino/node 依存を巻き込むため不可）`
- **モジュール／コンポーネント横断の協調は「俯瞰コメント」を冒頭に置く**。個々の関数コメントだけだと
  読み手が全体像を掴めない箇所（例: `ScreenGrid`＝文字編集の真実 ↔ `EmulatorPane`＝フィールド間移動 が
  native caret を単一の真実として協調する、等）はファイル/セクション冒頭に数行の設計メモを残す。
- **深い設計判断は `decisions.md` に切り出し、コードから参照する**（二層構成）。コード側は要点＋参照、
  背景・代替案は decisions に。
- **関数の責務は JSDoc ヘッダ**（`/** … */`）で。
- **密度は箇所に応じて**: 複雑・非自明なプロトコル/状態機械/UI 同期は厚く（目安 10% 超）、スキーマ等で
  自己説明的な箇所（MCP ツール登録など）は薄くてよい。
- **実資格情報・秘密をコメント/ドキュメント/成果物に書かない**（後述のセキュリティ参照）。

## コーディング規約

- **ログは stderr のみ**（stdio MCP の stdout 汚染禁止）。`console.*` は lint で禁止し、
  `@as400web/core` の `log`（pino/stderr）を使う。
- **core のピュアロジック層は Node API 非依存**（I/O は `transport/` と `log.ts` に隔離。`node:*` import は
  この 2 箇所のみ許可）。
- ESM / Node ≥ 20。周辺コードのスタイル（逐次 ByteReader・型で分岐を閉じる・オプショナルは存在時のみ付与）に合わせる。

## UI デザインガイド（web-ui）

web-ui の見た目・振る舞いの規約は **[docs/UI-DESIGN.md](docs/UI-DESIGN.md)** にまとめている。
新規の画面・コンポーネントを足すときはこれに従う。要点:

- 配色は CSS 変数で一元化し light/dark/system の 3 テーマに追従（生色を避ける）。
- ボタンは設置面の系統に合わせる（トップバー `.theme-btn`＝固定高 28px / CRT ペイン＝`.fk` 意匠）。
- トグルは固定幅 span（`.tv`）でレイアウトシフトを防ぎ、グループに `margin-left:auto` は 1 つだけ。
- 情報表示はバックドロップ＋本体の共通ポップオーバー（`InfoPopover`）。外側/再クリックで閉じる。
- 種別表記は「5250端末」「プリンター」（「表示/エミュレーター」は使わない）。チップは青/緑/破線で識別。
- 表示モードは `view-card`/`view-list` の専用クラスで分岐（`.list.list` の重複クラス指定は使わない）。
- ヘッダーの機能ボタンはアクティブ（フォーカス中）ペインの内容に連動させる。
- 信頼設定（PDF 出力先・自動印刷・秘密）の UI は認証オフ or admin のときだけ露出・編集（サーバーも同ゲート）。

## ビルド・テスト（retro 反映）

- **ビルドに vue-tsc を含める**: `vite build` はテンプレートの型チェックをしないため、テンプレート型エラーが
  潜伏しうる。`npm run build -w @as400web/web-ui`（`vue-tsc -b && vite build`）で必ず型チェックを通す。
- **web-ui のテストはパッケージ dir から実行**する（`cd packages/web-ui && npx vitest run`）。
  リポジトリルートから実行すると Vite の vue plugin が適用されず `.vue` の解析に失敗する。
- **test 方針に「人が触る操作感」の実機観点を含める**: Tab/矢印/ホイール/複数行フィールド/属性下線/
  パスワード欄など。自動 E2E（Playwright）だけでは拾えない操作系を明示的に確認する。
- 変更ごとにユニット/コンポーネントテストを追加し回帰資産化する。

## アカウント・権限設計（指針）

認証・所有・信頼境界の 3 層で整理する。**設定や API を新しく足すときは必ずこの 3 層に当てはめて考える**。

### 1. 実行モード（認証の有無）
- 既定は**無認証**（ローカル単一運用）。`--users <file>` 指定で認証が有効になる。
- **認証オフ = 単一の信頼ユーザー**として扱う。保存先の区別は UI に出さない。
- 認証オンは role を持つ: **admin / user**。ブラウザは Cookie セッション、MCP/自動化は API トークン。

### 2. 保存先（サーバー設定 / 自分の設定）
- **サーバー設定** = `profiles.json`（ファイル管理。全員から見える。**信頼設定を持てる唯一の置き場**）
- **自分の設定** = `connections.json`（`owner` スコープ。作成者のみ。信頼設定は持てない）

**保存先は自動で決まり、UI では選ばせない。**「共有 / 個人」という所有概念と移動 API は廃止した
（可視性を所有形態から導出すると、必要だったのは「信頼設定を書けるか」だけなのに概念が二重になるため）。

| | 参照 | 作成・編集 | 信頼設定 |
|---|---|---|---|
| 認証オフ | 全件 | 可（サーバー設定へ） | 可 |
| admin | 全件 | 可（サーバー設定へ） | 可 |
| 一般ユーザー | サーバー設定=読取／自分の設定 | 自分の設定のみ | 不可 |

`passwordEnv`（環境変数参照）は **CI/CD 用の別軸**。保存先の議論と混ぜない。

### 3. 信頼境界（trusted config）
- **サーバー上のパス書き込み・コマンド実行・秘密**につながる設定（`printer.autoPdfDir` / `autoPrint` / `signon` 等）は
  **信頼設定**。受理してよいのは **canEditProfiles（認証オフ or admin ＋ ファイル由来）** のルートだけ。
- 一般ユーザー・未認証のブラウザ入力からは**受理も露出もしない**。個人（connections）には**スキーマに持たせない**
  （型で不可能にするのが最も強い）。
- 「値を持たない操作」（実行時の ON/OFF 等）は所有者に許してよい。**値そのものを変えさせない**のが要点。

### 4. 秘密の扱い
- パスワードは**平文で保存しない**。UI 設定分は **AES-256-GCM**（`signon.passwordEnc` / `secretEnc`＝同一形式）で保存し、
  master key は環境変数（`AS400_SECRET_KEY` / `.env`）。単一利用者は `--auto-secret-key` で自動生成。
- 運用者向けは `passwordEnv`（環境変数名の参照）。**平文 `password` はプロファイルで廃止**（読み込み時に明示エラー）。
- API/ブラウザには**平文も暗号文も返さない**（有無だけ `hasSecret` / `autoSignon` で示す）。ログにも値を出さない。

### 5. 実装規約（認可はサーバーで担保する）
- UI の出し分けは**補助**であって境界ではない。必ずサーバー側で閉じる。
  - `assertOwner(owner, user)`: 認証オフ=全通過 / admin=全許可 / owner 一致で許可。
  - `requireAdmin()`: admin 専用ルート（`/api/admin/*`・所有の移動）。
  - `canEditProfiles`: 認証オフ or admin ＋ 永続化可能（`--profiles` 由来）。
- **受理だけでなく露出も絞る**: 一覧は `listForUser(user)`、編集者限定フィールドは
  `listPublic({ includeSignon })` のように明示フラグで出し分ける（一般応答に信頼設定・秘密・他人の設定を載せない）。
- 破壊的操作（削除・移動）は所有チェックの後に行い、**失敗時に元が残る順序**にする（追加→削除）。

### 6. 追加時のチェックリスト
- [ ] その設定は**信頼設定か**（パス書込・コマンド実行・秘密に触るか）。触るなら「サーバー設定 × editor」限定にしたか。
- [ ] **受理（API 入力）と露出（API 応答）の両方**でスコープを絞ったか。
- [ ] 認証オフ / admin / 一般ユーザーの **3 パターンでテスト**したか（403/404・可視範囲）。
- [ ] UI は環境に応じて概念を隠しているか（認証オフに所有ラベル・所有選択を出さない等）。

## セキュリティ

- **秘密情報（`.env` / `*.local.json`）はコミットしない**（`.gitignore` 済み）。パスワードは環境変数
  （`passwordEnv`）で渡す。
- **`.aidev` 成果物（`decisions.md` 等）にも実資格情報を書かない**。ドメイン説明で例が必要なら伏字（`***`）や
  ダミー値を使う。
- **deliver / 公開前に secret scan**（パスワード/トークン等の検出）を行う。混入が判明したら履歴からも除去し、
  露出した資格情報は**変更（無効化）**する。
