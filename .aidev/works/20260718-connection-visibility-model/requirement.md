# 要件: 接続設定から「所有（共有/個人）」の概念を UI から廃止する

## 背景 / 課題

接続設定の UI は「**共有 / 個人**」という所有形態を軸に組み立てられており、
新規作成時の所有セレクタ・一覧のチップ・所有の移動（`settings-move.ts`）が載っている。

しかし整理し直すと、本来必要だった区別は所有ではなく次の1点だけだった。

- `autoPdfDir` / `autoPrint` は**サーバーのファイルシステムとプリンター**を指す信頼設定であり、
  一般ユーザーに開放できない。だから「ローカル単一利用、または admin」でのみ設定可能にする。

「全ユーザーが共有で使うセッション」を作ること自体に目的があったわけではない。
所有形態を可視性の代理にしていたため、UI に本来不要な概念が載り、意図が分かりにくくなっていた。

さらに「共有」という語は**2つの別物**を指しており、これが混乱を増幅していた。

1. **ファイル管理のサーバー設定**（`--profiles` で渡す `profiles.json`。信頼設定と CI/CD 用 `passwordEnv` の置き場）
2. **UI 上の「全員から見える」という可視性ラベル**

不要なのは 2。1 は自動出力の置き場であり MCP の起動オプションでもあるため存続させる。

## 目的 / ゴール

**UI から所有（共有/個人）の概念を無くす。** 接続設定は全て個人の持ち物として扱い、
`profiles.json` は「サーバー設定（ファイル管理）」として、名前を実態に合わせて存続させる。

## スコープ

### 対象
- UI の**所有セレクタ**（新規作成時の 共有/個人）を廃止。新規作成は常に個人接続。
- 一覧の**共有/個人チップ**を廃止し、ファイル由来の設定には「**サーバー設定**」表記を用いる
  （認証オフでは従来どおり表示しない）。
- **所有移動 API `settings-move.ts`（共有⇄個人）を廃止**（`POST /api/settings/move`）。
- `profiles.json` は存続。UI からの編集ゲート（認証オフ or admin かつ永続化可能）は**現状維持**。
- README / UI-DESIGN の表記を「共有」から「サーバー設定」へ改め、
  `passwordEnv` が **CI/CD 用の別軸**であることを明記する。

### 対象外
- **公開フラグの追加は行わない**（見送り）。マルチユーザーでは自動出力は admin 自身のセッションのみ。
- `profiles.json` の廃止・connections への統合。
- `connectionInputSchema` の `.strict()` 緩和（信頼設定は引き続き connections に持てない）。
- 認証方式そのものの変更。MCP `--profiles` の挙動変更。

## 機能要件

- 新規作成フォームに所有の選択肢が出ない（admin でも出ない）。作成先は常に connections。
- 一覧・カードに「共有 / 個人」チップが出ない。ファイル由来の設定には「サーバー設定」表記を出す
  （認証オンのときのみ。認証オフでは区別を見せない）。
- 編集画面に「共有にする / 個人にする」ボタンが無い。
- ファイル由来の設定は、認証オフ or admin のときのみ編集可（**現状の挙動を維持**）。
- 一般ユーザーからはファイル由来の設定が**見えて接続に使えるが、編集できない**（現状維持）。
- 自動 PDF / 印刷の設定欄は、従来どおりファイル由来の設定かつ編集可能なときだけ表示する。

## 非機能要件 / 制約

- 信頼設定（`autoPdfDir` / `autoPrint` / `pdfFontPath`）の**安全境界は不変**。
  ブラウザ入力からは受け付けず、`.strict()` による一律拒否も維持する。
- 既存データ（`connections.json` / `profiles.json`）は**移行なしでそのまま読める**こと。
- `POST /api/settings/move` の廃止は破壊的変更だが、UI 以外の利用者は想定していない。
