# 調査: 取り込みを SQL INSERT 経路へ

requirement の未確定事項（バッチ送信の手段 / `prepareUpload` の縮小範囲 / commit 制御）と、
**そもそも何を実装する必要があるのか**を、原典と実機で確かめる。

**原典照合は主エージェントが直読した**（protocol 2.6）。
出典は `archived/jtopenlite/com/ibm/jtopenlite/database/`。実機確認は PUB400。

## 調査の問い

- Q1: 結果セットを返さない文を実行する要求は何か。我々は何を持っていないか。
- Q2: 複数行をまとめて送る手段はあるか。原典はどうしているか。
- Q3: 値をどう渡すか（パラメータマーカー / リテラル）。日本語は壊れないか。
- Q4: 型ごとの変換を自前で持つ必要があるか。

## 判明した事実

### F1: `prepareAndExecute` = **要求 `0x180D`**。我々はこれを持っていない

原典 `JDBCStatement.executeUpdate`（`:442-484`）は `conn.prepareAndExecute(pea, null)` を呼ぶ。
`DatabaseConnection.sendPrepareAndExecuteRequest`（`:293-391`）の `writeHeader(length, 0x180D)`。

我々の `db-datastream.ts` に定義済みなのは:

| 名前 | 値 | 原典での送出元 | 我々の利用 |
|---|---|---|---|
| `execute` | `0x1805` | `sendExecuteRequest` | 未使用 |
| `executeImmediate` | `0x1806` | `sendExecuteImmediateRequest` | 未使用 |
| **`prepareAndExecute`** | **`0x180D`** | `sendPrepareAndExecuteRequest` | **未定義** |

→ 追加が要るのは **1 つの要求コードと、その応答（SQLCA）の解釈**だけ。
`query.ts` の `prepareAndOpen` が既に prepare 系の組み立てとSQLCA解釈を持っているので、その隣に置ける。

### F2: **SQL 文はワイヤ上 UTF-16（CCSID 13488）**。日本語はそのまま届く ★最重要

`query.ts:61` `const SQL_TEXT_CCSID = 13488;`、`sqlText()`（`:81-88`）が
`charCodeAt` を 2 バイトで書き出している。

**実機で実証した**:

```sql
SELECT '日本語テスト' AS JP,
       HEX(CAST('日本語' AS VARCHAR(20) CCSID 5035)) AS H
  FROM SYSIBM.SYSDUMMY1
```

```json
{"JP":"日本語テスト","H":"0E4562456648E70F"}
```

- 文中の日本語がそのまま往復した。
- `CAST(... AS ... CCSID 5035)` の結果 `0E4562456648E70F` は、
  **前作業で `UX''` 経由で仕込んだ基準行のバイト列と完全に一致**した
  （前作業 `research.md` F10: `0E 4562 4566 48E7 0F`）。

→ 前作業で日本語が `0x3F` に化けたのは **CL / RUNSQL 経路だけの問題**であり、
database サーバー経由では起きない。**ホストが対象列の CCSID へ正しく変換する。**

### F3: 原典に**バッチ挿入の実装は無い**。プロトコルには枠がある

- `JDBCStatement.executeBatch()`（`:315-318`）は **`NotImplementedException`**。
  jtopenlite は複数行挿入を実装していない。
- ただしプロトコル側には属性が揃っている（`DatabaseConnection.java`）:

  | 属性 | 符号 | 書式 |
  |---|---|---|
  | SQL パラメータマーカーデータ | `0x3811` | 長さ(4) ＋ CP(2) ＋ 本体 |
  | 同・拡張 | `0x381F` | 同上 |
  | **パラメータマーカー・ブロック指示子** | **`0x3814`** | 長さ(4)=8 ＋ CP(2) ＋ **値(2)** |

  `0x3814` が「1 回の execute に何行ぶんのマーカーデータが入っているか」を示す
  （＝ブロック挿入）。**ただし原典に使用例が無く、データの並べ方は未確認**。

→ **原典を写せる道が無い**ので、バッチ手段は自分で選ぶことになる（F4 と合わせて判断する）。

### F4: **リテラルを使えば型ごとの変換を自前で持たなくてよい**

パラメータマーカー方式を採ると、値を「マーカーデータ形式」でバイト列に詰める必要がある。
これは我々が `db-decode.ts` で持っている**復号の逆変換一式**（パック/ゾーン 10 進・日付時刻・
可変長・DBCS・NULL 指標）を新たに書くことを意味し、**DDM 経路で苦しんだのと同じ作業**になる。

一方、値を SQL 文のリテラルとして埋めれば:

- **型の解釈はホストが行う**。`VARCHAR` も日付時刻も `GRAPHIC` も、こちらは何も知らなくてよい
  （requirement の「型は全部」を、型ごとのコードなしで満たせる）。
- 文は UTF-16 で送られる（F2）ので**日本語も壊れない**。

代償と対処:

- **引用符のエスケープが要る**（`'` を `''` に二重化）。ここを誤ると値が壊れる／文が壊れる。
  → 単体テストで固められる範囲。
- **文の長さに上限がある**。長い文には拡張ステートメントテキスト（`ExtendedSQLStatementText`）が
  用意されている。1 バッチの行数はこの上限から決める（DDM の LL 上限と同じ考え方）。
- 数値・日付は**引用符で囲むか否かの判断**が要る（列型を見る）。
  → `SYSCOLUMNS` の型は既に取れる（`fetchColumnLayout`）。

→ requirement の「値は**パラメータとして渡す**」は、
**「リテラル連結の危険（エスケープ漏れ・文字化け）を避ける」**という意図で書いた。
F2 で文字化けの懸念は消え、エスケープは局所的な問題に落ちる。
**手段としてのパラメータマーカーは必須ではない**——この点は spec で判断する。

### F5: 読み取り専用の不変条件は保てる

`query.ts` の `prepareAndOpen` は `sqlStatementType = SELECT(0)` と
`openAttributes = 0x80` を固定で送っており、**結果セットを持たない文は describe で落ちる**。
新しい実行経路は別関数（別の要求コード `0x180D`）にするので、
`query` の性質は変わらない。`/api/host/sql` は読み取り専用のまま保てる。

### F6: 接続は使い回せる

SQL には `DbPool`（`poolKey(user, hostAuth)`）が既にあり、`/api/host/sql` が使っている。
DDM には無く、取り込みのたびに signon ＋ 446 の 2 接続を張っていた
（前作業の実測 11.6 秒のうち大半がこれ）。**取り込みが同じプールに乗れば、
2 回目以降の接続確立が消える**——DDM 経路との性能差は往復数だけでなくここでも出る。

## 影響範囲

- `packages/core/src/hostserver/db/db-datastream.ts`: 要求コード `0x180D` の追加。
- `packages/core/src/hostserver/db/`: 実行経路（新規。`query.ts` の隣）。
- `packages/core/src/hostserver/ddm/upload-prepare.ts`: **役割の縮小**。
  列突き合わせ・数値検証・NULL 検証は残り、CCSID・バイト長・レコード配置の検査は不要になる。
  置き場所も `ddm/` から移すのが自然。
- `packages/server/src/host-upload.ts`: 実行経路の差し替え（API の形は変えない）。
- `packages/web-ui`: **変更なしで動くはず**（応答の形を保てば）。

## 実現性 / リスク

- **実現可能**。追加が要るのは要求コード 1 つと実行関数 1 つ。型変換は不要（F4）。
- **リスク（中）**: リテラルのエスケープ。誤ると値が壊れるか文が壊れる。
  単体テストで固める（引用符・改行・NULL・数値/日付の引用要否）。
- **リスク（中）**: 1 文あたりの長さ上限。実測して 1 バッチの行数を決める必要がある。
- **リスク（小）**: `0x180D` の応答解釈。SQLCA は既に `parseSqlca` がある。
- **リスク（小）**: ブロック挿入（`0x3814`）は**使わない判断**なら不要。使うなら未知が残る。

## spec への申し送り

1. **実行経路は `prepareAndExecute`（`0x180D`）で新設**し、`query` には手を入れない（F5）。
2. **値の渡し方はリテラル方式を推す**（F4）。パラメータマーカーは型ごとの符号化一式を
   新規に書くことになり、DDM で苦しんだ作業の再演になる。
   **ただしエスケープを 1 か所に閉じ、単体テストで固めること**を条件とする。
3. **バッチは複数行 `VALUES (...),(...)`** を第一候補とする。
   ブロック指示子（`0x3814`）は原典に使用例が無く、データの並べ方が未確認のため採らない。
   1 バッチの行数は**文の長さ上限から決める**（実測して定数化）。
4. **`upload-prepare` を縮小して `ddm/` の外へ移す**。
   残すのは列突き合わせ・NULL 検証・数値として解釈できるかの検査。
5. **接続はプールに乗せる**（F6）。取り込みが `/api/host/sql` と同じ `DbPool` を使う。
6. **UI と API の応答形は変えない**。`committedRows` / `uncertainRange` の意味も維持する
   （バッチ単位で失敗したとき、その範囲は確定不明であることに変わりはない）。

### 残った未確定事項

- **1 文あたりの長さ上限**（実測が要る。1 バッチの行数を決める根拠になる）。
- **commit 制御**。SQL 経路では可能になるが requirement は対象外としている。
  「途中で失敗したら残る」を維持するなら、UI の部分完了表示も現状のままでよい。

## 追記: パラメータマーカー方式の検証（ユーザー指示）

「`Column.java` という原典があるのでパラメータマーカー式を試す」との判断を受け、
**必要な要素を洗い出した**。結論として **F4 の見立て（3,464 行相当が要る）は過大**だった。

### F7: マーカー形式は**結果列の形式と同一構造**。既存パーサーを流用できる ★

原典 `DatabaseConnection.java:2555-2600`（`cp == 0x3813` の分岐）が読む
「super extended parameter marker format」は、フィールドごとに

```
descLL(2) type(2) fieldLength(4) scale(2) precision(2) ccsid(2) parmType(1) ...
```

我々の `db-reply-ext.ts:74-79` が読んでいる super extended **data** format と**同じ並び**である
（`+2` 型 / `+4` 長さ / `+8` 位取り / `+10` 精度 / `+12` CCSID）。

→ **マーカー形式の解析は新規実装が不要**。既存の `parseSuperExtendedDataFormat` を
マーカー用に流用（または一般化）できる。

### F8: 必要な要求・符号は 3 つだけ

| 要素 | 符号 | 我々の状況 |
|---|---|---|
| マーカー形式の受領（応答） | **`0x3813`** | **既存パーサー流用可**（F7） |
| ディスクリプタ登録（要求） | **`0x1E00`**（`sendChangeDescriptorRequest`） | 未実装 |
| マーカーデータ（要求属性） | **`0x3811`** | 未実装 |
| 実行（要求） | `0x1805` | **定義済み**・未使用 |
| ブロック指示子（複数行） | `0x3814` | 未実装（使うなら） |

手順（原典 `JDBCPreparedStatement` のコンストラクタ ＋ `JDBCStatement.execute`）:

1. `prepareAndDescribe` で**マーカー形式**を受け取る（`0x3813`）
2. `changeDescriptor`（`0x1E00`）で形式を登録し**ディスクリプタハンドル**を得る
3. `execute`（`0x1805`）に `SQLParameterMarkerData`（`0x3811`）として値のバイト列を載せる

### F9: 書くべきは「既にある復号の逆」だけ

`Column.java` が 3,464 行あるのは、**符号化と復号の両方**に加えて
JDBC の全 getter/setter と Java 型間の相互変換を持つため。我々に要るのは:

- **符号化方向のみ**（値 → バイト列）。復号は `db-decode.ts` に実装済み
- **扱う型のみ**（`db-decode.ts` が復号している型に対応する範囲）

つまり `decodeRow` の逆関数。**ライセンス上、逐語移植はできない**（IPL 1.0。AGENTS.md）ので、
バイト配置という事実に基づいて書き起こす。復号側が既に実機で検証済みなので、
**対称性を根拠に書ける**のが DDM のときとの違い。

### 残る未知（スパイクで潰す）

- **マーカーデータの行バッファ配置**。結果行データと同じ「形式どおりの固定長並び ＋ NULL 指標」かどうか。
  対称なら `decodeRow` の逆で足りるが、**未確認**。
- **NULL の表し方**（結果側の指標マップと同じか）。
- **複数行を 1 回の execute に載せる方法**（`0x3814` の意味と行の並べ方）。原典に使用例が無い。

## spec への申し送り（更新）

**方式はパラメータマーカーで進める**（ユーザー判断）。ただし未知が残るため、
**最初のタスクを最小スパイクにする**:

> `CHAR` と `INTEGER` の 2 列だけの表に、パラメータマーカーで **1 行** INSERT し、
> SQL で読み返して一致するまでを実機で通す。

これが通れば、残りは型を足す作業に落ちる（形式解析・ディスクリプタ登録・実行の経路は共通）。
通らなければ行バッファの配置が想定と違うということなので、
**リテラル方式へ切り替える判断をその時点で行う**（F2 で実現性は確認済み）。

前作業の反省として、**土台の実現性を UI まで作ってから確かめない**。

## 追記: スパイクの実行結果（実機 PUB400）

`MARO1.ZZPM (C CHAR(10), N INTEGER)` を作り、`INSERT ... VALUES (?, ?)` で検証した。

### F10: **マーカー形式の取得は成功**（`0x3813` が返る）✅

`prepareAndDescribe`（`0x1803`）に
`ORS.parameterMarkerFormat | ORS.extendedColumnDescriptors` を立て、
`sqlStatementType = 1`（INSERT/UPDATE/DELETE）を送ったところ、応答に `0x3813` が 138 バイトで返った。

```
ヘッダー: consistencyToken=0 numFields=2 reserved=0 recordSize=14
列1: descLL=0x0030 type=0x01c5(453=CHAR) length=10 scale=10 precision=0 ccsid=0x0111(273)
列2: descLL=0x0030 type=0x01f1(497=INTEGER) length=4 scale=0 precision=9 ccsid=0
```

**確認できたこと**:

- **F7 の見立ては正しかった**——並びは我々の `parseSuperExtendedDataFormat` と同一
  （`+2` 型 / `+4` 長さ / `+8` 位取り / `+10` 精度 / `+12` CCSID）。既存パーサーを流用できる。
- **`recordSize = 14` = CHAR(10) + INTEGER(4)**。行バッファは固定長フィールドの単純な連結で、
  結果行と同じ構造。**`decodeRow` の逆で書ける**という前提が裏付けられた。
- `ORS.parameterMarkerFormat`（`0x00800000`）は**既に我々の定数に定義済み**で、そのまま効いた。

### F11: **実行はまだ通っていない**（`changeDescriptor` が要る）❌

マーカーデータ（`0x3811`）に 14 バイトの行を載せて実行を試したが、**行が挿入されない**。

| 試した経路 | 結果 |
|---|---|
| `execute`（`0x1805`）・指標なし | 応答は `0x3807`（SQLCA）が **len=0**。挿入されず |
| `execute`・指標あり（2 バイト×列数を前置） | 同上 |
| `prepareAndExecute`（`0x180D`）・指標なし / あり | 同上 |

テンプレートエラーも SQLCODE も返らず、**サーバーは要求を受け付けたが文を実行していない**。

**原因の見当**: 原典 `JDBCPreparedStatement` のコンストラクタは、形式を受け取った後に
**`changeDescriptor`（`0x1E00`）でマーカー形式を登録し、ディスクリプタハンドルを得ている**。
そのハンドルは `execute(attribs, parameterMarkerDescriptorHandle)` に渡される。
このスパイクは**その 1 段を飛ばしている**ため、サーバー側がマーカーデータを解釈できていない可能性が高い。

→ **未検証のまま**。次にやるのは `changeDescriptor` の実装（要求 `0x1E00`・
`ExtendedSQLParameterMarkerDataFormat` を載せる）と、ハンドルを execute へ渡す経路。

### スパイクの現時点の結論

- **形式まわりは想定どおりで、既存資産が効く**（F10）。
- **実行経路にもう 1 段（`changeDescriptor`）ある**ことが分かった（F11）。
  原典を読んだ時点で拾えた情報だが、スパイクで「無いと動かない」ことが実地で確認できた。
- **リテラル方式へ切り替える判断材料にはまだならない**。
  残りは 1 要求の実装であり、行き止まりではない。

## 追記: スパイク成功。パラメータマーカー方式で全要件を実証

### F12: 手順が確定した ✅

```
1. prepareAndDescribe (0x1803)
     ORS |= parameterMarkerFormat | extendedColumnDescriptors
     sqlStatementType = 1（INSERT/UPDATE/DELETE）
   → 応答 0x3813 にマーカー形式（既存パーサーと同一の並び）

2. changeDescriptor (0x1E00)
     template の parameterMarkerHandle にハンドル（RPB ハンドルとは別欄）
     受け取った形式を 0x381E で送り返して登録

3. execute (0x1805)
     template の parameterMarkerHandle に同じハンドル
     値を 0x381F（拡張マーカーデータ）で送る
```

⚠ **`parameterMarkerHandle` は template のオフセット 16 で、RPB ハンドル（14）とは別欄**。
取り違えると準備した文を見失い、**エラーにならないまま何も起きない**（実際に踏んだ）。
`buildDbTemplate` には最初からこの欄があった。

⚠ **診断には `ORS.messageId | firstLevelText` を立てる**。立てるまで応答は空の SQLCA だけで、
立てた瞬間に `S0043 Unable to use the parameter marker values provided.` が返って原因が分かった。
**沈黙したら、まずこのビットを立てること。**

### F13: マーカーデータの構造

```
[0..3]   一貫性トークン = 1
[4..7]   行数 N            ← ここを増やすだけで複数行になる
[8..9]   列数
[10..11] 指標サイズ = 2
[16..19] 行サイズ（形式の recordSize）
[20..]   指標: 2 バイト × 列数 × N（NULL は 0xFFFF）
[20+2*列数*N..] 行データ: 行サイズ × N
```

行内のフィールド位置は**形式の長さの累積**。値の詰め方:

| 型 | 形式が返す長さ | 詰め方 |
|---|---|---|
| `INTEGER`(497) / `BIGINT` 等 | 4 | ビッグエンディアン |
| `CHAR`(453) | 宣言長 | 対象 CCSID で符号化し **0x40 で右詰め** |
| `VARCHAR`(449) | 宣言長 + 2 | **2 バイトの長さ ＋ 本体** |

### F14: 実機で要件を全部満たした ✅

`MARO1.ZZPM2 (ID INTEGER NOT NULL, JP CHAR(20) CCSID 5035, VC VARCHAR(30), NL INTEGER)`

| 検証 | 結果 |
|---|---|
| **日本語**（CCSID 5035 の列） | ✅ `日本語テスト` が往復。`HEX` = `0E4562456648E74394438E43950F4040…`（SO/SI + DBCS） |
| **NULL**（指標 0xFFFF） | ✅ 3 列とも `null` で返る |
| **`VARCHAR`** | ✅ **`quote'test`（引用符入り）がそのまま**入った——エスケープ不要 |
| **100 行を 1 往復** | ✅ `100 rows inserted` / **288ms** |

**DDM 経路との比較**（前作業の実測）:

| | DDM | SQL マーカー |
|---|---|---|
| 100 行 | 11.6 秒（接続確立が支配的） | **288ms** |
| 日本語 | ❌ 表を開けない | ✅ |
| `VARCHAR` / 日付時刻 | ❌ 対象外 | ✅ |
| レコード配置 | 自前計算（仮説） | **サーバーが形式を教える** |

→ **F4 でリテラル方式を推した判断は撤回する。** パラメータマーカーのほうが優れている:
エスケープが要らず（引用符入りの値が素通り）、型・CCSID はサーバーが形式で教えてくれる。
ユーザーの「`Column.java` という原典があるので試す」という判断が正しかった。

### F15: 準備と実行の間に別の SQL を挟めない ★設計上の制約

100 行の試行が一度 `Prepared statement S1 not found.` で落ちた。
原因は**スパイクが間に `query()` を挟んだこと**で、`query.ts` は固定の文名で
同じ RPB に別の文を準備するため、こちらの文が失われる。

→ 実装では **「準備 → ディスクリプタ登録 → 実行」を他の SQL で割り込ませない**。
接続をプールから借りている間は占有する、という制約になる。

## spec への申し送り（最終）

1. **パラメータマーカー方式を採用**（F12〜F14 で実証）。リテラル方式は採らない。
2. **バッチはヘッダーの行数を増やすだけ**。`BLOCKED INSERT`（種別 7）も `0x3814` も不要だった。
   1 バッチの行数はマーカーデータ長の上限から決める（未実測。spec で扱う）。
3. **`request()` に `parameterMarkerHandle` を通す**（スパイクで core に追加済み）。
4. **値の符号化は型ごと**だが、必要なのは**形式が返す型に対する詰め方**だけで、
   `Column.java` 相当は要らない（F13 の表が実装範囲）。
5. **準備〜実行を割り込ませない**（F15）。
6. **診断ビット（`messageId | firstLevelText`）を実装にも入れる**。
   立てないとエラーが沈黙する。
