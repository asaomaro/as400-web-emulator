### **ブラウザ検証で見つけた不具合 2 件**（自動テストは通っていた）

1. **拒否理由が消えていた**。core の `UploadRejection` を SQL 経路向けに作り直したのに
   UI の `rejectionText` を追従させておらず、`value-invalid` が既定値に落ちて
   「取り込めませんでした」としか出なかった。
   **単体テストも古い形のままだったので緑だった**——テストが実装と一緒に古びる典型。
2. **「21844 件/往復」と表示していた**。バッチ容量であって実際の往復ではなく、
   2 行の取り込みで出ても意味が分からない。実測の往復数を出すよう直した。

どちらも**画面を見なければ気づけない**類で、T13 を独立タスクにした判断が効いた。

## 追加要望への対応（test 工程で差し戻し）

「SQL メニューと同じく、カラム名の maxWidth を設定したうえでコンテンツ幅で表示し、
D&D でリサイズもできるように」との要望を受け、coding へ差し戻して対応した。

実装は **`useColumnWidths` として切り出し、SQL ペインも新実装に載せ替えた**。
複製すると片方だけ直す事故になる——打ち切り基準（CSS の `max-width`）と手動指定は
噛み合っている必要があり、片側だけ直すと「広げても隠れた文字が見えない」に戻る。

### 実ブラウザで確認（Chrome）

| 確認項目 | 結果 |
|---|---|
| コンテンツ幅で表示 | ✅ `C_JP` 列が 91px（内容の実幅） |
| 長い値の打ち切り | ✅ 120 文字の値が 278px（`40ch` 相当） |
| 全文が読める | ✅ `title` に 120 文字 |
| ドラッグで拡張 | ✅ 278 → 428px（**打ち切り基準ごと動く**） |
| ダブルクリックで既定へ | ✅ 復帰する |
| JS エラー | ✅ なし |

**ここでも実ブラウザ確認が効いた**——最初の実装は `table { width: 100% }` のままで
列が引き伸ばされ、「コンテンツ幅」になっていなかった（`C_JP` 列が 618px）。
SQL ペインは `width: auto` で、この 1 行の違いが**上限も打ち切りも無意味にしていた**。
自動テストでは幅の計算をしていないので気づけない。

### あわせて直したもの

`placeholder` の `MARO1` / `TESTPF` を削除した。**特定環境のライブラリ名と検証用の表名が
製品 UI に埋め込まれており**、他の利用者には意味が無いうえ「この名前を入れるもの」と
誤解させる。入力規則は `title` に移した。
同種の記述がコメントにも 2 箇所（`object-list.ts` の `"MARO1"`、
`sqlLog.ts` の `pub400.com:9471`）あったので直した。

## 未検証として残すもの（正直な記録）

1. **MCP ツール経由の取り込み**——`uploadRows` の中身が変わっただけで入口は不変であり、
   HTTP 経路が実機で通っているため動作する見込みは高い。ただし
   **MCP サーバーの再起動が要るため確認していない**。前作業でも同じ項目が未検証のままである。
2. **部分失敗（`uncertainRange`）の実機再現**——途中で意図的に失敗させる手段が無く、
   単体テストでの確認に留まる。
3. **列幅の変更が SQL ペインで壊れていないこと**——共通化にあたり SQL ペインも
   新実装に載せ替えたが、**実ブラウザでの再確認はしていない**（既存の 449 件のテストは通過）。
4. **ファイル D&D（ドラッグ&ドロップ）の実ブラウザ確認**——ファイル選択ダイアログ経由は
   確認したが、ドロップ操作は Playwright で再現しにくく行っていない。
   コンポーネントテスト（`file-drag.test.ts`）で分割・タブ移動が誤発火しないことは固定済み。
5. **DDM 経路のコード**は残してあるが、**取り込みから外れたため実行されなくなった**。
   backlog の「レコード・レベル・アクセス」項目の資産として保持している。
