# 振り返り: 取り込みを SQL INSERT（パラメータマーカー）経路へ

## サマリ

CSV 取り込みの土台を DDM から database サーバー経由の INSERT に移した（PR #105・マージ済み）。
**前作業（PR #104）で未達だった日本語が解消**し、DDM で対象外だった型もすべて通るようになった。

| | DDM（PR #104） | SQL（本作業） |
|---|---|---|
| 100 行 | 11.6 秒 | **6.4 秒** |
| 日本語 | ❌ 表を開けない | ✅ |
| `VARCHAR` / 日付時刻 / `GRAPHIC` | ❌ 対象外 | ✅ |
| レコード配置 | 自前計算（「仮説に基づく実装」） | サーバーが形式を教える |

### メトリクス

| 指標 | 値 |
|---|---|
| リードタイム | 8,944 秒（約 2.5 時間） |
| 手戻り（同一 phase の再 start） | 2 |

工程別で目立つのは **test が 2,551 秒**（全体の 29%）。差し戻し 1 回（列幅の追加要望）と
実ブラウザ確認を含むためで、**実際の検証にいちばん時間を使った**形になっている。
research が 884 秒と長いのはスパイクを含むためで、これは後述のとおり投資として回収できている。

## うまくいった点

### 前提を疑う問いが土台を変えた（最大の成果）

ユーザーの「**仕組み上、DDM を挟む必要があるのですか？**」という一言が転換点だった。
ACS でデータ転送を実行し接続先ポートを実測したところ:

```
TCP  192.168.0.95:53202  185.113.5.134:8471  ESTABLISHED
```

**本家 ACS は database サーバー（8471）を使っており、DDM（446）ではなかった。**
この事実がなければ、日本語対応を諦めるか `CPF427D` の回避策を延々と探していた。

### スパイクを先に走らせた

「原典があるならパラメータマーカーを試す」というユーザーの判断を受け、
**spec を書く前に最小スパイクで実機到達性を確かめた**。結果:

- 当初「`Column.java` 3,464 行相当が要る」と見積もったが、**過大だった**。
  マーカー形式は結果列の形式と同一構造で、既存パーサーを流用できた。
- 未知（マーカーデータの行バッファ配置）が spec を書く前に潰れ、
  **設計が事実に基づいて固まった**。

前作業では土台の実現性を UI まで作ってから確かめて手戻りした。今回はその逆をやれた。

### 実機の応答をそのままテストデータにした

`marker-format.test.ts` は**実機が返した 138 バイトを固定データにしている**。
手で書き写して 1 バイト落としたので取り直した——写した時点で回帰資産の意味が薄れる。

### 「型を計算しない」を設計の軸にした

挿入先の型・長さ・CCSID はサーバーが教える。DDM 経路の `record-layout.ts` が自ら
「仮説に基づく実装」と断ってレコード配置を推測していたのに対し、**推測の余地を構造から消した**。

## 課題 / 手戻り

### 「失敗を検出できない」バグを 3 回作った

この作業で最も重い問題。同じ形の穴を繰り返し作っている。

| # | 内容 | 気づいた経路 |
|---|---|---|
| 1 | マーカーのハンドルを RPB ハンドル欄に入れた | 実機（沈黙。3 回空振り） |
| 2 | メッセージ ID の頭文字で成否を判定し `S0002 Function completed successfully.` を失敗と誤検知 | 実機 |
| 3 | `assertOk` が fail-open（template 未確認・SQLCA 不在を成功扱い）／`changeDescriptor` に SQLCA を要求せず／`DECIMAL` が枠を検証せず隣の列を上書き | **レビュー** |

**1 と 2 を踏んで「直したつもり」でいたのに、3 で同じ形が 3 つ残っていた。**

前段で防げたか: **防げた**。1 の時点で「この経路は書き込みで、巻き戻せない。
**成功と確認できたときだけ通す**」という原則を立てて全経路を見直すべきだった。
実際には目の前の 1 箇所を直して先へ進んだ。

さらに悪いことに、**失敗判定の経路に単体テストが 1 件も無かった**（レビュー指摘 S10）。
バグが集中していた場所が丸ごと未検証だった。`DbConnection.request` を差し替えれば
単体で固められるもので、後から 12 件書いたら**修正の正しさが即座に確認できた**。
最初から書いていれば 3 のうち少なくとも 2 つは実装時に気づけた。

### 自動テストが緑のまま UI が壊れていた

`UploadRejection` を SQL 経路向けに作り直したとき、UI の `rejectionText` を追従させ忘れ、
拒否理由が「取り込めませんでした」としか出なくなった。
**単体テストも古い形のままだったので緑だった**——テストが実装と一緒に古びる典型。

実ブラウザ確認（T13）で発見した。前作業の反省として独立タスクにした判断が効いた。

前段で防げたか: **防げた**。UI が core の型を手で写していたのが原因で、
**型を共有していればコンパイルエラーになった**（レビュー指摘 S8 で修正済み）。

### 列幅の追加要望で `width: 100%` のまま出した

「コンテンツ幅で表示」という要望に対し、SQL ペインの CSS を確認せずに実装し、
`table { width: 100% }` のままで列が引き伸ばされていた（`C_JP` が 618px）。
SQL ペインは `width: auto` で、**この 1 行の違いが上限も打ち切りも無意味にしていた**。

前段で防げたか: **防げた**。「SQL ペインと同じに」という要望なら、
まず SQL ペインの該当 CSS を全部読むべきだった。ロジック（composable）だけを見て CSS を見ていない。

### 見積もりの誤り

- 「`Column.java` 3,464 行相当が要る」→ **過大**。実際は符号化方向・扱う型のみで、
  形式解析は既存パーサーを流用できた。スパイクで判明。
- リテラル方式を推した判断は**撤回**した。パラメータマーカーのほうがエスケープ不要で優れていた。
  ユーザーの「原典があるので試す」という判断が正しかった。

## 改善提案

### 製品 / コード（→ issue 候補）

- **`packages/core` の `test/` が型検査対象外**（`tsconfig.json` の `include` が `src` のみ）。
  実際にレビューで「`MarkerFormat` の `raw` を欠くテストが通っている」ことが判明した。
  **他のテストにも同じ穴がある**ので優先度が高い。
- **`insertRows` の列名検証**。現状 `fetchColumnLayout`（ホスト由来）からしか来ないため
  攻撃経路ではないが、core の公開 API であり、区切り識別子を含む列名では不正な SQL になる。
- **`/api/host/sql` の読み取り専用テストが実質何も保証していない**。
  `CONFIG_ERROR` は資格情報の解決で出るため、INSERT を実行し始めても通る。
  意味のある固定には接続のモックが要る。
- **MCP ツール経由の取り込みが 2 作業続けて未検証**。サーバー再起動が要るだけなので、
  次のセッション開始時に確認するのが自然。
- **DDM 経路が取り込みから外れた**。`ddm/` は backlog の「レコード・レベル・アクセス」の
  資産として残しているが、**実行されるのはテストと検証スクリプトだけ**になった。
  この状態をいつまで保つか（あるいは backlog 項目を進めるか）を決める必要がある。

### PJ プロセス / 規約（→ AGENTS.md）

- **「書き込み経路は成功と確認できたときだけ通す」を規約にする。**
  この作業で 3 回同じ形のバグを作った。具体的には:
  - 応答の成否は**判定材料（SQLCA・戻りコード）を明示的に確認**する。
    「エラーが無い」を成功と読み替えない。
  - **判定できない応答は失敗として扱う**（巻き戻せない経路では特に）。
  - サーバーが返す**件数を送信件数と照合**する。
- **UI と core で型を共有する（手で写さない）。**
  判別可能ユニオンを UI 側で再定義すると、種類が増えたとき既定値に落ちて情報が消える。
  実際に起きた。共有すればコンパイルエラーになる。
- **「既存の X と同じに」という要望では、X の実装を全部読んでから書く。**
  ロジックだけ見て CSS を見なかったために `width` の 1 行を落とした。
- 既存の「実機検証を単体テストの代替にしない」に対し、**逆向きも明記する**:
  **単体テストを実ブラウザ確認の代替にしない**。この作業だけで、テストが緑のまま
  ブラウザで壊れていた事象が 2 件あった（前作業を含めると 3 件）。

### ハーネス自体（→ aidev-* への提案・適用は人間）

- **research に「前提の妥当性を問う」観点を明示する。**
  この作業の最大の成果は「DDM は本当に必要か」を問うたことだが、**問うたのはユーザーであって
  ハーネスではない**。research skill の観点リストは「未確定事項を事実で埋める」ことに寄っており、
  **「上流から引き継いだ前提そのものを疑う」項目が無い**。
  前作業では backlog の項目名がそのまま土台の選定になり、2 作業ぶん積み上がった。
  → 観点に「**この作業が前提としている選定（技術・経路・土台）は、誰がいつ何を根拠に決めたか**」
  を加える案。根拠が「前工程がそう書いていた」だけなら、それは検証していないということ。
- **スパイクを research の正規の道具として位置づける。**
  今回「spec を書く前に最小スパイクで到達性を確かめる」ことが決定的に効いたが、
  これはユーザーの指示で行った。research skill は調査＝読むこと・問うことに寄っており、
  **「動かして確かめる」が手順に無い**。
  → 「実現性が未確認」の項目に対しては**最小の実行可能な検証（スパイク）を推奨する**旨を追記する案。
  ただし「スパイクは捨てる」ことも同時に明記する（今回もスパイクは破棄し、
  得た事実だけを research.md に残した）。
- **test 工程に「実ブラウザ / 実機で触る」を独立項目として要求する。**
  前作業の retro で挙げた提案を今回自分で実践し、**2 件の不具合を捕まえた**（累計 3 件）。
  現在は skill の散文に埋もれており、tasks.md に自分で書かなければ飛ばせてしまう。
  → test skill の手順に「UI を持つ変更では実ブラウザ確認を必須項目として扱う」を明記する案。
- **`review.md` の指摘を次工程の入力にする仕組みが無い。**
  今回 should のうち 3 件を「backlog へ」と判断したが、**backlog へ実際に書き込む手順が
  ハーネスに無い**ため、retro の文章に残るだけになっている。
  → review で見送った指摘を `.aidev/backlog/` へ落とす導線（`aidev-util-propose` との接続）を
  検討する案。
