# 要件: CSV の取り込みを SQL INSERT 経路に移す

## 背景 / 課題

前作業（`20260720-csv-upload-ui`、PR #104）で CSV 取り込みを DDM 経由で実装し、
**SBCS の表までは動くところまで到達した**（実機で 100 行を 1 往復・SQL で読み返して一致）。
しかし実機検証で 2 つの壁に当たり、受け入れ基準「日本語を含む行が書ける」が未達のまま着地した。

| 壁 | 内容 |
|---|---|
| **DBCS CCSID の表を開けない** | 5035 / 930 の列を持つ表は `CPF427D` で `open` が失敗する。UFCB は原典と完全一致で実装漏れではない |
| **DDS 由来の様式名** | 物理ファイルの様式名がファイル名と違うと `CPF4135`。利用者に指定させる回避策しかない |

さらに**前提そのものが誤っていた**ことが判明した（前作業 `research.md` F11）。
ACS のデータ転送で接続先ポートを実測したところ:

```
TCP  192.168.0.95:53202  185.113.5.134:8471  ESTABLISHED
```

**本家 ACS は database サーバー（8471）を使っており、DDM（446）ではない。**
DDM を選んだ根拠は「`query` は SELECT 専用で `RUNSQL` は 1 文ずつ」——
これは **IBM i の制約ではなく我々の実装の制約**だった。
backlog の項目名（「`jtopenlite/ddm/` 相当のレコードレベルアクセス」）がそのまま土台の選定になり、
必要性が一度も検証されないまま 2 作業ぶん積み上がっていた。

## 目的 / ゴール

取り込みの土台を **SQL（database サーバー）経由の INSERT** に移し、
DDM 経路で構造的に解けなかった問題を解消する。

| DDM で詰まっていたもの | SQL INSERT では |
|---|---|
| DBCS CCSID の表が開けない | **ホストが文字コード変換を行う**ので問題にならない |
| レコード様式名 | **無関係** |
| `VARCHAR` / 日付時刻 / GRAPHIC が対象外 | 型はホストが解釈する |
| レコード配置が「仮説に基づく実装」 | **バイト配置を自前で計算しない** |

## スコープ

### 対象

1. **core**: database サーバーで**結果セットを返さない文を実行する経路**を追加する
   （`DB_CP.execute` / `executeImmediate` は定義済みで、使っていないだけ）。
2. **core**: 値をパラメータとして渡す仕組み（リテラル連結を避ける）。
3. **core/server**: 取り込みの実行経路を SQL INSERT に差し替える。
   `prepareUpload` の役割は縮小し、**型・CCSID・バイト長の検査は不要になる**。
   **対応型の制限（CHAR/DECIMAL/NUMERIC/SMALLINT/INTEGER/BIGINT のみ）は撤廃する**——
   `VARCHAR`・日付時刻・`GRAPHIC` を含め、表が受け付ける型はすべて通す。
4. **検証**: 日本語を含む取り込みが実機で往復すること。

### 対象外

- **DDM 実装の削除**。core に残す（backlog「レコード・レベル・アクセス」の資産。
  バッチ書き込みと列単位 CCSID は実機で実証済み）。
- **取得側（ダウンロード）の変更**。既に SQL 経路で動いている。
- **UI の作り直し**。データ転送ペインはそのまま使う。
- **更新・削除（UPDATE / DELETE）**。取り込み＝追記のみを維持する。
- **トランザクション制御**。現状と同じく「途中で失敗したら書けた分は残る」を受け入れる
  （ただし SQL 経路では commit 制御が可能になるため、**将来の選択肢としては開く**）。

## 機能要件

- CSV を渡すと、指定した表に行が追加される。**取り込み先は SQL から見える表**とする。
- **日本語を含む行が書ける**（前作業の未達項目）。
- 値は**リテラル連結ではなくパラメータとして渡す**。
  CL 経由の `RUNSQL` で日本語が 0x3F に化けた事象（前作業 F9）を繰り返さない。
- 列名の突き合わせ・数値変換の誤りは、**書き込みを始める前に**行番号と列名を添えて拒否する
  （前作業の `prepareUpload` の振る舞いを維持する）。
- 途中失敗時に「**どこまで確定したか**」を返す（前作業と同じく、言い切れない範囲は不明として返す）。
- MCP ツールからも同じ取り込みができる。
- 既存のデータ転送ペインがそのまま動く（UI から見た振る舞いは変わらないか、良くなる）。

## 非機能要件 / 制約

- **性能**: 1 行 1 文では往復が行数に比例する。**複数行をまとめて送る**こと
  （複数行 `VALUES` かパラメータの繰り返し実行か、手段は spec で決める）。
  DDM 経路の実測（100 行 1 往復・全体 11.6 秒、うち大半が接続確立）を**下回らない**こと。
- **接続**: SQL には既に `DbPool` がある。DDM には無かった。使い回しの恩恵を受けられる。
- **セキュリティ**: `host-sql.ts` が明文化していた「`/api/host/sql` は構造的に読み取り専用」は
  **維持する**。更新経路は別関数・別ルートに閉じ、`query` の性質を変えない。
  認可は既存どおり「接続を持つユーザーなら誰でも」（書ける範囲は IBM i の権限が決める）。
- **移植の作法**（AGENTS.md）: 原典（jtopenlite の `DatabaseConnection` / `JDBCStatement`）を
  **直読してから**設計する。逐語移植はせず、参照コメントでクラス・メソッド名を明示する。

- [ ] **日本語を含む行が書け、読み返して一致する**（前作業の未達項目を解消）。

- [ ] 数値でない値・列名の不一致は、**1 行も書かずに**行番号と列名つきで拒否される。
- [ ] 100 行の取り込みが DDM 経路の実測（全体 11.6 秒）を下回る。
- [ ] MCP ツールから同じ取り込みができる。
- [ ] `/api/host/sql` が読み取り専用のままであることを、テストで固定する。
- [ ] 既存テストが通り、追加分の自動テストがある。

## 未確定事項 / 確認したいこと

- **複数行をまとめて送る手段**。複数行 `VALUES (...),(...)` か、
  パラメータ付き文の繰り返し実行（ブロック実行）か。原典の確認が要る。
- **`prepareUpload` をどこまで残すか**。列突き合わせと数値検証は要るが、
  CCSID・バイト長・レコード配置の検査は不要になる。
- **commit 制御を入れるか**。SQL 経路では可能になるが、対象外としている
  （「途中で失敗したら残る」という現在の仕様を変えると UI の部分完了表示も変わる）。

> **バッチ送信の手段**と**型の対応範囲**は、原典と実機で確かめるべき事項である。
