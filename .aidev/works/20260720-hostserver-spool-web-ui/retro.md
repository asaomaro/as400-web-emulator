# 振り返り: pull 型スプール取得の Web UI 対応

## サマリ

pull 型スプールを Web UI に繋いだ（PR #106）。機能としては完遂し、実機で動作を確認した。

一方この作業の本質的な収穫は**機能ではなく失敗の型**にある。
**「確かめずに書く」を 3 回**やり、そのうち 1 回は**誤りを直している最中に同じ誤りを犯した**。
最後の 1 件はハーネスのどのゲートも捕まえられず、**ユーザーの問い掛けで初めて露見した**。

## うまくいった点

### 1. research が requirement の誤りを 2 件、着手前に潰した

- 「監査は既存ルートの流儀に合わせる（`withAudit` 相当を欠かさない）」→ **両立不能**だった。
  既存 HTTP ルートに監査は存在しない
- 「CCSID は接続設定から引く」→ 既存 `ccsid` の流用は**前作業で明示的に否定済み**だった

どちらも実装後に気づけば設計からやり直しになる。**任意工程を挟んだ判断は正しかった**。

### 2. plan の R1 で退避策を先に決めてあったので、差し戻しが安かった

`total` が使えないと判明した時点で「`max + 1` 方式へ退避する。**この退避は共有関数の内部に閉じるため
HTTP/UI の契約は変えずに済む**」と plan に書いてあった。
おかげで **sent_back から test 合格まで 6 分**で済んだ。

**リスクを挙げるだけでなく「外れたらどうするか」まで書いておく**のが効いた。

### 3. 「ガードを外して落ちるか確認する」手順

F1 の反省から、競合状態の回帰テストは**ガードを一時的に外して実際に落ちること**を確認した。
2 件とも期待どおり失敗し、空振りでないと分かった。
さらにこの過程で **flaky を 1 件発見**した（disabled のボタンを押しても何も起きず素通りする）。

### 4. レビューのサブエージェント委譲

自分が書いたコードを自分で読むと、意図が先に立って欠陥が見えない。
別視点に読ませたことで、**自分で作り込んだ競合状態 2 件**を含む should 4 件が出た。

## 課題 / 手戻り

### C1【最重要】「確かめずに書く」を 3 回、しかも訂正中に再発した

| # | 内容 | 捕まえた場所 |
|---|---|---|
| 1 | 「`LIST_INFO.total` は定数が定義済み＝想定内の拡張」 | **test（実機検証）** |
| 2 | 「`assertMax` が要るのは MCP が zod を通らないから」 | **review** |
| 3 | 「`total` が総数でないのは非同期構築で未確定だから」 | **どのゲートも捕まえられず。ユーザーの問いで露見** |

3 つは同じ形をしている——**借り物の根拠・状況証拠を、確かめた事実として書いた**。

- #1: 定数が定義されていることは、その値が期待どおりであることを何も保証しない
- #2: `host-upload.ts` の文言を**根拠ごと引き写した**。実際は `registerTool` が `inputSchema` を検証する
- #3: **#1 を直している最中に犯した。** 辻褄が合う説明を思いつき、辻褄が合うことを裏付けと錯覚した

**#3 が最も重い。** 「原因を突き止めた」と報告しながら、その原因説明自体が未検証だった。
実測すると `total` は `min(max, 実件数)`＝**構築されたリストの件数**で、完了指示子は `max=1` でも
`C(complete)` を返す。「未確定」ではなかった。

#### なぜ #3 だけゲートを素通りしたか

- test は**振る舞い**を検証する → `max + 1` 方式は正しく動くので通る
- review は**コード**を読む → コメントは動作に影響しないので指摘されない
- **「もっともらしいが未検証の因果説明」を検査する段が無い**

これはハーネスの穴である（改善提案 H1）。

### C2 単体テストが「実機では起こらない入力」を作って誤りを通した

`spool-list.test.ts` は `listInfo(1234, 1)` のように **`total ≠ returned` の応答を人工的に作って**いた。
実機ではこの状態が発生しないため、**テストは緑のまま前提の誤りを通した**。

モックは自由に作れる分、**実装の思い込みをそのまま固定してしまう**。
AGENTS.md「実機検証を単体テストの代替にしない」の、具体的な失敗例が 1 つ増えた。

### C3 自分で競合状態を 2 件作り込んだ

`SpoolPane` で応答の追い越し対策を入れていなかった。
「見出しは B なのに本文は A」を**エラーも出さずに**見せる状態で、帳票ビューアとしては実害がある。
`HostListPane.vue:139-146` に同種の危険を警告するコメントがあったのに、参照実装として読んでいながら
**そこが実際には防げていないこと**にも気づいていなかった。

### C4 記録の穴

`tasks.md` の **T11（実機確認）が未チェックのまま**。test 工程で実施したが、
チェックを戻す手順が無く落ちた。`tasks_done=10 / tasks_planned=11` の差はこれに由来する。

## 改善提案

### 製品 / コード（→ issue 候補）

- **P1【優先】既存の一覧 API が打ち切りを黙って隠している。**
  `host-lists.ts:122` は `{ items }` だけを返し、jobs / objects / users は **200 件で黙って切れる**。
  本作業で F1 として戦った偽陰性と**まったく同じ種類**のものが、既に出荷済みのコードに残っている。
  `truncated` を返して UI に断らせる（本作業の `host-spools.ts` と同じ形にできる）。
- **P2 `host-upload.ts:96-98` の理由が誤り。**
  「MCP は zod を通らない」は成り立たない（`registerTool` は `inputSchema` を検証する）。
  検査自体は残す価値があるので、**理由だけを正しく書き直す**（本作業では新規コード側のみ是正した）。
- **P3 `.error` の生色 `#c62828` が 5 ペインに散在。**
  UI-DESIGN は生色を避けよと定めるが、`HostListPane` / `SqlPane` / `TransferPane` / `AdminPane` /
  `SpoolPane` が同じ値を直書きしている。CSS 変数に寄せる。
- **P4 DBCS スプールの PDF が未検証。**
  実機で使えたスプールが全て SBCS だった。日本語を含むスプールを作って目視確認する。
- **P5 HTTP ルートに監査が無い。**
  `host-lists` / `host-sql` / `host-upload` / `host-spools` のいずれも `withAudit` を通らない。
  MCP と WS だけが監査されている非対称を、意図なのか漏れなのか決める。

### PJ プロセス / 規約（→ AGENTS.md）

- **A1「原典を直読してから設計する」に、因果説明への適用を明記する。**
  現行の規約は実装（バイト配置・定数・アルゴリズム）を対象に読めるが、
  今回落ちたのは**「なぜそうなるか」の説明**だった。追補案:

  > **未確認の因果を断定形で書かない。** 「〜のため」「〜だから」と原因を書くときは、
  > 確かめた事実か推測かを区別する。推測なら「未確認」と明示し、確かめる手段を添える。
  > **辻褄が合うことは裏付けではない。**

- **A2 コメントの根拠を他ファイルから引き写さない。**
  今回 `host-upload.ts` の理由をそのまま持ってきて、その理由が成り立たない場所に貼った。追補案:

  > 既存コメントの**書き方**は真似てよいが、**根拠は真似ない**。
  > 引き写す前に、その根拠が自分の文脈でも成り立つか確かめる。

- **A3 モックは観測した応答だけを模す。**
  C2 の直接の対策。追補案:

  > プロトコル層のテストで、**実機で観測していない応答の組み合わせを作らない**。
  > 作ると「実機では起こらない状態」を前提にしたテストが緑になり、前提の誤りを通す。

- **A4 回帰テストは「落ちること」を確認してから採用する。**
  本作業で途中導入して有効だった手順を規約化する。追補案:

  > 欠陥に対する回帰テストを書いたら、**修正を一時的に外して実際に失敗することを確認する**。
  > 落ちないテストは、その欠陥を守っていない。

### ハーネス自体（→ `aidev-*` への提案・適用は人間）

- **H1【最重要】review 工程に「主張の検証」観点を追加する。**
  C1 のとおり、test は振る舞いを・review はコードを見るが、
  **コメントや成果物に書かれた因果説明が検証済みかを問う段が無い**。
  `aidev-60-review` の点検観点に追加する案:

  > **主張の検証**: コード・成果物に書かれた「なぜ」のうち、
  > **断定形で書かれた因果**を拾い、根拠が示されているか／推測なら推測と明示されているかを見る。
  > 他所からの引き写しは、その文脈で成り立つかを確かめる。

  実際、今回の #2 は review が捕まえた（理由の誤りとして）が、それは
  委譲先が偶々そこまで踏み込んだためで、**観点として定義されていたわけではない**。
  観点化すれば #3 も捕まえられた可能性がある。

- **H2 test 工程に「否定された前提の記録」を求める。**
  今回 F1 で spec 方針5 が否定されたとき、`spec.md` 本文は**旧前提のまま残っていた**
  （review 工程で気づいて直した）。
  `aidev-50-test` の差し戻し手順に、**否定された前提を spec に反映すること**を明記する案。
  decisions.md への記録だけでは、spec を読んだ人が古い前提を信じる。

- **H3 差し戻しから戻ったとき、tasks.md のチェックを戻す手順が無い。**
  C4 の原因。`aidev-50-test` / `aidev-60-review` の差し戻し節に、
  「差し戻し対象のタスクのチェックを外す」または「test 工程で実施したタスクは test 側で
  チェックする」を明記する。`tasks_done` と `tasks_planned` の差が
  **記録漏れなのか未実施なのか判別できない**のは、retro の分析を弱める。

- **H4（小）deliver 後の訂正を記録する場が無い。**
  本作業では deliver 承認後に `4b306b7`（`LIST_INFO.total` の実測）を追加した。
  `metrics.yml` は deliver approved で終わっており、この訂正は痕跡に残らない。
  retro で拾えたから良かったが、**deliver 後の追い足しを記録する術**があるとよい。
