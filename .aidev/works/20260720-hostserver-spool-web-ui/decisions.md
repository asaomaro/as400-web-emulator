# 決定記録

## D1: `spoolCcsid` を core の `ConnectOptions`（session.ts）にも足した

- 背景: spec 方針2 は「`config-resolver` が解決済み `ConnectOptions` へ転記し、ルートがそれを読む」と
  決めていたが、その受け皿である `ConnectOptions` は core 側（`packages/core/src/session/session.ts:21`）に
  あり、spec の「変更するファイル」一覧に挙げていなかった。
- 決定: `ConnectOptions` に `spoolCcsid?: number` を追加した。
- 理由 / 代替案: `ConnectOptions` は 5250 セッション専用ではなく、**ホストサーバー接続でも使われる
  共通の接続設定**（`hostAuthFrom(opts)` がここから host/user/password/tls を取る）。
  既存の `ccsid` も同じ場所にあるため、並べるのが自然。
  代替案として「resolver の戻りに別フィールドを足す」も考えたが、`resolve().connect` の型が
  `ConnectOptions` である以上、結局どこかで型を広げる必要があり、遠回りになる。
  5250 セッションはこの項目を読まないので、既存の挙動には影響しない（その旨を JSDoc に明記した）。
- 影響: core の公開型が 1 項目増える（後方互換。optional）。spec 方針2 の実現方法の具体化であって、
  方針そのものの変更ではない。

## D2: `total` が実件数を下回った場合に下限を合わせる

- 背景: `listSpooledFiles` は受信バッファが足りないと `returned` 件すべてを詰められず、
  途中で `break` する（`spool-list.ts` の既存ロジック）。この場合 `total < entries.length` が
  起こり得る。
- 決定: `Math.max(total, entries.length)` を返す。
- 理由 / 代替案: そのまま返すと `truncated = total > entries.length` が**偽**になり、
  「打ち切られているのに打ち切られていないと表示する」偽陰性になる。打ち切りの見落としは
  利用者が全件見たと誤解する実害があるため、安全側（実件数を下限）に倒した。
  代替案「バッファ不足を例外にする」は既存の寛容な挙動を変えるので採らなかった。
- 影響: なし（新規フィールドの計算のみ）。テストで固定した。

## D4: `LIST_INFO.total` を打ち切り判定に使うのをやめ、`max + 1` 方式へ退避した

- 背景: spec 方針5 は「QGYOLSPL のリスト情報 offset 0 が一致総数を返す」という前提で
  core を拡張し、UI に「1234 件中 100 件」と出す設計にしていた。
  **test 工程の実機検証でこの前提が否定された**（test-result.md F1）。
  スプール 3 件の状態で max=1 なら total も 1、max=2 なら 2——`total` は常に `returned` と同値だった。
  オープンリスト API は非同期に構築するため、要求分しか作られていない時点では総数が確定しないと思われる。
- 決定: plan.md R1 に定めていた退避策を発動した。
  - core の `SpoolListResult` を撤回し、`listSpooledFiles` は `SpoolEntry[]` に戻した
  - `listSpools`（server）で **`max + 1` 件要求し `> max` なら打ち切り**と判定する
  - HTTP 応答から `total` を落とし、UI 文言を「先頭 N 件のみ表示しています」に変えた
  - `LIST_INFO` に**この値を使ってはいけない理由**をコメントで残した
    （次に同じフィールドを見た人が同じ推測をしないように）
- 理由 / 代替案: 「総件数を出せる」は利便性の向上に過ぎないが、`truncated` が常に false になるのは
  **利用者が全件見たと誤解する偽陰性**で、害の大きさが違う。正確さを取れない以上、
  正しく「切った」と言える方を採る。
  代替案「QGYGTLE でハンドルを使った継続取得なら総数が確定するかもしれない」は、
  原典（JTOpen）の確認からやり直しになりスコープを大きく超えるため見送った（backlog 候補）。
- 影響: **spec 方針5 は無効**。requirement の受け入れ基準「打ち切りが画面に示される」は
  「先頭 N 件のみ」という形で満たす（総件数は示せない）。D1・D2 のうち D2（`Math.max` の下限合わせ）は
  対象コードごと撤回した。core の変更は `ConnectOptions.spoolCcsid`（D1）のみに縮小した。
- 教訓: **「定数が定義済み＝想定内の拡張」は根拠にならなかった。**
  `LIST_INFO.total` は定義されてはいたが一度も読まれたことがなく、
  「定義されている」ことは「期待どおり動く」ことを何ら保証しない。
  さらに、この誤りを**単体テストが緑のまま通した**——テストが `total ≠ returned` という
  実機では起こらない応答を人工的に作っていたため。AGENTS.md「実機検証を単体テストの代替にしない」
  がそのまま当てはまる。

## D3: T11（実機確認）を coding では実施せず test 工程へ送る

- 背景: `tasks.md` の T11 は PUB400 での実機確認。plan では「検証」節に置いていた。
- 決定: coding 工程では T1〜T10（実装とテスト）までとし、T11 は test 工程で行う。
- 理由: 実機確認は外部システム（PUB400）への接続と、検証用 OUTQ / スプールの作成を伴う。
  工程としては検証であり、coding の完了条件（spec の受け入れ基準を満たすコードが書けている）
  とは別。**R1（`total` の実機挙動が未検証）は依然として残っており、これを
  「実装したから通るはず」で閉じない**（AGENTS.md「実機検証を単体テストの代替にしない」）。
- 影響: coding 承認時点で受け入れ基準のうち実機依存のものは未検証。test 工程で解消する。
