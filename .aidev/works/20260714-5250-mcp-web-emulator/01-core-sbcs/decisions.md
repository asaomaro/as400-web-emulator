# 決定記録（01-core-sbcs）

## D1: T7 の trace 採取はサインオン画面のみとし、メニュー遷移 trace は T13 へ移す

- 背景: T7 実施時点で PUB400 の資格情報（PUB400_USER/PASSWORD）が未設定。サインオン画面の受信までは
  資格情報不要で採取できるが、メニュー遷移はサインオンが必要。
- 決定: T7 では `pub400-signon.jsonl`（ネゴシエーション＋サインオン画面 884 バイトレコード）のみ採取。
  サインオン後のメニュー遷移・オペレータエラー行の trace は、資格情報が揃う T13（実機検証）で採取する
  （T13 は元々その内容を含む）。パーサ（T10）・セッション（T12）の開発はサインオン画面 trace で十分駆動できる。
- 理由 / 代替案: ダミー資格情報での送信は PUB400 の利用規約・アカウントロックのリスクがあり不採用。
- 影響: T12 のリプレイ E2E は「サインオン画面表示＋フィールド入力＋送信バイト列検証」までを対象にし、
  「メニュー画面の受信」はモック trace（手書きの応答レコード）で補う。実データでの最終確認は T13。

## D3: PUB400 は画面フィールド方式のサインオンを受け付けず、RFC 4777 NEW-ENVIRON 自動サインオンが必要

- 背景: T13 で、サインオン画面のユーザー/パスワードフィールドに入力して Enter 送信する方式（spec D8 の
  「画面フィールド自動入力方式」）を実機検証したところ、送信バイトが `0810@MARO` の標準 EBCDIC エンコードと
  **完全に一致（probe で機械照合済み）**するにもかかわらず、ホストが `CPF1120`（ユーザー不存在/パスワード不一致）で
  拒否する。ユーザーは同一資格情報で ACS からサインオン成功（MAIN Menu 到達）を確認済み。
- 調査: ACS/tn5250j の実装（tnvt.java `negNewEnvironment`）を確認したところ、パスワード設定時は
  **telnet ネゴシエーションの NEW-ENVIRON で `VAR USER`＋`USERVAR IBMRSEED`（ESC＋0x00×8 のゼロシード）＋
  `USERVAR IBMSUBSPW`（平文パスワード）を送る RFC 4777 自動サインオン**を行い、サインオン画面自体をスキップして
  ログインする。ゼロシードのときパスワードは非暗号化で送られる。これが PUB400 で機能する経路。
- 決定: 画面フィールド方式に加え、**RFC 4777 NEW-ENVIRON 自動サインオン（ゼロシード・非暗号化）を TelnetLayer に実装**する。
  `ConnectOptions.user` / `password` を渡すと、NEW-ENVIRON 応答に上記 3 変数を載せる。
- 理由 / 代替案: 画面フィールド方式は tn5250j でも「パスワード未設定時のみ」の経路で、PUB400 の
  カスタムサインオンとは相性が悪い（実機で拒否を確認）。全実用クライアント（ACS/tn5250j）が採る NEW-ENVIRON
  方式が事実上の標準。暗号化（非ゼロシード）は引き続き将来拡張とする。
- 影響: **spec D8 を修正**（画面フィールド方式 → RFC 4777 NEW-ENVIRON 自動サインオンを第一とする。画面方式は
  フォールバック）。subtask 01 のスコープに telnet 層の自動サインオン送信を追加（TelnetLayer は既に NEW-ENVIRON
  DEVNAME を送っており、その自然な拡張）。パスワードは telnet ネゴ時に送られるため、trace は既定 maskTx で保護。

## D2: サインオン画面 capture は maskTx: false で記録する

- 背景: TraceRecorder の既定は tx 伏字化（パスワード保護）。
- 決定: `scripts/capture-signon.mjs` はサインオンを行わない（送信はネゴシエーション応答のみで
  秘匿情報ゼロ）ため、maskTx: false でフル記録する。デバッグ・リプレイの忠実度を優先。
- 理由 / 代替案: 資格情報を送るキャプチャ（T13）では既定どおり伏字化する。
- 影響: なし（fixtures に秘匿情報は含まれない）。
