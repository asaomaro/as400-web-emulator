# 要件: AS400 5250 MCP サーバー ＋ Web エミュレーター

## 背景 / 課題

AS400（IBM i）の業務画面は 5250 プロトコルの専用エミュレーターでしか操作できず、
次の 2 つのニーズに応えられていない。

- **LLM からの操作**: AI エージェント（MCP クライアント）が 5250 画面をテキストベースで
  読み取り・操作できれば、AS400 業務の自動化・支援が可能になる。
- **ブラウザからの操作**: 専用エミュレーターをインストールせず、Web ブラウザだけで
  5250 画面を表示・操作したい。

## 目的 / ゴール

1 つの 5250 通信基盤の上に、次の 2 つの利用ケースを両立させる。

- **ケース A（MCP）**: MCP クライアントから MCP ツール経由で 5250 画面をテキストベースで
  取得・操作できる（接続 → 画面取得 → 入力・キー送信 → 更新後画面の取得）。
- **ケース B（Web）**: Web アプリの 5250 エミュレーター画面から、人間が 5250 画面を
  表示・操作できる。

## スコープ

### 対象

- **MCP サーバー**: AS400 と 5250 プロトコルで通信し、画面の表示情報の取得と
  画面操作（フィールド入力・AID キー送信）を MCP ツールとして提供する。
- **Web アプリ（5250 エミュレーター）**: 5250 画面をブラウザに表示し、
  フィールド入力・ファンクションキー操作ができる。
- 両者は **共通の 5250 通信コアを共有**し、MCP ツールと Web 向け API（別系統）を併設する。

### 対象外

- 5250 プリンターセッション（印刷エミュレーション）。
- ファイル転送（IFS / DB アクセス等）、5250 画面以外の AS400 機能。
- マクロ・スクリプト記録再生などの高度なエミュレーター機能（将来拡張）。
- AS400（IBM i）側のサーバー実装・シミュレーター（検証は PUB400 等の実機で行う）。

## 機能要件

### 共通（5250 通信）

- AS400 へ TCP で接続し、5250 データストリームを解釈して画面情報を構築できる。
  - 通信実装は **ACS（IBM i Access Client Solutions）の jar が利用可能なら優先採用**。
    不可能な場合は既存の 5250 ライブラリを利用する（採否は research/spec で決定）。
- **DBCS（日本語）対応は必須**（EBCDIC ⇔ Unicode 変換、シフト文字を含む日本語画面の表示・入力）。
- **複数セッション**を同時に扱える（セッション ID で識別し、MCP クライアントと Web が
  それぞれ独立にセッションを持てる）。
- 画面サイズ **24x80 および 27x132（ワイド画面）** に対応する。
- **TLS（telnet over SSL、ポート 992 等）接続**に対応する。
- フィールド属性（入力可否・非表示/パスワード・反転・下線等）とカラー情報を保持する。
- **カーソル位置の管理と送信**: 画面情報に現在のカーソル位置（行・桁）を含める。
  AID キー送信時にカーソル位置を指定してホストへ送信できる
  （例: SEU/PDM 等で F4 押下時、カーソル位置の入力項目に応じたプロンプトウィンドウを
  ホストが表示するユースケースに対応するため）。
- **DBCS の SO/SI 桁の維持**: DBCS 文字列の前後のシフトアウト/シフトイン制御は画面上
  半角 1 桁分（見た目はスペース）を占有する。この桁を欠落・詰めずに保持し、
  画面のどの行でも桁位置（カラム）がズレないこと。
- 接続情報の指定は **都度入力**（ホスト指定＋サインオン画面での認証）と
  **設定プロファイルによる自動サインオン**の両方をサポートする。

### ケース A: MCP サーバー

- MCP ツールとして最低限、次の操作を提供する（ツールの詳細設計は spec で確定）。
  - セッションの接続 / 切断 / 一覧
  - 現在画面の表示情報の取得（テキストベースで LLM が読解できる形式。
    フィールド位置・属性・カーソル位置を含む）
  - フィールドへのテキスト入力
  - AID キー送信（Enter、F1〜F24、PageUp/Down 等）と、操作後の更新画面の返却。
    送信時に**カーソル位置を指定できる**こと（F4 プロンプト等のユースケース）

### ケース B: Web アプリ

- 5250 画面をブラウザ上に忠実に表示する（文字グリッド・フィールド属性・カラー）。
  - 属性（制御コード）に応じた**文字色の再現**を含む（例: ソース表示＝SEU 等で
    制御コードにより行内の文字色が切り替わる画面を、指定された色どおりに表示する）。
  - 全行・全桁が 5250 画面の桁位置と 1:1 で対応する固定グリッド表示とし、
    DBCS の SO/SI 桁（見た目半角スペース）を含んでも桁位置がズレないこと。
- 入力可能フィールドへの文字入力、Enter・F1〜F24 等のキー操作ができ、
  操作後の画面更新が反映される。
- カーソル位置の表示と移動（クリック・キー操作）ができ、ファンクションキー押下時には
  現在のカーソル位置がホストへ送信される（F4 プロンプト等のユースケース）。
- 画面の描画は、入力フィールドをオーバーレイ配置するのではなく、
  出力テキストと入力要素をインラインに並べる方式
  （例: `<span>ABC</span><input>…</input><span>DEF</span>`）を第一候補として検討する
  （より良い方式があれば spec/design で比較・提案してよい）。

## 非機能要件 / 制約

- 技術スタックは **TypeScript / Node.js に統一**する（Web フロントエンド含む）。
  - ACS jar（Java）を採用する場合の Node からの利用方式（子プロセス連携等）は research で確認する。
- 検証環境は **PUB400.com 等の公開 IBM i 環境**を使用する。
- 認証情報（パスワード等）を平文でログ・画面情報に漏らさないこと
  （非表示フィールドのマスク、設定プロファイルの安全な保管）。
- MCP サーバーは標準の MCP 仕様（公式 SDK）に準拠する。

## 完了条件 (受け入れ基準)

- [ ] MCP クライアントから PUB400 に接続し、サインオン画面の表示情報（テキスト・フィールド）を取得できる。
- [ ] MCP ツールでユーザー・パスワードを入力して Enter を送信し、サインオン後のメニュー画面が返る。
- [ ] MCP ツールで F キー・ページキーを送信し、画面遷移が正しく反映される。
- [ ] Web ブラウザで 5250 画面が表示され、フィールド入力と Enter/F キー操作で画面遷移できる。
- [ ] 日本語（DBCS）を含む画面が Web・MCP の両方で正しく表示される。
- [ ] フィールド属性（非表示＝パスワードのマスク、入力可否）とカラーが Web 表示に反映される。
- [ ] 24x80 / 27x132 の両画面サイズで表示が崩れない。
- [ ] TLS 接続で PUB400（または TLS 対応ホスト）に接続できる。
- [ ] MCP クライアントのセッションと Web のセッションを同時に開いて独立に操作できる。
- [ ] 設定プロファイルによる自動サインオンが機能する。
- [ ] カーソル位置を指定して F4 等を送信すると、ホスト側でカーソル位置に応じた
      プロンプトウィンドウが表示される（SEU/PDM 等で検証）。
- [ ] ソース表示（SEU 等）で制御コードに応じた文字色が Web 画面に再現される。
- [ ] DBCS を含む行（SO/SI 桁あり）とそれ以外の行で、桁位置が Web・MCP のテキスト表現とも
      ズレない（SO/SI 桁は半角スペース等で 1 桁分保持される）。

## 未確定事項 / 確認したいこと

- **ACS jar の利用可否**（research で解消）: ライセンス上の再配布・組込み可否、
  Node からの連携方式（子プロセス / ソケット等）、ヘッドレス利用の可否。
- **代替 5250 ライブラリの調査**（research で解消): Node/TypeScript で使える既存 5250
  クライアント実装の有無と DBCS 対応状況（例: tn5250 系 npm パッケージ）。
  純 TypeScript でのプロトコル自前実装が必要になる場合の規模感。
- **PUB400 の検証条件**（research で解消）: TLS ポート・日本語 CCSID 画面の検証可否、
  アカウント取得手順。
- Web 向け API の形式（WebSocket / REST / SSE）: spec で決定。
- MCP ツールの画面表現フォーマット（LLM が読みやすいテキスト形式）: spec で決定。
