## 差し戻しと修正（1 件）

**一覧を最後まで辿れないディレクトリで、部分的な zip を黙って返していた。**

上限拒否が速いことを `/QSYS.LIB` で確かめている最中に、同じ経路の別の分岐に気づいた。
`listAll` は `canContinue` が false のとき部分的な結果を返し、`collectFiles` はそれを
`ok: true` として扱っていた。**ファイルが欠けた zip が正常応答として返る**状態で、
利用者は欠けに気づけない。

`CollectResult` に `incomplete` を足して拒否し、409 `INCOMPLETE_LISTING` で
辿れなくなったパスを伝えるようにした（decisions D6）。
実機で `/QSYS.LIB` が 409 になり、正常なフォルダは 200 のままであることを確認済み。

## 未検証の範囲（deliver に引き継ぐ）

green だが**全数検証ではない**。

- **認証を有効にした状態**（`--users`）での動作。今回は認証オフで起動して確認した。
  認可は他のホスト API と同じ経路（`resolveSource`）に乗せているので同じ挙動になるはずだが、
  実際に確かめてはいない
- **上限に近い規模の zip**。実機で確かめたのは 4 ファイル / 267 バイトの zip と、
  上限超過の拒否のみ。20MB 近い実データでの所要時間（推定 3.5 分）は測っていない
- **同時要求**。接続は要求ごとに開くので競合しない設計だが、並行で叩いてはいない
- **権限エラー（rc=13）と使用中（rc=1/32/33）の実機再現**。
  PUB400 で該当する状況を作れていない。写像は単体テストで固定してある
- **`write` の上書き確認**。spec は「上書きになる場合は事前に確認する」としているが、
  これは UI 側の責務（03）。サーバーは既定で上書きする

## 気づき（03-web-ui へ）

- `list` は `hasMore` / `canContinue` / `nextRestartId` をそのまま返す。
  **`hasMore` だけで判断しないこと**（`entries` が空でも true になりうる／
  `canContinue` を見ないと無限ループする）。実機で両方とも再現している
- `/QSYS.LIB` 配下は zip 化できない（409）。「この場所は一括ダウンロードできません」と伝えること
- 413 の本文に `files` / `bytes` / `maxFiles` / `maxBytes` が入っているので、
  「N ファイル / M バイトで上限 X を超えました」と具体的に出せる

## 新たに判明した挙動（03 と backlog へ）

**権限の無いサブディレクトリが 1 つあると、zip 全体が失敗する。**
`/` を zip しようとすると `/QDLS` の権限で 403 になる。
これは D6（欠けたアーカイブを返さない）と一貫した挙動だが、
「1 つ読めないフォルダがあるだけで一括ダウンロードできない」という UX でもある。

- 03-web-ui は、この 403 を「一部のフォルダに権限が無いため一括ダウンロードできません」と伝えること
- 「読めないものを除いて固める」を選べるようにするかは backlog
