# 決定記録

## D1: rc=1/32/33 は `RESOURCE_BUSY` として core で投げ分ける（01 の D11 の前提を訂正）

- 背景: 01 の D11 は「rc=1（使用中）/ 32（共有違反）/ 33（ロック違反）の写像は 02 で決める」としていた。
  だが **core はこれらを `PROTOCOL_ERROR` で投げており、`statusOf()` は `e.code` で分岐する**。
  02 の中だけでは区別できない。D11 の前提が誤っていた。
- 決定: core の `ErrorCode` に `RESOURCE_BUSY` を足し、`fileFailure` で rc=1/32/33 を投げ分ける。
  HTTP は **409** に写す。
- 理由 / 代替案: `ACCESS_DENIED` に混ぜる案は退けた。あちらは**権限**の問題で待っても変わらないが、
  こちらは**時間**の問題で待てば通りうる。同じにすると案内が変わってしまう。
  502 のまま残す案も退けた。「他の誰かが掴んでいる → 時間をおいて再試行」は利用者が対処できるので、
  「ホストが落ちている」を意味する 502 は誤誘導になる。
- 影響: `statusOf()` の戻り型に 409 を追加した（`400 | 403 | 404 | 409 | 502`）。

## D2: `ifs-collect` は `IfsConnection` ではなく narrow な口を受け取る

- 背景: 01 の review M3 で「`listFiles` が単体テストできない構造だった」ことが最大の指摘だった。
  具象クラスを要求すると、テストのたびに実接続かそれに準ずるものが要る。
- 決定: `listFiles` と `readFile` だけを持つ `IfsReader` interface を定義し、これを受け取る。
- 理由: プレーンなオブジェクトで再帰収集・上限判定・ページングの罠を検証できる。
  同じ轍を踏まないための構造上の予防。
- 影響: `ifs-collect.test.ts` が偽の口で 13 件。実機なしでページングの無限ループを再現・検証できる。

## D3: zip の上限判定は一覧のメタデータで行う

- 背景: 実効スループットが約 100KB/s（research F4-1 の実測）。
- 決定: 一覧が返す `size` を積算し、**上限を超えると分かった時点で中身を 1 バイトも読まずに拒否**する。
- 理由: 500MB を読み切ってから「大きすぎます」と言うのは、利用者を 80 分待たせてから断ること。
  拒否は一覧の往復だけで済むので速い。
- 影響: 413 の本文に実際の件数とバイト数、および上限値を載せる。
  「拒否するとき読み取りが 0 回であること」をテストで固定した。

## D4: ZIP は自前で組み、検証は外部ツールに任せる

- 背景: 依存を増やさず `node:zlib` で組む方針（親 spec）。
- 決定: 局所ヘッダ / セントラルディレクトリ / 終端レコードを自前で書き、CRC-32 も自前。
  **zip64 は非対応**と明示し、`--ifs-zip-max-bytes` に 4GB 以上が来たら起動時に弾く。
- 検証: **自分のパーサで読み返さない**（同じ誤解で書いて同じ誤解で読めば一致する）。
  `unzip -t` と Python の `zipfile` に通す。
- 気づき: 非 ASCII 名のテストが最初落ちたとき、原因は「bit 11 の実装漏れ」と
  「`unzip` 側の制限」の 2 通りありえた。バイトを直接確認して **bit 11 は立っており
  名前も正しい UTF-8** であることを示し、Python の `zipfile` と `unzip -O UTF-8` が
  どちらも正しく読めることで切り分けた。`unzip` 6.00（2009）が既定で bit 11 を尊重しないだけだった。
  自分のパーサで読み返していたら、この切り分けはできていない。

## D5: 自分のテストが弱かったのを変異テストで検出した

- 背景: `canContinue` を見ずに回すと無限ループする、というガードのテストを書いた。
  変異させたところ**素通りした**。偽の口が `nextRestartId` を返しておらず、
  `nextRestartId === undefined` で先に止まっていたため、ガードが実行されていなかった。
- 決定: 矛盾した入力（`canContinue: false` かつ `nextRestartId` あり）を与えてガードを通す。
  ループ回数の上限も置き、変異時にヒープを食い潰す前に明確に失敗させる。
- 理由: core は `canContinue` が false のとき `nextRestartId` を返さないので、この組み合わせは
  本来起きない。だが「`nextRestartId` があるかどうか」だけで回す実装はその契約に寄りかかっており、
  契約が破れたときに無限ループする。**契約が破れても止まること**を確かめる価値がある。
- 影響: 変異させると 1 件が明確に落ちる（以前は 13 件すべて緑のまま）。
  01 の review M3 と同じ形の欠陥を、今回は自分で検出できた。

## D6: 一覧を辿り切れないなら zip を作らない（黙った取りこぼしを禁じる）

- 背景: test 工程で `/QSYS.LIB` に zip を要求したときの挙動を確かめていて気づいた。
  `listAll` は `canContinue` が false のとき**部分的な結果を黙って返して**おり、
  `collectFiles` はそれを「全部取れた」として `ok: true` を返していた。
  つまり **`/QSYS.LIB` のような場所では、ファイルが欠けた zip が正常応答として返る**。
  利用者は欠けていることに気づけない。
- 決定: `CollectResult` に `{ ok: false, reason: "incomplete", path }` を足し、
  辿り切れないディレクトリがあれば**拒否する**。HTTP は 409 `INCOMPLETE_LISTING` で、
  どのディレクトリで辿れなくなったかを添える。
- 理由 / 代替案: 部分的な zip に警告ヘッダを付けて返す案は退けた。
  ダウンロードしたファイルに警告は残らない。**欠けに気づけないアーカイブは、失敗するより悪い**。
  「先頭 N 件だけの zip」を明示的に要求できる形にする案もあるが、今回の要求には無い。
- 影響: `/QSYS.LIB` 配下は zip 化できない（一覧と単一ダウンロードは可能）。
  03-web-ui はこの 409 を「この場所は一括ダウンロードできません」と伝えること。
- 気づきの経緯: 上限拒否が速いことを実機で確かめている最中に、
  同じ経路の別の分岐が黙って成功していることに気づいた。
  **確認しに行った対象とは別のものが見つかった**例。

## D7: `/read` の CCSID 復号は 02 では実装しない（が、黙って壊さない）

- 背景: review M1。spec は復号を「中身 → タグ → 手動」の決定表で行い、応答に
  `ccsid` / `detectedBy` を載せると定めているが、実装は**無条件 UTF-8** だった。
  `TextDecoder` は非 fatal なので、EBCDIC（IBM i の主流）は U+FFFD の羅列になり
  **エラーも警告も出ない**。それを編集して書き戻すと元ファイルが壊れる。
- 決定: 決定表の 2 段目（タグに従う）は **02 では実装しない**。
  `IfsConnection` が CCSID タグを露出していないため（`readFile(path)` しか無い）、
  core に口を足さないと実装できない。
  ただし**黙って壊さない**——`new TextDecoder("utf-8", { fatal: true })` にして、
  読めないものは 415 `UNSUPPORTED_ENCODING` で明示的に断る。
  spec の決定表の最終行「復号しない。手動選択かダウンロードを促す」に倒した形。
- 影響: **後続タスクへ送る**。core に「ファイルの CCSID タグを取る口」を足したうえで、
  決定表の全段と `ccsid` / `detectedBy` を実装する必要がある。
  03-web-ui は当面「UTF-8 で読めないファイルはダウンロード」の UX になる。
- 反省: spec に書いた決定表を実装せず、**その事実を記録もしていなかった**。
  「実装しない」判断自体は妥当だが、黙って落とすと後続が気づけない。

## D8: 空ディレクトリは zip に含めない（対象外として明示）

- 背景: review S2。`collectFiles` はディレクトリを再帰の対象にするだけで出力に含めないため、
  子を持たない空ディレクトリは zip から消える。
  plan のテスト方針に「空ディレクトリ」とあったが、実際に書いたのは「空のファイル」のテストで、
  項目を取り違えていた。
- 決定: **含めない**。対象外として明示する。
- 理由: ZIP のディレクトリエントリ（末尾 `/`・サイズ 0）を足せば対応できるが、
  今回の要求は「フォルダを固めて中身を取り出す」ことで、空フォルダの構造保存は求められていない。
  対応するなら `collectFiles` が「子を 1 つも持たないディレクトリ」を判定して出力に含める必要があり、
  再帰の途中で判断できないため 2 パスになる。
- 影響: 展開結果は「元のファイル構成と一致」するが「元のディレクトリ構成と完全一致」ではない。
  backlog に送る。

## D9: ルートのハンドラ本体をテストできる構造にした（3 度目の同種の欠陥）

- 背景: review M4。`host-ifs.test.ts` は入力検証（zod）と `statusOf` の直接呼び出ししか通しておらず、
  **413 を 500 に、409 を 599 に書き換えても 19 件すべて緑**だった（変異テストで確認）。
  ハンドラ本体（413/409 の分岐、ヘッダの組み立て、符号化）は 1 行も実行されていなかった。
- 決定: `HostIfsDeps` に `connect?` を足し、偽の `IfsConnection` を差し込めるようにする。
  `host-ifs-routes.test.ts`（19 件）でルート本体を実際に通す。
- 検証: 同じ変異で **2 件が落ちる**ようになった。`fatal: true` を外す変異でも 1 件落ちる。
- 反省: **この作業で 3 度目**の「テストが本体を通っていない」欠陥。
  01 の `canContinue`（レビュアが検出）、02 の `ifs-collect`（自分で検出）、
  そして今回のルート（レビュアが検出）。
  共通する原因は「実接続しか差し込めない構造のまま、周辺だけをテストした」こと。
  **新しい層を書いたら、まず偽物を差し込める口があるかを確かめる**のが先。

## D10: パスの正規化は入口で 1 回だけ行う（特別扱いを分けて退行させた）

- 背景: review RM2。M3（`path: "/"` が「path is empty」になる）の修正で、
  `base` 側に `root === "/"`、`full` 側に `dir === "/"` という**特別扱いを 2 箇所**に入れた。
  この 2 つが噛み合わず、`path: "//"` で `base` が `"/"`（長さ 1）・`full` が `"/alpha.txt"` となり、
  `full.slice(base.length + 1)` が 1 文字多く食って **`"lpha.txt"` を黙って返した**。
  修正前は `base = "/"` と `full = "//alpha.txt"` で辻褄が合っていたので、**これは退行**。
- 決定: 特別扱いを分けず、**入口で `root.replace(/\/+$/, "")` と 1 回だけ正規化**する。
  末尾のスラッシュは何本でも落ちる。`dir === "/"` の連結規則は 1 つに保つ。
- 検証: `/` `//` `/d` `/d/` `/d//` の 5 パターンで、一覧に渡すパスと zip 内の相対パス、
  読み取り先のフルパスをテストで固定した。正規化を旧版に戻す変異で **3 件が落ちる**。
- 学び: **辻褄が合っている計算に特別扱いを片方だけ足すと壊れる。**
  「片方を直したらもう片方も」ではなく、規則を 1 つにするのが正解だった。

## D11: 読み取り上限は読む前に判定する（zip で正しく解いた問題を read で逆にやっていた）

- 背景: review RM3。S5 で足した `/read` のサイズ上限が **`readFile` の後**に効いていた。
  500MB のファイルなら 100KB/s で 80 分かけて全部読み、全部メモリに載せてから断る。
  D3（zip の上限は中身を読む前に判定する）とまったく同じ論点を、同じ作業の別ルートで逆にやっていた。
  しかもコメントは「ダウンロードへ誘導する」と、守られている前提で書かれていた。
- 決定: 親ディレクトリの一覧（1 往復）でサイズを引き、**読む前に 413 を返す**。
  親を一覧できない場合に備えて、読んだ後の判定も保険として残す。
- 影響: `/read` に一覧の往復が 1 回増える（約 0.5 秒）。

## D12: 復号できないテキストはエラーにせず、200 で「表示手段が無い」と伝える

- 背景: review RS8。当初 415 で返していたが、RFC 9110 の 415 は
  **リクエストのメディアタイプ**に対する応答で、意味が反転している。
- 決定: **200 で `{ content: null, bytes, encoding: null, code: "UNSUPPORTED_ENCODING" }`** を返す。
- 理由: 読み取り自体は成功していて、足りないのは表示手段だけ。
  4xx にすると UI は「失敗した」画面を出すが、実際に出したいのは
  「文字コードを選ぶ / ダウンロードする」という**続きの操作**（spec の決定表の最終行）。
  `bytes` を返せている時点で半分成功しているのも 200 の傍証。
- 影響: 03-web-ui は `content === null` を見て、エラーではなく選択肢を出すこと。

## D13: ディレクトリ数の上限はファイル数と揃えない

- 背景: review RS6。`maxDirectories` の既定を `maxFiles`（500）と同じにしていたうえ、
  ルートから渡しておらず**外から動かせなかった**。
- 決定: 既定を **5,000** にし、`HostIfsDeps` / `AppDeps` / CLI（`--ifs-zip-max-dirs`）に通す。
- 理由: コストの性質が違う。ファイルは転送量に律速される（100KB/s）が、
  ディレクトリは往復 1 回。500 と 5,000 の差は「数十分」ではなく「数十秒」。
  ソースツリーやビルド成果物は 500 フォルダを簡単に超えるので、
  揃えると「合計 1MB しか無いのに弾かれる」ことが普通に起きる。
- あわせて: 超過時のコードを `TOO_MANY_DIRECTORIES` に分け（`TOO_LARGE` と同じだと 03 が
  本文の形で分岐する羽目になる）、文言を「階層」から「数」に直した（深さではなく総数のため）。
