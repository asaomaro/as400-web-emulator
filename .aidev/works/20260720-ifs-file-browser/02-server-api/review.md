# レビュー: 02-server-api

## ラウンド 1（差し戻し）

独立した第三者視点のレビュー。**must 4 / should 9 / nit 10**。
主要な指摘は実装者側でも変異テストで再現・確認した。

### must

**M1. `/read` が spec の CCSID 復号を実装しておらず、EBCDIC を不可逆に壊す**

無条件 UTF-8 で復号していた。`TextDecoder` は非 fatal なので、EBCDIC（IBM i の主流）は
U+FFFD の羅列になり**エラーも警告も出ない**。spec の「復号に使った文字コードで
再符号化して書き戻す」経路に乗ると、read → write で**元ファイルが壊れる**。
spec が定める `ccsid` / `detectedBy` も返しておらず、**実装しない判断を記録もしていなかった**。

**M2. 「4GB 未満を強制する」ガードが実際には ZIP の破損を防いでいない**

`ZIP_MAX_BYTES_LIMIT = 0xffffffff` は 4GB-1 を通すが、ヘッダとセントラルディレクトリの分で
アーカイブは必ず 4GB を超える。`setUint32` は黙って 2^32 で剰余を取るため、
**例外なく壊れた ZIP が返る**。しかもコメントが「起動時に 4GB 未満を強制する」と**事実に反していた**。

**M3. `path: "/"` の zip / download が「path is empty」で 400 になる**

`root.replace(/\/$/, "")` が `"/"` を `""` にし、core が空パスとして弾く。
`/list` は `"/"` を受け付けるので、ルート間で挙動が食い違っていた。

**M4. ルートのテストがハンドラ本体を 1 行も通していない**

**変異テストで確認**: 413 を 500 に、409 を 599 に書き換えても **19 件すべて緑**。
検証していたのは zod の 400、未知システムの 404、`statusOf` の直接呼び出しだけで、
`withIfs` から先は未実行だった。`Content-Disposition` を素の `filename=` に落としても緑。

### should

S1 更新日時が端から端まで無検証（`dosDateTime` を壊しても、time/date を入れ替えても全件緑）/
S2 空ディレクトリが zip から消える（plan の項目を「空のファイル」と取り違えていた）/
S3 ディレクトリ数がどの上限にもカウントされず、0 ファイル・大量ディレクトリで何時間も往復しうる /
S4 上限の検査が CLI 経路にしか無い / S5 `/read` にサイズ上限が無い /
S6 413 の文言が部分集計を総数のように見せる / S7 write・delete に記録が無い /
S8 列挙時のサイズと実読み取りの乖離 / S9 ピークメモリが合計の約 3 倍

### レビュアが検証・確認した論点（指摘なし）

- **ZIP のフィールド配置**を APPNOTE 6.3.x と 1 バイトずつ突合。局所ヘッダ 30+n /
  セントラルディレクトリ 46+n / EOCD 22 バイト、すべて正しい
- CRC-32（多項式・初期値・最終 XOR・既知ベクタ・非圧縮データに対して取ること）
- 汎用フラグ bit 11 は立っており、bit 3（データ記述子）は立てていない（サイズ既知なので正しい）
- 圧縮方式の切り替え、65,535 件の境界
- `.` / `..` による無限再帰は起きない（core が除外済み）
- symlink 除外・`canContinue` ガード・「読む前に拒否」は**変異で落ちることを確認**
- `statusOf` の新コードは IFS 以外で未使用。既存の写像は変わっていない
- **Content-Disposition のヘッダインジェクションは成立しない**
  （`encodeURIComponent` が CR/LF・引用符・セミコロンを全てエスケープ。実装者側でも確認）
- **パストラバーサルへの追加防御は不要**。IBM i の権限に委ねる方針は
  既存の書き込みルート（`host-sql` / `host-upload`）と一貫している

### 対応

must 4 件・should 8 件すべて対応（decisions D7〜D9）。S9 は S4 の上限検査で実効的に抑える。
nit のうち事実に反するコメントと、MS-DOS 日時の上端（2108 年で 1980 年に化ける）も直した。

**M4 の対応後、同じ変異で 2 件が落ちるようになった**（以前は 0 件）。
`fatal: true` を外す変異でも 1 件落ちる。

実機で再確認済み: `path: "/"` の list が 26 件、zip が `/QDLS` の権限で 403（動いたうえで断る）、
EBCDIC の read が 415。

### この作業で 3 度目の「テストが本体を通っていない」欠陥

- 01: `canContinue` の判定式をテスト側にコピーしていた（レビュアが検出）
- 02: `ifs-collect` の偽の口が `nextRestartId` を返さず、ガードが実行されていなかった（自分で検出）
- 02: ルートのハンドラ本体が未実行だった（レビュアが検出）

共通の原因は「**実接続しか差し込めない構造のまま、周辺だけをテストした**」こと。
新しい層を書いたら、まず偽物を差し込める口があるかを確かめるべきだった（decisions D9）。

## ラウンド 2（再レビューでの追加指摘）

**must 3 / should 8**。前回の対応は**コードとしては全て正しく直っていた**が、
「直したがテストが守っていない」形と、**修正が生んだ退行**が見つかった。

### must

**RM1. 「4GB の防波堤」テストがガードに一度も到達していなかった**
偽オブジェクトが `deflateRawSync` で `TypeError` を起こし、型を指定しない `toThrow()` が
それで通っていた。**ガードを丸ごと削除しても 14 件全緑**。
M4 で指摘したのとまったく同じ形が、今度は zip-writer に出ていた。

**RM2. M3 の修正が `path: "//"` で先頭 1 文字を消す退行を生んだ**
`base` 側と `full` 側に特別扱いを分けて入れたため噛み合わず、`"lpha.txt"` を**黙って**返す。
修正前は辻褄が合っていたので、これは私の修正が作った欠陥。
D6 で「欠けに気づけないアーカイブは失敗より悪い」と決めたのと同じ性質。

**RM3. `/read` のサイズ上限が `readFile` の後に効いていた**
500MB なら 80 分かけて全部読んでから断る。S5 の 2 つの目的（待たせない・OOM させない）が
どちらも未達。**D3 で正しく解いた問題を、同じ作業の別ルートで逆にやっていた**。
コメントも「守られている」前提で書かれていた。

### should

RS1 `readCollected` の上限が未テストかつ 400（同じ事象が経路で違うステータスになる）/
RS2 `too-many-directories` の分岐が未テスト・コードが `TOO_LARGE` と同じ・文言が「階層」で誤誘導 /
RS3 `maxDirectories` の変異が OOM でしか捕まらない（D5 の規約を新テストに適用していなかった）/
RS4 上端年のテストが範囲チェックで、ガードを外しても通る /
RS5 `zipMaxFiles` / `readMaxBytes` が buildApp の検査から漏れ、502 に落ちうる /
RS6 `maxDirectories` を外から動かせず、既定 500 は厳しすぎる /
RS7 `/read` の応答が分岐で形が違う / RS8 415 は意味が反転している

### レビュアが確認した論点（指摘なし）

- **M1〜M4 のコードはすべて正しく直っている**
- **M3 のパス組み立て**を実際に走らせて確認: `/` `/d` `/d/` の 3 ケースは正しい（`//` のみ退行）
- **`connect?` は本番の安全性を損なわない**——`index.ts` から `HostIfsDeps` を出しておらず、
  `package.json` の `exports` もサブパス無し、HTTP 入力から到達する経路も無い

### 対応

must 3 件・should 8 件すべて対応（decisions D10〜D13）。

- RM2 は**特別扱いを分けるのをやめ、入口で 1 回正規化する**形に直した。
  `/` `//` `/d` `/d/` `/d//` の 5 パターンをテストで固定
- 変異テストで確認: パス正規化を旧版に戻すと 3 件、`readCollected` の上限を消すと 2 件、
  ディレクトリ上限を消すと 2 件が落ちる（いずれも以前は 0 件）
- 実機で list / read / zip（末尾スラッシュ含む）が正常であることを再確認

### この作業で得た、繰り返し現れた教訓

1. **テストが本体を通っているか、変異させて確かめる**（3 回踏んだ）
2. **辻褄が合っている計算に特別扱いを片方だけ足すと壊れる**（RM2）
3. **同じ論点を別の場所で逆に解いていないか確かめる**（RM3。D3 で正しく解いた問題を read で逆にやった）
4. **自分のパーサで読み返さず、外部の実装で検証する**（ZIP の bit 11 の切り分けができた）
