# 決定記録

## D1: 一覧の打ち切りは戻りコードで判定する（件数比較では判定できない）

- 背景: spec は「`maxCount` 指定時に打ち切られたら `hasMore: true`」とだけ書いており、
  判定方法を決めていなかった。当初は「返ってきた件数が `maxCount` に達したか」で実装した。
- 決定: **終端フレーム（`0x8001`）の戻りコードを正とする。rc=0 が打ち切り、rc=18 が全件返しきり。**
- 理由 / 代替案: 実機に `maxCount=2` を投げて判明した 2 つの事実による。
  1. 打ち切り時の終端は **rc=18 ではなく rc=0** で返る（バイト 22-23 が `00 00`）。
     原典にこの区別の記述は見当たらない
  2. **`.` と `..` もサーバーの件数上限を消費する**。`maxCount=2` では `.` と `..` だけが返り、
     除外後の件数は 0 になる。件数比較では「打ち切られたのに全件取れた」と誤判定する

  当初の実装は rc=0 を `listReplyKind` の `error` に落としていたため、
  `failed to list ...: error 0` という無意味なエラーで失敗していた。
- 影響: `IfsListResult.hasMore` は **`entries` が空でも true になりうる**。
  呼び出し側（02-server-api / 03-web-ui）は「空 = 終わり」と解釈してはならない。
  この注意は `ifs-types.ts` の型コメントにも書いた。

## D2: 続きの起点は「除外前」の最後のエントリから取る

- 背景: D1 の結果、`entries` が空で `hasMore: true` という状態が実在することが分かった。
  このとき `entries` の最後から `restartId` を取る設計では、**続きの起点を失う**。
- 決定: `IfsListResult` に `nextRestartId` を足し、**`.` / `..` を含む受信した最後のエントリ**の
  `restartId` を入れる。
- 理由 / 代替案: 呼び出し側に「`entries.at(-1)?.restartId` を使え」と委ねる案は、
  上記の空ケースで静かに破綻する。ページングが 1 ページ目で止まり、
  しかもエラーにならないので気づけない。
- 影響: 実機で 3 ページ（0 件 → 2 件 → 2 件、合計 4 件）まで通しで確認済み。
  全件一覧の結果と一致した。

## D3: 生接続の露出をやめ、フレームを覗く診断フックに置き換えた

- 背景: research の spike で `IfsConnection.rawConnection` を露出させ、
  レイアウト実測に使っていた。plan では「削除し、生の hex ダンプ機能は残す」としていた。
- 決定: `rawConnection` を削除し、`listFiles(path, { onRawFrame })` を足した。
  **解析より先に呼ぶ。**
- 理由 / 代替案: 生接続を返すと呼び出し側が任意の要求を割り込ませられ、
  連鎖の途中に別要求が入ってフレームがずれる余地が残る。覗くだけの口ならその余地が無い。
  解析前に呼ぶのは、**レイアウトが想定と違って解析に失敗する場面こそダンプが要る**ため。
- 影響: `tools/hostserver-check` の `ifs-list --raw` がこの口を使う。
  実際、D1 の rc=0 はこのダンプで特定した。

## D4: `traced()` の `requestStream` を任意ではなく必須にした

- 背景: spike は `requestStream?` を任意メソッドとして扱っていた。
- 決定: 型で必須にする。呼び出し元 4 箇所（IFS / command / netprint / db）はすべて
  `HostConnection` なので影響はない。
- 理由: `frame-trace.ts` のコメントが警告しているとおり、素通しを忘れると
  「ラップした接続からだけ機能が消える」という気づきにくい沈黙になる。
  任意にしておくと足し忘れがコンパイルを通ってしまう。
- 影響: 今後 `HostConnection` にメソッドを足すと、`traced()` の更新漏れが型エラーになる。

## D5: 連鎖を放棄した接続はトランスポート層で使用不能にする

- 背景: design で「解析に失敗して連鎖を放棄したら接続を破棄する」と決めたが、
  これは呼び出し側の規律に依存しており、守り忘れると静かに壊れる。
- 決定: `IfsConnection.listFiles` で `close()` する（設計どおり）ことに加え、
  **トランスポート側でも `desynced` フラグを立て、以後の要求を即座に失敗させる**。
- 理由: 二重にする価値がある。連鎖の残骸を次の応答として読むと、
  症状が要求と無関係な場所に出て原因が追えない。
  「黙ってずれる」より「はっきり失敗する」方が debug できる。
- 影響: `host-connection-stream.test.ts` に、放棄後の要求が
  `abandoned mid-chain` で失敗することを固定するテストを置いた。

## D6: Restart ID が進まない場所では継続を提供しない（Restart Name は実装しない）

- 背景: test 工程で `/QSYS.LIB` に `maxCount` 付きの一覧を投げたところ、
  **全エントリの Restart ID が 0 で返り、同じ 3 件が無限に返り続けた**。
  `--page` の 50 ページ上限で止めなければ終わらない。UI に載せればハングする。
  `/home` では Restart ID が 6 → 1401 → 2157 と進み、正しくページングできる。
  つまり `/QSYS.LIB` 固有で、原典の注記（QSYS は Restart *Name*、/root は Restart *ID*）と鏡像で一致する。
- 決定: `IfsListResult.canContinue` を足し、**Restart ID が 0、または渡した値から進んでいない場合は
  継続を提供しない**。`hasMore && !canContinue` は「まだあるが、この場所では辿れない」を意味する。
  Restart Name（CP 0x0007）は実装しない。
- 理由 / 代替案: Restart Name を実装すれば `/QSYS.LIB` も最後まで辿れるが、
  要求の組み立てと切り替え判定が増え、両方の経路を実機で検証する必要がある。
  `/QSYS.LIB` は 21,192 件のライブラリ一覧で、ファイルブラウザとしての価値は低い
  （オブジェクト一覧は既存の `host_list_objects` の領分で、requirement の対象外でもある）。
  無限ループを塞ぐガードは価値が明確なので入れ、辿れるようにする方は backlog に送る（ユーザー判断）。
- 影響: 02-server-api / 03-web-ui は `canContinue` を見ること。
  `hasMore` だけを見て `nextRestartId` を渡し続ける実装にすると、同じ無限ループを再現する。
  UI は「この場所は先頭 N 件まで」と伝える必要がある。

## D7: エラーを区別できるコードに分け、終端フレームのエラーでは接続を捨てない

- 背景: review（M2 / M4）。IFS の失敗をすべて `PROTOCOL_ERROR` にしていたため、
  `statusOf()` が既定の **502** に落としていた。`host-api.ts` 自身のコメントが
  「502 は上流との通信失敗に限る」と明言しており、自分のコメントに違反していた。
  さらに終端フレーム（`0x8001`）で例外を投げていたため、**連鎖が終わっている接続まで毒化**していた。
- 決定:
  - `ErrorCode` に `NOT_FOUND` / `ACCESS_DENIED` / `ALREADY_EXISTS` を追加し、
    純粋関数 `fileFailure(what, rc, replyId)` で rc から投げ分ける
  - 終端フレームのエラーは `missing` と同じく **`requestStream` 解決後**に投げる。
    コールバック内 throw は `parseListEntry` の真の途中失敗に限る
- 理由: 前者は spec の 404/403/409 を、文字列照合ではなくコードで満たすため。
  後者は design の `ifs-collect`（1 接続で再帰的に集めて zip 化）で、
  **権限の無いサブディレクトリが 1 つあるだけで走査全体が死ぬ**のを避けるため。
- 影響: 02-server-api は `statusOf()` に 3 コードの分岐を足し、戻り型に 409 を加える必要がある。

## D8: `listFiles` を単体テストできるようにした（テストが見せかけだった）

- 背景: review（M3）。`canContinue` のテストが**判定式をテスト側にコピーして、そのコピーを検証**していた。
  本体のガードを削除しても反転させても全件緑のまま。D6 が記録するとおり実機で無限ループを起こしてから
  入れた、この変更で最も重要な振る舞いが実質未テストだった。
  根本原因は構造で、`IfsConnection` が実ソケットを開く経路しか持たず、偽の接続を差し込めなかった。
- 決定:
  - 判定を `canRestartFrom()` として `ifs-datastream.ts` に切り出し、**本体の関数を直接呼ぶ**テストにした
  - `IfsConnection.forTesting(conn)` を足し、偽の `HostConnection` で連鎖ループを駆動できるようにした
  - `ifs-listfiles.test.ts`（18 件）で `.`/`..` の除外・種別判定・打ち切り・継続の起点・
    異常系・接続を捨てる／捨てないの区別を固定した
- 理由 / 代替案: 判定の切り出しだけでは連鎖ループ本体（D1・D2 の核心）が依然テストできない。
  実ソケット経路しか無いことがテスト不能の原因なので、そこに口を開けるのが筋。
- 検証: ガードを `return true` に変異させると **3 件が落ちる**ことを確認した
  （以前は 0 件だった）。テストが本物であることを主張ではなく実測で確かめている。

## D9: 進行チェックを単調増加にした

- 背景: review（S3）。`lastRestartId !== opts.restartId` は「送った値がそのまま返る」1 種類しか
  検出せず、A→B→A→B の 2-cycle や後戻りは素通りしていた。
  コメントは「進んでいない場合は」と実装より広いことを主張していた。
- 決定: `last > (requested ?? 0)` の単調増加で判定する。
- 理由: 実機の `/home` は 6 → 1401 → 2157 と単調に増える。単調増加を条件にすれば
  「0 が返る」「同じ値が返る」「後戻りする」をまとめて弾ける。コメントと実装が一致する。

## D10: 成功応答の ReqRep ID は要求ごとに違う（類推で揃えて実機で壊した）

- 背景: review R1/R2 で「`open` / `writeFile` / `deleteFile` も `fileFailure` に揃えよ」と指摘され、
  共通の `assertOk()` を作って 3 箇所に適用した。**そのとき成功応答の ID を確かめなかった。**
  実機で書き込みが `unexpected reply 0x800b` で全部落ちた。
- 決定: 成功応答の ID は要求ごとに違うものとして扱う。
  - mkdir / delete: `0x8001` で rc=0
  - **WRITE: `0x800B`**
  - OPEN: `0x8002`（テンプレート先頭がハンドル）
  `assertOk(reply, what, successReplyId?)` に成功 ID を渡せるようにした。
- 理由 / 代替案: 「同じ形に揃える」という指摘自体は正しかった（`replyReturnCode` が非 `0x8001` に
  0 を返す仕様で、想定外の応答を成功と誤認する穴は実在した）。誤っていたのは、
  揃え方を**実機で確かめずに類推で決めた**こと。
- 影響: `ifs-listfiles.test.ts` に成功 ID の違いを固定するテストを 2 件置いた。
  同種の要求を足すときは、成功応答の ID を実機で確かめること。
- 学び: レビュー指摘への対応でも、プロトコルの事実は実機で確かめる必要がある。
  指摘が正しいことと、その直し方が正しいことは別。

## D11: rc=1 / 32 / 33（使用中・共有違反・ロック違反）の扱いは 02-server-api で決める

- 背景: review R8。これらは今も `PROTOCOL_ERROR` → 502 に落ちる。
  spec に規定が無く、`NOT_FOUND` を足した理屈（利用者が対処できる状況を
  「ホストが落ちている」と伝えない）はこれらにも当てはまる。
- 決定: **core では専用コードを足さず、02-server-api でルートを書くときに決める**（ユーザー判断）。
- 理由: 「他の誰かがファイルを掴んでいる → 時間をおいて再試行」は利用者が対処できるので
  502 は誤誘導だが、400 番台のどれに当てるかは HTTP の話で spec に無い。
  ルートを書く文脈で決める方が素直。
- 影響: **02-server-api への申し送り**。`statusOf()` に分岐を足すときに、
  rc=1/32/33 をどう扱うか（409 か 503 か、あるいは 502 のままか）を決めること。

## D7 追記: 02-server-api への申し送り（R1 を受けて訂正）

ラウンド 1 の D7 は「02 は `statusOf()` に 3 分岐を足せばよい」と読めたが、
当時は `open` / `writeFile` / `deleteFile` が素の `PROTOCOL_ERROR` のままで、
**7 ルート中 5 つで新コードが出ない**状態だった。現在は 5 箇所すべてを `fileFailure` に通してある。

02-server-api がやること:

1. `statusOf()` に `NOT_FOUND` → 404、`ACCESS_DENIED` → 403、`ALREADY_EXISTS` → 409 を足す
2. 戻り型に **409 を加える**（現在は `400 | 403 | 404 | 502`）
3. rc=1/32/33 の扱いを決める（D11）
4. `hasMore` が true でも `entries` が空になりうること、
   `canContinue` を見ないと無限ループすることに注意（D1・D2・D6）
