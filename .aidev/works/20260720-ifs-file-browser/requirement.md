# 要件: IFS ファイルブラウザを Web UI に載せる

## 背景 / 課題

IFS（統合ファイルシステム）へのアクセスは、プロトコル層〜MCP ツール層までは実装済みで実機検証も通っている。
しかし現状は **ファイル単体の読み書きに限定**され、しかも **MCP 専用**で Web UI からは一切触れない。

現状の切り分け（調査結果）:

| 範囲 | 状態 | 根拠 |
|---|---|---|
| ファイル読み `readFile` | 実装済み | `packages/core/src/hostserver/ifs/ifs-connection.ts:104` |
| ファイル書き `writeFile` | 実装済み | 同 `:133` |
| ファイル削除 `deleteFile`（core のみ） | 実装済み・MCP 非公開 | 同 `:155` / `packages/server/src/host-server-tools.ts:452` |
| MCP `host_read_file` / `host_write_file` | 実装済み | `host-server-tools.ts:417-479` |
| **ディレクトリ一覧** | **未実装** | `FILE_REQ.listFiles = 0x000a` が `ifs-datastream.ts:20` に定数だけ定義され、ビルダも呼び出しも無い |
| ディレクトリ作成 | 未実装 | `host-server-tools.ts:451-452`（「親ディレクトリが無い場合は失敗する」） |
| **HTTP API `/api/host/ifs/*`** | **未実装** | ルート登録なし |
| **Web UI パネル** | **未実装** | `packages/web-ui/` に `ifs` の文字列が 1 件も無い |
| 32KB 超の複数ブロック読み書き | コードはあるが未検証 | ループは `ifs-connection.ts:110-125 / 138-148` にあるが `.aidev/backlog/hostserver.md:177` が未チェック |

そのため、利用者は IFS 上のファイルを見るのに MCP 経由でフルパスを指定するしかなく、
「どこに何があるか分からないと何も出来ない」状態になっている。

なお、アクセス経路は **SSH ではない**。ホストサーバー QZLSFILE（8473/9473）への生ソケット接続で、
SQL・スプールと同じ transport / signon / port-mapper を共有する（`ifs-connection.ts:1-8`）。
SSH への言及は `port-mapper.ts:55` のコメント（ポート上書きの説明）と過去 spec の構想メモだけで、実装は無い。

## 目的 / ゴール

Web UI から IFS を **探索・閲覧・取得・更新**できるようにし、MCP を介さずに IFS 上のファイルを扱えるようにする。

## スコープ

### 対象

- `listFiles`（要求 ID `0x000a`）のプロトコル実装 — ビルダ・応答解析・実機での往復確定
- 32KB（`DEFAULT_CHUNK`）を超える複数ブロック読み書きの実機検証（backlog `hostserver.md:177` の消化）
- HTTP ルート `/api/host/ifs/*`（一覧・読み取り・書き込み・zip 取得）
- Web UI の IFS パネル: 左ペインにディレクトリツリー、右ペインにファイル一覧
- プレビュー
  - テキスト（文字コード変換あり）
  - PDF（ブラウザ内表示）
  - 画像（PNG / JPEG 等）
  - 上記以外はプレビューせずダウンロードのみ
- フォルダの zip 一括ダウンロード（**サブフォルダを含めて再帰**、件数・総バイトの上限あり）
- 書き込み系: ファイルのアップロードと、テキストファイルの編集・保存

### 対象外

- SSH / SFTP 経路の実装（本 PJ は QZLSFILE 直結で一貫させる）
- ファイル属性・権限（オーナー、authority）の表示・変更
- シンボリックリンクの作成・追跡
- IFS 上のファイル移動・リネーム
- zip の**アップロード**（展開して配置する方向）
- QSYS.LIB 配下をオブジェクトとして解釈する表示（既存の `host_list_objects` の領分）

## 機能要件

### 一覧・ナビゲーション

- 指定パスのディレクトリ内容を、ファイル名・種別（ファイル/ディレクトリ）・サイズ・更新日時つきで取得できる
- 左ペインのツリーでディレクトリ階層を展開・折りたたみできる
- 右ペインで選択中ディレクトリの内容を一覧できる
- 存在しないパス・権限の無いパスは、原因が分かるエラーとして表示される

### プレビュー

- テキストファイルは文字コードを変換したうえで表示できる
- PDF はブラウザ内で閲覧できる
- 画像は表示できる
- プレビュー非対応の形式は、その旨とダウンロード手段を示す
- プレビュー対象が大きい場合に、UI が固まらない形で扱える

### 取得

- 単一ファイルをダウンロードできる
- フォルダを zip として一括ダウンロードできる（サブフォルダ再帰）
- zip 化は件数・総バイトの上限を持ち、超過時は落ちるのではなく理由が分かる形で拒否される

### 書き込み

- ローカルのファイルを IFS 上の指定パスへアップロードできる
- テキストファイルを編集して保存できる
- 上書きが発生する操作は、事前に上書きであることが利用者に分かる

## 非機能要件 / 制約

- 認証・セッションは既存の host パネル（SQL・スプール等）と同じ資格情報経路に乗せる。IFS 専用の認証は作らない
- 既存の `IfsConnection` は 1 ファイルあたり `MAX_FILE_BYTES = 256MB` の歯止めを持つ（`ifs-connection.ts:40`）。この方針を崩さない
- 文字コードは UTF-8 前提にしない。PUB400 は CCSID 273 という実績があり（'@' が化けて CPF1120 になる事例）、変換の指定・推定を扱えること
- **プロトコルの応答レイアウトを推測で決めない。** 過去に「応答のデータ開始位置は宣言されたテンプレート長から求められない。整理を試みたが実機で往復が壊れ固定配置に差し戻した」という失敗が記録されている（`.aidev/backlog/hostserver.md:105-106`）。listFiles でも同じ罠を踏みうる
- zip 生成でサーバのメモリを総取りしない（大量ファイル・大容量時の振る舞いを決めること）

- [ ] プレビュー非対応形式でダウンロードが行える

- [ ] ファイルのアップロードと、テキストの編集・保存が実機で反映される

## 未確定事項 / 確認したいこと

いずれも **research 工程での実機採取・原典確認で解消したい**（推測で埋めない）。

- **listFiles の応答レイアウト**: エントリの並び、可変長ファイル名の位置、種別・サイズ・日時の表現。
  jtopenlite の該当実装の確認と、実機での往復採取が要る
- **ディレクトリ作成の要否**: アップロードをスコープに入れたため、「親ディレクトリが無いと失敗する」現状の制約が
  UX に直接出る。mkdir（要求 ID の特定を含む）を今回に含めるかは research の結果を見て判断したい
- **文字コードの決定方法**: ファイルごとの CCSID を IFS から取得できるのか、利用者に選ばせるのか、既定値を置くのか
- **ツリーの列挙コスト**: ルート直下（`/QSYS.LIB` 等）が巨大な場合の既定の振る舞い（遅延展開・件数打ち切り）
- **zip の上限値**: 件数・総バイトの具体的な閾値と、超過時の伝え方
- **既存 PR #106 との関係**: スプール Web UI が未マージ。パネル登録まわりで衝突しうる
