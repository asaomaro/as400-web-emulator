# 統合レビュー: IFS ファイルブラウザ（親 work）

3 つの subtask はいずれも単独 review 通過済み（01: 3 ラウンド / 02: 3 ラウンド / 03: 4 ラウンド）。
ここでは**結合起因の欠陥**と**subtask 横断の一貫性**を見る。

## ラウンド 1

第三者エージェントに 4 観点で点検してもらった。**must なし / should 1 / nit 3**。

### 総評（レビュアより）

- **spec に書かれていて実装も記録もない機能は見つからなかった。**
  未実装項目（CCSID 決定表・手動文字コード切替・テキスト編集保存・プレビュー上限とヌルバイト判定・
  rmdir）はすべて decisions / test-result に記録済み。**silently-dropped は該当なし。**
- ページングの罠（01-D1/D2/D6）が 3 層で一貫。「空 = 終わり」と誤解する経路が無い。
- エラーコードが core（`fileFailure`）→ server（`statusOf`）→ web-ui（`messageFor`）と繋がり、
  利用者が対処できるコードに web-ui が知らないものは残っていない。
- subtask 間の判断の食い違いはすべて後続層が明示的に訂正（01-D11 → 02-D1、
  01-plan の「ツリーを作った」虚偽記録 → 03-D1）。**未解決の矛盾なし。**
- デバッグコード・TODO・console.log・コメントアウトは差分 9 ファイルすべてで 0 件。

### should（記録の言い回しの過大主張）

`messageFor` の網羅を「server が実際に返す**全コード**を日本語化」と書いていたが、
正確には**利用者が対処できる 7 コード**に限る。インフラ障害系（`PROTOCOL_ERROR`=502 /
`CONFIG_ERROR`=400）は `default` でサーバーの生文言（英語のこともある）にフォールバックする。
これは他のホストパネルと同じ意図的な方針だが、記録の文言が実態より広く読めた。
→ decisions D12 と test-result を「対処可能な全コード」と正確化した（コードは変更なし）。

### nit（いずれも問題なし）

- 4GB 起動時ガードの実値 `0xf000_0000`（約 3.75GB）は zip64 境界より安全側に厳しい
- `usePreview.IMAGE_EXT` は spec の拡張子に svg/bmp/ico を足した上位集合（無害）
- `--ifs-read-max-bytes` は spec の既定値表に無い自然な追加

### 判定

**must なし。** should は記録の正確化で対応（実装の欠陥ではない）。
結合は繋がっており、未実装は漏れなく記録されている。

## 受け入れ基準に対する注記（deliver でユーザーに諮る）

spec の受け入れ基準「アップロードと**編集・保存**が実機で反映される」のうち、
**編集・保存は未実装**（アップロードは実装・E2E 確認済み）。
requirement 段階でユーザーが「アップロード・編集も含める」を選んだ項目なので、
backlog に送る前に deliver で明示的に諮る。
