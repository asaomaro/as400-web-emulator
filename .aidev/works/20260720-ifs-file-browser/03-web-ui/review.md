# レビュー: 03-web-ui

## ラウンド 1（差し戻し）

独立した第三者視点のレビュー。**must 8 / should 9 / nit 13**。
レビュア側で**変異テスト 35 種**を実施し、テストの空白地帯を実測で示している。

### must

**M1. `.theme-btn` / `.link` は `App.vue` の scoped スタイル。IfsPane には効かない**
規約の意匠に合わせたつもりのクラス名が、**実際には何も適用されていなかった**。
`<label class="theme-btn">` のアップロードはただの文字列として描画されていた。
scoped の適用範囲を確かめずに使っていた（decisions D2）。

**M2. システム切り替えを見ていない（別システムのファイルを消しうる）**
`systemsStore.selected` への watch が無く、一覧が前のシステムのまま残る。
その状態で削除・上書きすると別システムのファイルに飛ぶ。
`HostListPane` は同じ理由で watch を置き、コメントに理由まで書いている（D3）。

**M3. アップロードのループがファイル単位の失敗を握りつぶす**
`withAction` が冒頭でエラー表示を消すため、1 件目が失敗しても 2 件目が成功すれば消えた（D4）。

**M4. 上書き確認が「読み込み済みの entries」しか見ていない**
1000 件超のフォルダで、未取得ページに同名があると**確認なしで黙って上書き**していた（D4）。

**M5. アップロード経路がテストで一切押さえられていない**
base64 変換を消す・`encoding` を `utf8` にする・上書き確認を無効化する・
変換ループを壊す——**4 種とも 33/33 緑のまま素通り**。

**M6. プレビュー失敗時にダウンロードボタンごと消える**
`/read` の 413 は「大きすぎるのでダウンロードしてください」と言うのに、
**画面からダウンロード手段が消えていた**（D5）。

**M7. 「ツリー」が実装されていないのに tasks は完了扱い／コメントが事実と違う**
**この作業で最も重い指摘。** T5「左ペインにツリー」・T7「ツリーの展開」を `[x]` にし、
ファイル冒頭にも「左にツリー、右に一覧」と書いていたが、**ツリーは存在しなかった**。
`useIfsTree` の `expanded` / `toggle` / `loadAll` は死にコードだった。
spec のテキスト編集・保存も未実装で、未検証リストにも挙げていなかった（D1）。

**M8. キーボードで操作できない**
一覧の行が `<li @click>` でフォーカス不能。`hidden` な `<input>` + `<label>` の
アップロードは**キーボードで到達する経路が存在しなかった**。

### should（主要分）

S1 `canContinue` のガードが loadPage 側だけ未検証（01-D8 / 02-D5 と同型の再発）/
S2 操作中に別の操作ができる（`busy` を使っていない）/ S3 プレビューの競合（遅い応答が勝つ）/
S4 エラーと完了メッセージが移動しても消えない / S5 成功時に `error` が残る /
S6 `RESOURCE_BUSY` と `ALREADY_EXISTS` を扱っていない / S7 プレビューのサイズ上限が未実装 /
S8 `<input>` の value を戻していない / S9 E2E のコメントが実態と違う・ダイアログ未処理

### レビュアが確認した論点（指摘なし）

- **配色は全て CSS 変数経由で生色なし**。`--t-red` / `--accent-soft` などが light / dark 双方に
  定義されていることも確認済み（既存 `HostListPane` が生色直書きなのに対し、こちらの方が規約に忠実）
- blob URL の解放は 2 箇所 + `onBeforeUnmount` で正しく、テストも生きている
- ページングの罠（01-D1/D2/D6）と `content: null`（02-D12）は**設計・実装・テストとも良く押さえられている**
- E2E の `.catch(() => {})` は握りつぶしではない（待った後に実状態を読んで `check()` している）。
  固定待機への退行も無く、スキップを NG 扱いにしているのも良い
- アップロードの `String.fromCharCode` は spread を使っていないので**スタックは溢れず、
  バイナリも壊れない**（実装者側でも 5MB / 全 256 バイト値で確認済み）

### 対応

must 8 件と should の主要分を修正（decisions D1〜D6）。**ツリーはユーザー判断で実装**した。

**修正の過程で、自分の修正が 2 度、新たな不具合を生んだ。どちらも E2E が捕まえた:**

- S8 の対応で入れた `input.value = ""` が **`FileList` をその場で空にし、アップロードが動かなくなった**。
  jsdom では再現しない挙動で、単体テストは通っていた
- ダイアログハンドラを二重登録して Playwright が落ちた

追加したテストは、いずれも変異させて本体を壊すと落ちることを確認済み。
実ブラウザ E2E は **12/12 成功**（ツリー操作の 2 項目を追加）。

### この作業を通じた総括

レビュアの総評がそのまま当たっている——
「**ページングの罠と `content: null` という過去に痛い目を見た 2 つは良く押さえられているが、
今回初めて書いた周辺（アップロード、上書き確認、パス組み立て、種別ごとの操作、システム切替）が
まるごとテストの外**にあり、変異が軒並み素通りする」。

02-D9 の教訓（新しい層を書いたらまず偽物を差し込める口があるかを確かめる）を
composable には適用したが、**コンポーネント自身の操作系には適用していなかった**。

## ラウンド 2（再レビューでの追加指摘）

**must 4 / should 8 / nit 8**。前回の must は的確に直っていたが、
「修正が新たな欠陥を生む」が 2 件、そして**検証そのものが嘘だった**箇所が見つかった。

### must

**RM1. E2E のツリー検証に恒真の check があった（12/12 の 1 件が無意味）**
`check("ツリーから移動できる", true, ...)` の第 2 引数がリテラル `true`。永久に成功する。
前回「作っていないものを完了と記録した」ことを反省した直後に、同じことを検証側でやっていた。

**RM2. アップロード中のシステム切替で、残りが別システムのルートに書かれる**
M2 の watch が進行中の操作を止めないため、ループが 1 件ごとに `source()` / `currentPath` を
評価していると、切替後の残りが新システムに飛ぶ。**守ろうとした事故を watch 自身が起こせた**。

**RM3. ツリー展開の通信が `run()` の外にあり、待機表示も操作禁止も効かない**
`openFromTree` が `tree.toggle`（本番の往復・実測 20 秒）を `run()` の外で呼んでいた。
S2 で直したばかりの `busy` が、新しい経路で無効化されていた。

**RM4. アップロードの base64 が依然として未検証**
テストは 2 件増えたが画面の文言しか見ておらず、要求本文を検証していなかった。
`input.value = ""` の退行を単体で捕まえられないのは、テストが `files` をデータプロパティに
差し替えているため（構造的に検出不能）と特定された。

### should（主要分）

S1 `loadPage` の `canContinue` 未検証（01-D8 / 02-D5 と同型）/ RS1 ツリーが現在地を示さない /
RS2 展開済みフォルダを畳まずに移動できない / RS3 ルートの `aria-expanded` が誤情報 /
RS4 `walk()` の死にコード / RS5 切替後が無言のブランク / RS7 確認がファイル数だけ出る /
RS8 ドロップの多重起動

### レビュアが確認した論点（指摘なし）

- **ツリーの実装は本当に動く**（スクラッチのテストで `walk` の再帰・平坦化・深さ・
  同名フォルダの階層違いを観測）。ただしそれを担保するテストが 0 件だった
- M1 / M3 / M6 は的確に直っている。M4 の `existsHere` の設計判断も妥当
- 配色は全て CSS 変数経由（生色なし）、`.theme-btn` / `.link` の使用は 0 件

### 対応

must 4 件・should の主要分を修正（decisions D7〜D9）。

- **RM1**: 恒真の check を実測（開閉が両方向に効く・現在地の `.sel`・パンくずの文言）に置き換えた。
  直した検査が**私の前提の誤り**（「押せば増える」→ 実際は畳まれる）を捕まえ、13/13 になった
- **RM2**: 操作の入口で `src` / `dir` を固定。アップロード中に切り替える回帰テストを足し、
  固定を外すと落ちることを確認
- **RM3**: `openFromTree` を `run()` で包み、`.tree-row` に `disabled` を付けた
- **RM4**: `defineExpose` で `uploadFiles` を直接叩き、要求本文の `content` が
  期待の base64 と一致することを固定。全 256 バイト値の往復も
- ツリーのテスト 5 件を追加（それまで 0 件・変異 0/7 捕捉だった）。
  新規 5 変異すべてが落ちることを確認

### この工程で判明した、自分の失敗の型

**「確かめずに成功を報告する」が 3 回**（decisions D1 / D7 / D8）:
ツリー未実装を完了と記録 / 恒真の check で 12/12 と報告 / 出力を捨てて「ビルド OK」。
いずれも**検証したことにして、実際には検証していない**。形が同じ。

**「修正が新たな欠陥を生む」が通算 6 件**。新しい入口を足したとき、
既存の入口が満たしていた不変条件（`run()` を通る・`disabled` が効く・
操作は開始時のコンテキストに固定される）を照合していないのが共通の原因。

## ラウンド 3（再々レビュー）: 実測で裏付いた

**must 1（QM1）/ should 4 / nit 6**。レビュアが**変異 30 種を実測**し、
前回まで 0 だった捕捉が本物になったことを確認した:
ツリー 8/8・符号化 4/4・RM2 2/2・S1 いずれも本体を壊すと落ちる。
「今回は『直した』の主張が概ね実測で裏付いた」と総評。

### 残った 1 系統（QM1）

**`disabled`（操作中の禁止）が、実装は正しいのにテストで固定されていない。**
S2 → RM3 で 2 度直した不変条件が、`disabled = computed(() => false)` に変異させても緑だった。
D9 が「新しい入口を足したとき既存の不変条件を照合しない」と自認したが、
**その照合を回帰テストとして据えるところまで届いていなかった**（照合の検証側の裏返し）。

### 対応

- QM1 / QS1 / QS2: 保留 fetch で `busy` を立てた状態のテストを追加。
  disabled 常時 false / ヘッダー disabled 除去 / 切替後ガード除去 / 個別上書き確認除去 /
  二重起動ガード除去 の **5 変異すべてが落ちる**ことを確認（以前はすべて素通り）
- QS3: spec の未実装（プレビュー 5MB 上限・ヌルバイト判定）を decisions D11 に
  「実装しない理由」付きで記録し backlog へ（02-D7 の反省に従う）
- QS4: test-result を実態（web-ui 508、追加テストの内訳）に追随

### レビュアが問題なしとした点

- 恒真 check は解消（`grep 'check("...", true'` 0 件をレビュアも確認）
- RS1 の現在地表示・RS2 の名前/キャレット分離・RS3 のルート aria は実装・テストとも一致
- S3（プレビュー競合）は backlog で妥当（データは壊れず表示 path で気づける）

## ラウンド 4（統合 review 派生: 編集・保存の追加）

親の統合 review で「spec の受け入れ基準『編集・保存』が未実装」と判明し、
ユーザー判断で実装（decisions D13）。この追加分の自己レビュー:

- **UTF-8 で読めたテキストに限る**制約が二重防御（テンプレート + `saveText` 入口）で効く。
  復号できないファイルを書き戻して壊すことはない
- 保存の要求本文が `{ encoding: "utf8", content: 編集後 }` になることを UI 経由テストで固定。
  encoding を base64 にする変異・dirty を常に false にする変異が落ちる
- 実ブラウザ E2E 15/15（編集 → 保存 → サーバーから読み戻して反映を確認）
- `<pre>` → `<textarea>` の変更に伴い、E2E の古いセレクタ 3 箇所を追随（保守の教訓を D13 に記録）

判定: must なし。spec の受け入れ基準が完全に満たされた。
