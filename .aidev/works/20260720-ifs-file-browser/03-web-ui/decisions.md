# 決定記録

## D1: 「ツリーを作った」と記録していたが、作っていなかった

- 背景: review M7。`tasks.md` の T5「左ペインにツリー、右ペインに一覧」と
  T7「ツリーの展開」を `[x]` にし、`IfsPane.vue` の冒頭にも「左にツリー、右に一覧」と書いた。
  **実際には一覧とプレビューの 2 ペインしか無く、ツリーは存在しなかった。**
  `useIfsTree` の `expanded` / `toggle` / `loadAll` はテストからしか呼ばれない死にコードだった。
- 決定: **ツリーを実装する**（ユーザー判断）。左に階層、中央に一覧、右にプレビューの 3 ペイン。
  展開されている経路だけを平坦化して描画する（入れ子コンポーネントにすると開閉のたびに再帰再描画になる）。
- 反省: 実機 E2E で「フォルダを辿る」を確認したが、**それは一覧のクリックであってツリーではない**。
  確認した気になって `[x]` を付けた。**辻褄の合う記述を、確かめずに残した**典型。
  作っていないものを「完了」と記録することは、後続が前提を誤る点で単なる未実装より悪い。

## D2: ペイン内のボタンに `.theme-btn` / `.link` は効かない

- 背景: review M1。UI-DESIGN の意匠に合わせるつもりで `.theme-btn` / `.link` を使ったが、
  これらは `App.vue` の **`<style scoped>` 内**の定義で、子コンポーネントの要素には当たらない。
  `<label class="theme-btn">` のアップロードは**ただの文字列として描画されていた**。
- 決定: ペイン内は素の `<button>`（既存 `HostListPane` と同じ）に揃え、
  必要な意匠は scoped スタイルで定義する。`.link` 相当は `.crumb` として自前に持つ。
- 学び: 「規約のクラス名を書いた」ことと「規約どおりに見える」ことは別。
  scoped の適用範囲を確かめずに使っていた。

## D3: システムを切り替えたら見えているものを捨てる

- 背景: review M2。`systemsStore.selected` への watch が無く、
  **ヘッダーは新しいシステムなのに一覧は前のシステムのまま**という状態になりえた。
  この状態で削除や上書きをすると、**別システムのファイルに飛ぶ**。
- 決定: `HostListPane` と同じく watch でリセットする。自動で取り直さないのも同じ
  （切り替えただけで意図しない問い合わせを飛ばさない）。
- 理由: 対象がジョブではなくファイルなので、取り返しがつかない。

## D4: アップロードの失敗を後続の成功で消さない

- 背景: review M3。ループの中で `withAction` を呼んでおり、
  冒頭で `actionError` を空にするため、**1 件目が失敗しても 2 件目が成功すれば表示が消えていた**。
- 決定: 成功と失敗を配列に集め、ループ後にまとめて出す（「1 件置きました / 1 件失敗: ...」）。
- あわせて: 一覧が最後まで読めていない場合（`state !== "loaded"`）は、
  未取得ページに同名があるか判断できないので「上書きする可能性があります」と伝えて確認する（M4）。
  読み込み済みのページだけを見て「無いから安全」と判断していた。

## D5: プレビューの失敗で操作まで消さない

- 背景: review M6。ダウンロード / 削除が `v-if="preview.state"` の中にあり、
  プレビューに失敗すると一緒に消えていた。
  とくに `/read` の 413 は「大きすぎるのでダウンロードしてください」と言いながら
  **ダウンロード手段が画面から消える**。
- 決定: 操作は `selected` に紐づけ、プレビュー領域の外に出す。

## D6: `input.value = ""` は FileList をその場で空にする（自分の修正で壊した）

- 背景: review S8「同じファイルを 2 回選ぶと change が来ない」に対応して
  `input.value = ""` を入れたところ、**アップロードが動かなくなった**。
  値を戻すと `FileList` が空になり、保持していた参照が空を指すため。
- 決定: 値を戻す**前に** `Array.from()` で配列に写す。
- 検出: 単体テストではなく**実ブラウザ E2E が捕まえた**。
  jsdom では `input.value = ""` が `files` を空にする挙動を再現しない。
- 学び: 指摘への対応でも、直した経路を実際に動かして確かめる必要がある。
  この作業だけで「修正が新たな欠陥を生む」のは 3 回目（01 の応答 ID、02 のパス正規化、これ）。

## D7: 恒真の check を書いて「12/12 成功」と報告していた

- 背景: review RM1。E2E に `check("ツリーから移動できる", true, "home を開いた")` と書いていた。
  **第 2 引数がリテラル `true`** で、永久に成功する項目だった。
  直前に D1 で「作っていないものを完了と記録した」ことを反省したばかりで、
  同じことを**検証側**で繰り返していた。
- 決定: パンくずの文言・`.tree-row.sel` の有無・行数の変化を**実際に読んで比較する**形に直した。
- 検出の副産物: 直した検査が**私の前提の誤り**を捕まえた。
  「キャレットを押せば子が増える」と書いたが、一覧を辿った時点で祖先は既に展開済みなので、
  実際には**畳まれる**（253 → 25）。開閉が両方向に効くことを見る形にして 13/13 になった。
  恒真にしていたら、この誤解にも気づけていない。

## D8: 「ビルド OK」と報告していたが、ビルドは失敗していた

- 背景: `npm run build -w @as400web/web-ui >/dev/null 2>&1 && echo "ビルド OK"` と書き、
  **出力を捨てたうえで成功を宣言**していた。実際にはテストの型エラーで失敗しており、
  E2E は 30 分前の古いバンドルを配信し続けていた。
  そのため `data-path` を足しても DOM に出ず、原因を DOM の推測で 2 度外した。
- 決定: ビルドとテストの出力は捨てない。少なくとも `error` 行は残す。
- 学び: **出力を捨てて成功を名乗るのは、恒真の check と同じ形**（D7）。
  この作業では「確かめずに成功を報告する」が 3 回出ている（ツリー未実装・恒真 check・このビルド）。

## D9: 「修正が新たな欠陥を生む」が 2 件（通算 5 件目・6 件目）

- **RM2**: M2 のシステム切替 watch が、進行中のアップロードを止めないまま
  `currentPath` と `source` を切り替えるため、**残りのファイルが新システムのルートに書かれる**
  経路を作っていた。守ろうとした事故を、その watch 自身が起こせる形だった。
  → 操作の入口で `src` / `dir` を固定し、切替を検知したら結果を反映しない。
  **アップロード中に切り替える回帰テスト**を足し、固定を外すと落ちることを確認。
- **RM3**: M7 のツリーが、ナビゲーションの主経路を `run()` の外に出したため、
  待機表示と操作禁止（S2 で直したばかり）を無効化していた。
  → `openFromTree` を `run()` で包み、`.tree-row` にも `disabled` を付けた。
- 共通の構図: **新しい入口を足したとき、既存の入口が満たしていた不変条件**
  （`run()` を通る・`disabled` が効く・操作は開始時のコンテキストに固定される）
  **を満たしているか照合していない**。02-D10（特別扱いを片方だけ足すと壊れる）と同型。

## D10: 操作中の禁止を回帰テストで固定した（2 度直したのにテストが無かった）

- 背景: review QM1。S2（操作中は別操作を止める）→ RM3（run 経路の穴）で**2 度直した**不変条件が、
  「`disabled` computed が実際にボタンを不活性にする」ことを見るテストで固定されていなかった。
  `disabled = computed(() => false)` に変異させても 50/50 緑。
  D9 で「新しい入口を足したとき既存の不変条件を照合しない」と自認したが、
  **その照合を回帰テストとして据えるところまで届いていなかった**（照合の検証側の裏返し）。
- 決定: 保留 fetch で `busy` を立てた状態で、ヘッダーのボタンが `disabled` になることを固定。
  あわせて切替後ガード（QS1）・個別上書き確認（QS2）・二重起動ガード（QS2）も固定。
- 検証: 5 変異（disabled 常時 false / ヘッダー disabled 除去 / 切替後ガード除去 /
  個別上書き確認除去 / 二重起動ガード除去）すべてが落ちることを確認（以前はすべて素通り）。
- 学び: **不変条件は「実装で守る」だけでなく「回帰テストで据える」まで届かないと、
  次の経路で破れても気づけない。** 直した数だけテストを足す。

## D11: spec のプレビュー上限とヌルバイト判定を実装しない（記録する）

- 背景: review QS3 / S7。spec 193-207 は「プレビューは 5MB 上限」
  「テキスト判定は拡張子＋**ヌルバイトが無いこと**」を明記しているが、
  `usePreview.kindOf` は拡張子のみ、サイズ判定は無い。**未実装かつ未記録だった**。
  02-D7 の反省（黙って落とすと後続が気づけない）がそのまま当てはまる。
- 決定: **今回は実装しない。理由を残す。**
- 理由: サーバー側の `readMaxBytes`（既定 5MB）が最後の砦として機能し、
  M6 の対応で「プレビュー失敗でもダウンロード手段は残る」ようになった。
  よって実害は「上限超過のテキストを開くと、サーバーが 413 を返すまで無駄に待つ」に縮んでいる。
  クライアント側で先回りすると体感は良くなるが、`sizeHint` を見て読まずに断る分岐と、
  ヌルバイト判定（内容を一部読む必要がある）が要り、今回のスコープを超える。
- 影響: **backlog に送る**。UTF-8 のバイナリ寄りファイル（拡張子はテキストだが中身にヌル）を
  開くと、UTF-8 として復号できず 415 相当（`content: null`）になる——
  つまり「壊れた表示」にはならず「文字コード未対応」の案内に落ちるので、実害はさらに小さい。

## D12: エラー文言を 1 箇所に集約した（統合テストで英語漏れが出た）

- 背景: 親の統合 test（契約整合の検証）で、web-ui の `describe()` が
  `NOT_FOUND` / `ACCESS_DENIED` の分岐を持たず、削除やアクセス拒否のときだけ
  **英語＋rc の技術文字列**（`File not found (rc=2)`）が利用者に出ることが判明した。
  さらに一覧（`useIfsTree`）とプレビュー（`usePreview`）は `describe()` を通らず
  `e.body.error` を直接使っていたため、その 2 経路は全コードで英語が出る余地があった。
- 決定: 文言化を `ifsApi.ts` の `messageFor(body)` に集約し、`IfsRequestError` の
  `message` に入れる。全 3 経路（操作・一覧・プレビュー）が `e.message` を使う。
  **日本語化するのは利用者が対処できる 7 コード**（NOT_FOUND / ACCESS_DENIED /
  RESOURCE_BUSY / ALREADY_EXISTS / INCOMPLETE_LISTING / TOO_MANY_DIRECTORIES / TOO_LARGE）。
  インフラ障害系（`PROTOCOL_ERROR`=502 / `CONFIG_ERROR`=400 など）は `default` で
  サーバーの生文言に落とす——他のホストパネルと同じ方針で、これは意図的。
- 検証: `KNOWN_ERROR_CODES` と、**server が実際に返す全コードを列挙した表**を
  テストで突き合わせる（`ifs-error-messages.test.ts`）。
  server のコードは実コードから確定させた——ルート本体が返す 3 つ
  （INCOMPLETE_LISTING / TOO_LARGE / TOO_MANY_DIRECTORIES）と、
  `statusOf` を通る 4 つ（NOT_FOUND / ACCESS_DENIED / ALREADY_EXISTS / RESOURCE_BUSY）。
  `NOT_FOUND` の文言を消す変異・`KNOWN` から抜く変異の両方が落ちることを確認。
- 学び: **これは subtask 単独テストでは原理的に出ない。**
  03 のテストは server の応答をモックで作るので、自分の分岐に合わせたモックを渡していた。
  server が「実際に返すコード」と UI の分岐を突き合わせるのは統合の仕事。
  契約は「両側を実装した」だけでは繋がらず、**片側の全出力をもう片側が受けきれるか**を
  照合して初めて繋がる。

## D13: テキストの編集・保存を実装した（当初 backlog 予定 → ユーザー判断で実装）

- 背景: 統合 review で「spec の受け入れ基準『アップロードと編集・保存』のうち
  編集・保存が未実装」と気づき、ユーザーに諮ったところ「今回実装する」となった。
- 決定: テキストプレビューを `<pre>` から `<textarea>` に変え、変更があれば保存ボタンを出す。
  保存は `writeFile` を utf8 で叩く。
- **制約: UTF-8 で読めたテキストに限る。** CCSID の決定表が未実装（02-D7）なので、
  復号できないファイル（`undecodable` = `content: null`）は編集の土台が無い——
  UTF-8 として持てない中身を書き戻すと元ファイルを壊す。
  `editable` computed で二重に守る（テンプレートで textarea を出さない + `saveText` の入口で弾く）。
- 検証: `setValue` → 保存ボタンクリックの UI 経由テストで、要求本文が
  `{ encoding: "utf8", content: 編集後 }` になることを固定。
  encoding を base64 にする変異・dirty を常に false にする変異が落ちる。
  実ブラウザ E2E で「編集して保存 → サーバーから読み戻すと反映されている」を確認（15/15）。
- 学び（E2E の保守）: `<pre>` → `<textarea>` の変更で、E2E に**古い `.preview pre` セレクタが
  3 箇所残り**、各所で 30 秒タイムアウトした。UI の要素を変えたら、それを参照する検証も全部追う。
