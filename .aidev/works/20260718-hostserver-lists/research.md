# 調査: 各種一覧

## 判明した事実

### F1: 3 つとも QGY のオープンリスト形式

| 一覧 | プログラム | ライブラリ |
|---|---|---|
| ジョブ | `QGYOLJOB` | QGY |
| オブジェクト | `QGYOLOBJ` | QSYS |
| ユーザー | `QSYRAUTU` | QSYS |

骨格（受信変数 ＋ リスト情報80 ＋ 形式名 ＋ エラーコード）はスプール一覧と同じ。

### F2: 実機で判明した「通らない指定」

- **QGYOLOBJ の選択制御は 21 バイト**。20 だと `CPF21AC "Length or displacement value 20 not valid."`
  末尾 1 バイトに状態 `*` を EBCDIC で置く
- **追加属性を要求しないときは「数 0 ＋ 空のキー配列」**。
  数 1・値 0 にすると `CPF1867 "Value 0 in list not valid."`
- **QSYRAUTU のグループ指定は `*NONE`**（選択条件が `*MEMBER` 以外のとき）。
  `*ALL` だと `CPF22ED`
- **AUTU0150 の配置**: 名前(10) ＋ ユーザー/グループ指標(1) ＋
  **グループメンバー指標(1)** ＋ テキスト(50)。2 つ目の指標を飛ばすと 1 文字ずれる

### F3: 受信変数の余白を件数に数えない

実機で「248 件返した」と申告しながら、受信変数 8192 バイト・レコード長 62 で
132 件分しか無い状態を観測した。**サーバーの申告を鵜呑みにせず**、
入りきる件数と空レコード（全 0）で打ち切る必要がある。

### F4 ジョブ一覧の 0 件は、出力パラメータの読み違いだった

`QGYOLJOB` はパラメータ 17 個（12 個だと `MCH0802`）。
17 個で呼ぶとエラーは出ないが常に 0 件で、一度は「手がかりが無い」として見送りかけた。

**原因は出力パラメータの位置の誤り**だった。原典の `getParameterOutputLength` を読むと:

```
出力: 0=受信変数 / 3=レコード形式情報 / 5=リスト情報(80) / 12=エラー / 15=経過時間(16)
```

**リスト情報は index 5** なのに index 3 から読んでいた。API は最初から正しく動いており、
解釈が誤っていただけ。

修正後、実機で 30 件を取得（`SCPF` / `QSYSARB` 等のシステムジョブ）。
ユーザー絞り込みも動作（`user=MARO` で自分の 1 件）。

> **反省**: 「エラーが出ないから手がかりが無い」と判断したが、
> 原典の出力パラメータ定義を読めば一目で分かった。
> **出力の割り当てを確認せずに推測で並べた**のが原因。
> エラーが出ないときこそ「送る側」ではなく「読む側」を疑うべきだった。
