# 要件: データ待ち行列サーバー（Data Queue Server）

## 背景 / 課題

IBM i の **データ待ち行列（Data Queue / DTAQ）** は、ジョブ間・プログラム間で非同期にデータを
やり取りするための軽量なキューで、実務では「片方が SNDDTAQ で積み、もう片方が RCVDTAQ で待つ」
という連携に広く使われる。

この PJ は IBM i のホストサーバー機能を TypeScript で自前実装してきた
（signon / SQL / コマンド / IFS / スプール）。しかし**データ待ち行列は完全に未実装**で、
コード・ドキュメント・backlog のどこにも存在しない。

混同しやすい既存機能があるので切り分けておく（背景の明確化）:

- **MSGW（応答待ちメッセージ）** — スプール（netprint, `0xE003`）配下に部分実装があるが、
  これは「スプールファイルに付いた応答待ちメッセージ」であって**データ待ち行列とは別物**
- **コマンドの CPF メッセージ解析** — コマンド実行結果のメッセージであって待ち行列ではない

アクセス経路は SSH ではない。**データ待ち行列サーバー（as-dtaq / QZHQSSRV、8472/9472）へ
生ソケット直結**で、既存のホストサーバー機能と同じ signon / transport / port-mapper を共有する。

## 目的 / ゴール

データ待ち行列を **core のプロトコル層から Web UI まで**扱えるようにし、
IBM i のプログラム間連携を Web から観測・操作できるようにする。

具体的には:
- 待ち行列に**データを積む**（送信）、**取り出す／覗く**（受信／ピーク）
- 待ち行列を**作る・空にする・消す・属性を見る**
- FIFO / LIFO / キー付きのすべてに対応する

## スコープ

### 対象

- **core プロトコル層**: データ待ち行列サーバー（`0xE007`）への接続と、以下の要求
  - 送信（SNDDTAQ 相当）
  - 受信（RCVDTAQ 相当）— FIFO/LIFO の取り出し、キー付きのキー検索、ピーク（消費せず覗く）
  - 作成 / クリア / 削除 / 属性取得
  - 待機時間の指定（すぐ返す / 秒数待つ / 無限に待つ）
- **MCP ツール**: 上記を MCP から操作できるよう公開する
- **Web UI パネル**: 待ち行列の指定、エントリの一覧・送信・受信を UI から行う
- FIFO / LIFO / キー付きキューの区別を扱う

### 対象外

- 出口プログラム（データキューの exit program）連携
- キューのオブジェクト権限（authority）の変更 UI
- DDM 経由のリモートデータキュー
- 送信者情報（save sender information）の構造分解 —
  原典（JTOpen）でも 36 バイトを文字列化するだけで構造解釈していないため、そのまま踏襲する

## 機能要件

### 送信

- 指定した待ち行列（ライブラリ + 名前）にデータを積める
- キー付きキューには、キーを付けて積める
- テキストと、バイナリの両方を積める（データ本体は文字コード変換の要否を扱う）

### 受信

- 待ち行列からデータを取り出せる（FIFO/LIFO はキュー種別に従う）
- **ピーク**（消費せずに先頭を覗く）ができる
- キー付きキューでは、キーと検索条件（一致・不一致・より大きい 等）で取り出せる
- 待機の指定ができる — すぐ返す / 指定秒数待つ / 無限に待つ
- **空だった場合は「エラー」ではなく「空」として扱える**

### 管理

- 待ち行列を作れる（FIFO/LIFO/キー付き、最大エントリ長、説明などの指定）
- 待ち行列を空にできる（キー付きは特定キーだけのクリアもできる）
- 待ち行列を削除できる
- 待ち行列の属性（種別・最大エントリ長・キー長など）を取得できる

### UI / 操作

- 待ち行列を指定してエントリを一覧できる
- UI から送信・受信・ピークができる
- 送信者情報が有効なキューでは、それを表示できる（構造は分解せず、そのまま）

## 非機能要件 / 制約

- 認証・接続は既存のホストサーバー機能（SQL・IFS 等）と同じ資格情報経路に乗せる。DTAQ 専用の認証は作らない
- **プロトコルの応答レイアウトを推測で決めない。** 過去に「宣言テンプレート長からデータ開始位置は
  求められない。整理を試みたが実機で往復が壊れ固定配置に差し戻した」という失敗が記録されている
  （IFS 作業）。データ待ち行列の受信応答も同じ罠を踏みうる
- **無限待ち（wait=-1）を扱えること。** ソケットの read タイムアウトが自前実装を先に切らないよう、
  待機の設計を成立させる
- 名前・ライブラリ・検索タイプ・説明は EBCDIC。PUB400 は CCSID 273 で、variant 文字（`@` `#` `$`）が
  化けて CPF になる実績がある。文字コードの扱いを誤らないこと
- 最大エントリ長は 64512 バイト（プロトコルの上限）。この歯止めを持つ

## 完了条件 (受け入れ基準)

- [ ] 実機（PUB400）で、作成した待ち行列に送信し、受信して**同じ内容が取り出せる**
- [ ] 応答レイアウト（受信正常応答のデータ開始位置・送信者情報）が実測に基づいて確定している
- [ ] ピーク（消費せず覗く）が実機で動き、覗いた後もエントリが残っている
- [ ] キー付きキューで、キーと検索条件を指定して目的のエントリが取り出せる
- [ ] FIFO と LIFO で取り出し順が期待どおりになる
- [ ] 無限待ち（wait=-1）で、後から積まれたデータを受信できる（先にタイムアウトしない）
- [ ] 空の待ち行列を受信したとき、エラーではなく「空」として扱われる
- [ ] 作成・クリア・削除・属性取得が実機で動く
- [ ] MCP ツールから送受信できる
- [ ] Web UI パネルで、待ち行列を指定してエントリの一覧・送信・受信ができる
- [ ] 既存のホストサーバー機能のテストが壊れていない
- [ ] 自前プロトコルの結果と、QSYS2 の SQL サービス（`DATA_QUEUE_ENTRIES` 等）の結果が一致する

## 未確定事項 / 確認したいこと

いずれも **research 工程での実機採取・原典確認で解消したい**（推測で埋めない）。

- **受信正常応答（0x8003）のレイアウト**: 送信者情報 36 バイトの位置、エントリ（CP=0x5001）の
  実際の開始オフセット。原典は offset22/58 を仮定するが、宣言テンプレート長から導けないため実測が要る
- **無限待ちの成立条件**: wait=-1 のとき、ソケット read タイムアウトをどう扱えば自前実装が
  切れずに待てるか。ホストサーバーが待機中に何か（キープアライブ等）を送るのか
- **空キューの応答**: 0x8002 rc=0xF006 が実機で本当にそう返るか。タイムアウト（0秒待ちで空）と
  無限待ち中断の違い
- **エントリデータの文字コード**: プロトコルは生バイト。テキストとして扱うとき、システム CCSID
  （PUB400 は 273 / 日本語機は 5026・1399 等）でどう変換するか。IFS のテキストと同じ論点
- **UI の位置づけ**: 待ち行列は「見る」より「プログラム間連携の道具」。UI で何を主目的にするか
  （デバッグ用の覗き見か、実運用の投入か）
