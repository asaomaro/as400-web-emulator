# 要件: CSV を IBM i の物理ファイルへ取り込む（Web UI / MCP）

## 背景 / 課題

ACS のデータ転送に相当する機能のうち、**ダウンロード側（IBM i → CSV）は 2026-07-19 に完了**している
（`20260719-hostserver-web-ui`。SQL ペインで実行して CSV を落とせる）。
**アップロード側（CSV → IBM i）は未着手**で、往復の片道しか通っていない。

DDM による物理ファイルへの追記は core に実装済み（`20260719-hostserver-upload-ddm`。実機で書いて
SQL で読み返し、負数・小数・ゼロ・空文字・NULL の一致を確認済み）だが、**この機能を使える経路が
どこにも無い**。現状の利用者はリポジトリ内の検証スクリプト `tools/hostserver-check/src/ddm.ts` だけで、
サーバー API も MCP ツールも Web UI も存在しない。

つまり「土台はあるが誰も使えない」状態であり、本作業でその経路を通す。

## 目的 / ゴール

- Web UI に CSV を渡すと、指定した IBM i の物理ファイルへ行が追記される。
- 同じ機能を MCP ツールからも呼べる（エージェント経由の取り込み）。
- 日本語を含む実データが**壊れずに**書ける。書けない場合は黙って壊さず明示的に失敗する。

## スコープ

### 対象

1. **core**: 列メタデータ取得の共通化（現在は検証スクリプト内にのみ存在する
   `QSYS2.SYSCOLUMNS` 問い合わせを、製品コードとして使える形にする）。
2. **core**: 書き込み時の **CCSID を可変**にする（現在 CCSID 37 が直書き）。
3. **core**: **バッチ書き込み**（ブロッキング係数 > 1）。現在は 1 レコード 1 往復。
4. **server**: アップロードの HTTP API。
5. **server**: 同機能の **MCP ツール**公開（backlog「MCP ツールとして公開」を本作業で消化）。
6. **web-ui**: CSV パーサー（現在の `csv.ts` は生成専用）。
7. **web-ui**: 取り込み UI（表の指定、CSV の受け渡し、進捗、結果表示）。

### 対象外

- **読み取り・更新・削除・キー付きアクセス**（SQL があるため動機が弱い。追記のみを維持する）。
- **DDM 経由での表の作成**。取り込み先の物理ファイルは既に存在する前提。
- **コミットメント制御**（DDM 層に無い。「途中で失敗したら書けた分は残る」を仕様として受け入れる）。
- **`VARCHAR` / 日付時刻列への対応**。現在 `record-layout.ts` が明示的に拒否している型のうち、
  本作業で広げるのは文字コードの問題（CCSID）だけで、**型の対応範囲は広げない**。
- ダウンロード側の変更（完了済み）。

## 機能要件

- 取り込み先の **ライブラリ / ファイル（/ メンバー）** を指定できる。
- CSV を渡すと、**ヘッダー行と対象表の列を対応づけて**行を追記する。
- 対象表の列型が対応範囲外（`VARCHAR` / 日付時刻 / DBCS 等）の場合、**取り込みを開始する前に**
  その旨と該当列を示して中止する（1 行も書かずに止まる）。
- 文字が指定 CCSID で表現できない場合、**置換文字を書かずに失敗させる**（現在の core の方針を維持）。
- **途中失敗時に「何行目まで書けたか」を利用者に示す**。巻き戻せない以上、再開・重複投入の判断は
  利用者が行うため、その材料を必ず返す。
- MCP ツールからも同じ取り込みができる。
- 進捗が見える（行数の多い取り込みで無反応にならない）。

## 非機能要件 / 制約

- **性能**: **接続確立が 4〜7 秒**（PUB400・インターネット越し・TLS）で、**確立後の 1 往復は 1 秒未満**
  （T8 の実機確認で 724ms。当初「1 往復 4〜7 秒」と書いていたのは誤り。`decisions.md` D2）。
  DDM 接続は signon に加えてポート 446 への別接続を張るため、**1 回の取り込みで 2 回接続する**。
  バッチ書き込みにより、行数に比例した往復を避ける（100 件なら 100 往復＝70 秒超が 1 往復に収まる）。
- **文字コード**: PUB400 は CCSID 273。37 固定のままでは `@` 等のバリアント文字も化ける
  （既知の落とし穴として記録済み）。
- **セキュリティ / 認可**: 既存のホスト API と同じく、**接続を持つユーザーなら誰でも**呼べる方針を
  踏襲する（見える範囲・書ける範囲は IBM i 側の権限が決める）。
  ただし本作業は `host-sql.ts` が明文化していた「**この API は構造的に読み取り専用**」という
  不変条件を破る。同ファイルは「更新系を通す改造を入れるときは必ずこの方針を再検討すること」と
  警告しており、**その再検討をここで行い、判断を記録する**（設計判断として design/decisions に残す）。
- **入力の安全性**: 現在の列メタデータ問い合わせはライブラリ名・表名を SQL に文字列連結している。
  製品コードでは連結しない形にする。
- **既存 UI との衝突**: ワークスペースはペイン分割・タブ移動で `dragover` / `drop` を既に使っている。
  ファイルのドラッグと取り違えない形にする。
- **列順序**: レコードのレイアウトは列の宣言順に依存する（`ORDER BY ORDINAL_POSITION` が要）。
- **レイアウト計算は仮説**: `record-layout.ts` は「仮説に基づく実装」と自ら明記しており、
  ホスト申告の `recordLength` との一致でしか検証されていない。この照合を実行時にも維持する。

## 完了条件 (受け入れ基準)

- [ ] CSV を渡して、実機の物理ファイルに行が追記される（**SQL で読み返して一致を確認**する。
      ダウンロード側の検証と同じやり方）。
- [ ] **日本語を含む行**が書け、読み返して一致する（CCSID を対象ホストに合わせた場合）。
- [ ] 表現できない文字を含む CSV は、**置換文字を書かずに**失敗する。
- [ ] 対応外の型の列を持つ表を指定すると、**1 行も書かずに**列名つきで拒否される。
- [ ] 100 行規模の取り込みが、1 行 1 往復の場合より明確に速い（バッチ書き込みの効果を実測で示す）。
- [ ] 途中で失敗したとき、**何行目まで書けたか**が利用者に伝わる。
- [ ] MCP ツールから同じ取り込みができる。
- [ ] ペイン分割・タブ移動の D&D が、ファイルのドラッグで誤動作しない。
- [ ] 既存テストが通り、追加分の自動テストがある（実機を要する部分は手順を記録する）。

## 未確定事項 / 確認したいこと

- **バッチ書き込みの要求形式**（ブロッキング係数 > 1 でのレコード送信）。DDM の原典で確認が必要。
  現在の実装は 1 件ずつで、まとめて送る形式を**確かめていない**。
- **CCSID の決定方法**。利用者に選ばせるのか、ホストのジョブ CCSID から引くのか、表の列 CCSID を
  見るのか。`QSYS2.SYSCOLUMNS` に列単位の CCSID があるかを含めて要確認。
- **DBCS 列の扱い**。対象外としているが、日本語を書くこと自体が目的に含まれるため、
  「SBCS 列に日本語を入れる」で足りるのか、DBCS 列が要るのかを確認する必要がある。
- **行数の上限**。バッチ化後の実測を見てから決める。
- **CSV の文字コード入力側**（UTF-8 前提でよいか、Shift_JIS を受けるか）。

> 上記のうち **バッチ書き込みの要求形式** と **CCSID の決定方法** は、ユーザーへの質問ではなく
> **原典・実機で確かめるべき事項**である。仮定のまま spec に進むと手戻りが大きいため、
> research 工程を挟むことを推奨する。
