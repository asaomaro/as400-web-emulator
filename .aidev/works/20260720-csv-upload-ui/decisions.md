# 決定記録

## D1: アップロード専用ペインをやめ、取得と取り込みを持つ「データ転送」ペインにする

- 背景: coding 着手直後（T1 実装中）に、ユーザーから「アップロード画面は SQL アプリと共通か」と問われた。
  当初の design は `UploadPane.vue`（`upload:table`）という**取り込み専用**の新規ペインで、
  取得（CSV ダウンロード）は既存の SQL ペインに残したままだった。
  ユーザーの指摘は「SQL アプリは ACS の **Run SQL Scripts** の位置づけなので、
  取得と取り込みをまとめた **Data Transfer** 相当のアプリにすべきではないか」。
- 決定: `TransferPane.vue`（タブ ID `transfer:data`）とし、**方向切替（取得 ⇄ 取り込み）を持つ 1 つのアプリ**にする。
  SQL ペインは Run SQL Scripts のまま据え置き、実行結果の CSV エクスポートも残す。
- 理由 / 代替案:
  - 当初案は**実装の現状**（「ダウンロードは既に SQL ペインにある」）から出発しており、
    概念ではなく既存コードの都合で境界を引いていた。結果として取り込みだけが別ペインという非対称になっていた。
  - ACS でも SQL 結果のエクスポートは Run SQL Scripts 側にあり、専用の Data Transfer と共存している。
    よって「SQL ペインの CSV ダウンロードを残す」ことと「データ転送アプリを作る」ことは矛盾しない。
  - 退けた案: SQL ペインに取り込みモードを統合する。SqlPane は既に 696 行あり、
    流れ（SQL を書く / ファイルを投入する）も別物。境界が曖昧になる。
- 影響:
  - **core / server の設計は変更なし**。事前検査・バッチ書き込み・CSV 解析・API はそのまま使える。
  - design.md の UI 節、spec.md の対象範囲、tasks.md の web-ui タスクを更新。
  - 取得側はサーバーの新規実装が不要（既存 `POST /api/host/sql` に組み立てた SELECT を投げる）。
    ただしライブラリ名・表名の検証は取り込み側と同じ関数を使う。
  - 経緯: coding → design への差し戻し（`aidev event coding sent_back` → `aidev event design start`）。
    デザイン案を HTML モックで提示し、承認を得てから再開した。

## D2: 「1 往復 4〜7 秒」は誤り。**接続確立**が 4〜7 秒で、往復自体は 1 秒未満

- 背景: requirement・design に「実機は 1 往復 4〜7 秒」と書き、UI に想定往復数を出す根拠にしていた。
  T8 の実機確認で **3 件を 1 往復・724ms**（`writeAll` 全体）と出た。桁が違う。
- 決定: 数値を訂正する。**接続確立（signon ＋ 446 への別接続）が 4〜7 秒**で、
  確立後の 1 往復は **1 秒未満**。
- 理由 / 経緯: 出典は backlog の記述
  「PUB400（インターネット越し・TLS）では **4〜7 秒/呼び出し**で、処理量に比例せず**接続確立が支配的**だった」。
  原文は「呼び出し（＝接続込み）」と言っており、しかも「接続確立が支配的」と明記していた。
  それを「往復」と読み替えて転記したのは**私の誤読**である。元の文自体が答えを書いていた。
- 影響:
  - **バッチ化の価値は下がらない**。1 往復 724ms でも 100 件で 100 往復なら 70 秒以上かかる。
    1 往復に収まれば 1 秒未満で、差は依然として大きい。
  - **UI の「想定往復数」は残す**が、待ち時間の見積もりは
    「接続 4〜7 秒 ＋ 往復数 × 1 秒未満」という組み立てになる。**支配的なのは接続のほう**なので、
    ステータスバーでは往復数より「接続に数秒かかる」ことのほうが利用者には効く。
  - requirement.md / design.md の該当箇所を訂正した。
  - T19(d) の性能確認は、この訂正後の数値で測り直す。

## D3: `HOST_SERVER_UNSUPPORTED` を一律 400 にせず、`UNSUPPORTED_TYPE` を新設する

- 背景: spec の「エラー処理」で「`HOST_SERVER_UNSUPPORTED` は現在 502 に落ちるが、
  対応外の列型は利用者側の入力の問題なので **400 に写像し直す**」と決めていた。
  T10 で実装しようとしたところ、このコードが**2 つの異なる意味**で使われていた。
- 決定: 一律写像はやめ、**`UNSUPPORTED_TYPE` を新設**して 400 に写す。
  `HOST_SERVER_UNSUPPORTED` は 502 のまま。DDM の対応外列型は新コードに切り替えた。

  | 用途 | 意味 | 写す先 |
  |---|---|---|
  | `password.ts`（DES パスワードレベル） | ホストが要求する機能を**こちらが実装していない** | 502（据え置き） |
  | `command-connection.ts`（古いデータストリーム） | 同上 | 502（据え置き） |
  | `record-layout.ts`（対応外の列型） | **利用者が選んだ表**の問題 | **400**（新コード） |

- 理由 / 代替案: 一律 400 にすると「DES 非対応」が `Bad Request` として返る。
  利用者には直せない問題を「直せる」かのように案内することになり、
  `host-api.ts` の既存コメント（「呼び出し側が『ホストが落ちている』のか
  『指定が間違っている』のかを区別できなくなる」）が禁じている状況そのものになる。
  退けた案: spec のとおり一律 400。実装は 1 行で済むが、上記の取り違えが起きる。
- 影響: `ErrorCode` に 1 つ追加（core の公開型）。既存の写像は変えていないため、
  既存テストへの影響なし（server 318 件・core 611 件とも通過）。
  `db-decode.ts`（SQL の対応外型）は**今回触っていない**——本作業の範囲外で、
  変えるなら SQL 経路のテストと合わせて別途行う。

## D4: DDM は取り込みの土台として誤り。本作業は「SBCS 表まで」で着地し、SQL INSERT へ移る

- 背景: T19 の実機検証で 2 つの壁に当たった（`research.md` F12）。
  - **DBCS 対応 CCSID（5035/930）の列を持つ表は DDM で開けない**（`CPF427D` ×2）。
    UFCB は原典と完全一致で実装漏れではない。
  - DDS 作成の物理ファイルは様式名がファイル名と違い、指定しないと開けない（`CPF4135`）。
  これにより requirement の受け入れ基準「日本語を含む行が書け、読み返して一致する」が満たせなくなった。
- **前提の検証**（ユーザーの「仕組み上、DDM を挟む必要があるのか？」という問いから）:
  ACS でデータ転送を実行し、接続先ポートを実測した（`research.md` F11）。

  ```
  TCP  192.168.0.95:53202  185.113.5.134:8471  ESTABLISHED
  ```

  **本家 ACS は database サーバー（8471）を使っており、DDM（446）ではなかった。**
  ACS のサービス一覧でも「データベース サービス（8471）」と
  「レコード・レベル・アクセス サービス（446）」は別項目である。
- 決定:
  1. **本作業はここで着地させる**。DDM 経路のまま「SBCS（CCSID 273 等）の表への取り込み」として
     PR にする。動いている範囲（(a)(c)(d) 実証済み）は利用者価値がある。
  2. **取り込みの土台を SQL INSERT へ移す作業を別 work として起こす**。
- 理由 / 代替案:
  - DDM を選んだ根拠は前作業 requirement の「`query` は SELECT 専用で、`RUNSQL` は 1 文ずつ」——
    これは **IBM i の制約ではなく我々の実装の制約**だった。backlog の項目名
    （「`jtopenlite/ddm/` 相当のレコードレベルアクセス」）がそのまま土台の選定になり、
    **必要性が一度も検証されないまま**進んでいた。research 工程で問うべき論点だった。
  - SQL INSERT にすると、本作業で当たった 4 つの壁が構造的に消える:
    CCSID 変換（ホストが行う）・レコード様式名（無関係）・型制限（VARCHAR/日付時刻も通る）・
    レイアウト計算の仮説（不要）。
  - 退けた案: DDM のまま CPF427D の回避策を探す。**本家が通っていない道**であり、
    原典にも手掛かりが無い。
  - 退けた案: 直ちに作り直して 1 つの PR にする。差分が大きくなりすぎ、
    今日実証した DDM バッチ書き込みの価値（backlog「レコード・レベル・アクセス」として有効）も
    埋もれる。
- 影響:
  - requirement の受け入れ基準「日本語」「対応外の型」は**次 work へ引き継ぐ**。
    本作業の完了条件からは外し、制約として明記する。
  - core の DDM 実装（バッチ・列単位 CCSID）は**残す**。backlog のレコード・レベル・アクセス項目に
    紐づく資産で、SQL INSERT 化後も無駄にならない。
  - `recordFormat` を API / MCP / UI に通した（DDS 由来ファイルの回避手段）。
