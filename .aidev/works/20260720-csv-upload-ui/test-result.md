## 実機検証（PUB400）

`tools/hostserver-check/src/upload.ts`。サーバーの `uploadRows` をそのまま呼ぶので、
HTTP と MCP が通る道と同一。**書いた経路（DDM）と確かめる経路（SQL）を分けている**。

```
node --env-file=.env dist/upload.js --tls
```

### (a) CSV から投入 → SQL で読み返す ✅

```
{"ok":true,"committedRows":3,"batchSize":100,"ms":11236}
{"ID":1,"NAME":"alpha     ","AMT":"12.34"}
{"ID":2,"NAME":"beta      ","AMT":"-0.50"}
{"ID":3,"NAME":null,"AMT":"0.00"}
```

負数・小数・空欄→NULL（`emptyAsNull`）がすべて往復した。

### (b) 日本語（CCSID 5035 / 930） ❌ **未達**

```
❌ [PROTOCOL_ERROR] ファイルを開けませんでした（MARO1/ZZUP2）:
   CPF427D Substitution characters may be used in data conversion. ×2
```

**DDM の制約**であり実装の不備ではない（`research.md` F12・`decisions.md` D4）。
UFCB は原典（jtopenlite `sendS38OpenRequest`）と完全一致。
CCSID だけが違う表（273 のみ）は同じ手順で開けている。

→ requirement の完了条件「日本語を含む行が書け、読み返して一致する」は**満たせていない**。
次 work（SQL INSERT 化）へ引き継ぐ。

### (c) 書けない文字は 1 行も書かずに拒否 ✅

```
{"ok":false,"rejections":[{"kind":"value-unencodable","row":2,"column":"S",
                           "chars":["日","本","語"],"ccsid":273}],"truncated":false}
表の行数: 0（0 であること）
```

**どの文字が書けないかを特定**し、置換せず拒否。表は 0 行のまま。

### (d) 100 行の性能 ✅

```
{"ok":true,"committedRows":100,"batchSize":100,"ms":11633} / 実測 11633ms（接続を含む）
表の行数: 100
往復数: 1（1 件 1 往復なら 100 往復）
```

**100 行が 1 往復**。全体 11.6 秒のうち大半は接続確立で、
1 件 1 往復なら `11 秒 + 100 × 約 0.7 秒 ≈ 80 秒` になる計算。**約 70 秒の短縮**。

これは `decisions.md` D2（「1 往復 4〜7 秒」は誤りで、4〜7 秒は接続確立）の裏付けにもなっている。

### (e) 取得側の CSV 保存 — **未検証**

ブラウザでの操作が要るため実施していない。SQL の読み取り経路（日本語を含む）は
`research.md` F10 で往復を確認済みだが、**UI のダウンロード導線は目視していない**。
