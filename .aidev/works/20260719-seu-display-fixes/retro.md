## 課題 / 手戻り

### 仮説を証拠より先に進めた（3 回）

F1 ハングの原因究明で 3 回外した。

1. 「F1 が無効でキーボード禁止」→ ACS がヘルプを出すので否定
2. 「レコードを取りこぼして捨てている」→ **この否定自体が誤り**だった。
   ヘルプ画面（+71.31 秒のレコード）が正しく描けることは、+0.28 秒のレコードについて何も示さない
3. 「未知コマンドで `default:` に落ちている」→ コマンドは既知で処理も通っていた

前段で防げたか: **防げた。** 実機トレース（`--trace-records`）を最初に取っていれば 12 バイトの
`SAVE SCREEN` が即座に見えた。推論を 3 周する前に観測手段を用意すべきだった。

### 到達しないコードを実装した

Tab の件で「保護欄にフォーカスがある状態」を前提に実装したが、このアプリは
`onCursor` が入力欄を blur してペインへ移すため、**保護欄に focus は留まらない**。
テストが通らず DOM を調べて初めて判明し、実装ごと作り直した。

前段で防げたか: **防げた。** 「保護欄にフォーカスが当たる」を確かめずに前提化した。
既存の `reconcileFocus` を読めば分かる範囲だった。

### 何も守っていないテストを書いた

「入力欄が無い画面で Tab → 1 行 1 桁へ」のテストが、修正前のコードでも通っていた。
テスト用スナップショットのカーソルが常に `1,1` で、変化を検証していなかった。

前段で防げたか: **防げた。** 「修正前に落ちるか」を確かめる手順は踏んでいたが、
落ちない 1 件を「挙動が一致するため」と早合点して見逃しかけた。
前提側にも `expect` を置く（`toEqual({row:8,col:20})`）ことで解決した。

### 記録の欠落

ハーネスを通していないため `metrics.yml` / `review.md` が無い。
手戻り 3 回・実装やり直し 1 回という事実は会話ログからしか復元できず、
次回同種の作業と比較できない。

## 発見: 同型の不具合が 2 件あった

#86（下線）と #88（カーソル）は、症状も層も違うが**原因が同じ型**だった。

> **ホストが送っている情報を、こちらが無条件に上書きしていた。**

- 下線: ホストが下線属性を送っているのに、入力欄なら常に `border-bottom` を引いていた
- カーソル: ホストが桁まで指定しているのに、常に欄の先頭・第 1 欄へ寄せていた

さらに #86 には**真実の二重化**もあった。`hidden` を SF 記録時の属性バイトから、
セルの `nonDisplay` を画面上の実効属性から導いており、同じ桁について 2 つの答えがあった。

これらは個別の不具合ではなく**傾向**であり、同じ型が他にも眠っている可能性が高い。

## 改善提案

### 製品 / コード（→ issue 候補）

- **「ホスト由来の情報を UI 側で上書きしていないか」の棚卸し。** 色・反転・下線・非表示は
  今回対処したが、blink / columnSeparator / 保護属性の見せ方は未点検。
  同型の不具合が残っている可能性がある。
- **DBCS の SAVE SCREEN 応答が不完全。** lead/tail に元のバイト対を保持していないため、
  エンコードし直せない文字は空白になる（#85 のコメントに記載）。現状 RESTORE は
  ローカルの退避スタックから復元するため実害は無いが、前提が崩れると壊れる。
- **非表示欄とパスワード欄を区別できない。** ACS 準拠で伏せ字を廃止したが、
  属性から両者を分ける根拠が見つかっていない。区別が必要になったら調査からになる。

### PJ プロセス / 規約（→ AGENTS.md）

- **ホストが送る情報を UI 側の既定値で上書きしない。** 属性・カーソル位置・
  フィールド定義はホストが真実であり、見た目の都合で無条件に決め打ちしない。
  上書きする場合は理由をコメントに残す。
- **同じ事実に対する導出元を 2 つ持たない。** `hidden`（SF 属性バイト由来）と
  `nonDisplay`（実効属性由来）のように、同じ桁について別経路で答えを作らない。
  片方を唯一の真実として他方はそこから導く。
- **プロトコルの切り分けは観測を先に用意する。** 「ホストが何を送ったか」が争点になったら、
  推論を重ねる前に生レコードを取る手段（`--trace-records`）を先に確保する。
- **ガードテストは「修正前に落ちる」ことを必ず確認する。** 落ちない場合は
  「挙動が一致するため」で済ませず、前提が検証されているかを疑う。
  前提側にも `expect` を置く。

### ハーネス自体（→ aidev-* への提案・適用は人間）

- **不具合修正の軽量入口が無い。** 今回のような「報告 → 調査 → 修正 → PR」を
  requirement から回すのは重く、結果としてハーネスを通さず記録が残らなかった。
  調査ログと根拠を残すことに絞った軽量パス（例: `aidev-05-bugfix`）があれば、
  外した仮説や決め手になった証拠を構造化して残せた。
- **retro を事後作成した work で回す経路が曖昧。** `.aidev/current` は別作業を指しており、
  対象を切り替えるために work を新規作成した。ハーネス外の作業を後から記録する手順が
  protocol に無い。「記録欠落の検知」（retro 手順3）は欠落を書けと言うが、
  そもそも works が無い場合の扱いが未定義。
