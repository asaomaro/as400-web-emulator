# 要件: API トークンの自己発行と再発行による失効

## 背景 / 課題
- `issueToken`（`auth.ts:134`）は `tokenHashes` に**追加**するため、発行するたび有効なトークンが増える。
  どれがいつ発行されたか分からず、漏洩時に**特定して失効できない**（全消ししかない）。
- 発行経路は `POST /api/admin/users/:username/token` の **admin 専用**のみ。一般ユーザーは
  MCP を使いたくても自分のトークンを作れず、admin への依頼が必要。

## 目的 / ゴール
**1 ユーザー 1 トークン**とし、再発行で以前のトークンを失効させる。
各ユーザーが**自分のトークンを自分で発行**できるようにする。

## スコープ
### 対象
- `issueToken` を追加から**置き換え**に変更（再発行＝以前の失効）。
- 自己発行 API（`POST /api/me/token`）。自分の分のみ。
- アカウント UI（ヘッダーのユーザー名から開く）で発行・コピー・失効警告・発行状態表示。
- ドキュメント: MCP クライアントへのトークン設定手順。

### 対象外
- **有効期限**（今回は不要と確認済み）。
- OAuth 2.1 等の対話的認可フロー。
- **MCP ツールでの資格情報受け渡し**（D13 違反。トークンはトランスポート層のヘッダで渡す）。
- admin による他ユーザーへの発行（`/api/admin/...` は現状維持）。

## 機能要件
- 再発行すると**以前のトークンは即座に無効**になる。
- 一般ユーザーも自分のトークンを発行できる。他ユーザーの分は発行できない。
- 平文トークンは**発行時に 1 回だけ**返す（保存はハッシュのみ）。
- UI は発行**前**に「以前のトークンが使えなくなる」と警告する（押した後では遅い）。
- UI は発行済み / 未発行を表示する。
- 既存 `users.json`（`tokenHashes` が複数ある状態）はそのまま読める。

## 完了条件 (受け入れ基準)
- [ ] 再発行後、以前のトークンで認証できない。新しいトークンでは認証できる。
- [ ] 一般ユーザーが `POST /api/me/token` で自分のトークンを発行できる。
- [ ] 未認証では発行できない（401/403）。
- [ ] 平文トークンは応答にのみ含まれ、`users.json` にはハッシュしか残らない。
- [ ] 複数 `tokenHashes` を持つ既存ファイルが読め、再発行すると 1 本に置き換わる。
- [ ] UI に失効警告と発行状態が出る。
- [ ] README にトークンの設定手順がある。
