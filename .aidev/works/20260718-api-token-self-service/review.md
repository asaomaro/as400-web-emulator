# レビュー: API トークンの自己発行と再発行による失効

## ラウンド 1

### 観点別チェック
- **失効**: `issueToken` が配列を置き換え。旧トークンで `/api/me` の user が null になることをテストで固定。
- **権限**: `/api/me/token` はパスパラメータを取らず `c.get("user")` からユーザー名を得る＝他人の分を発行できない構造。
  未認証は 401、認証オフは 400。
- **秘密の扱い**: 平文はレスポンス限り。`users.json` に平文が含まれないことをテストで確認。
  `/api/me` の応答に 48 桁の hex が現れないことも確認（`hasToken` は真偽値のみ）。
- **後方互換**: `tokenHashes` は配列のまま。複数要素の既存ファイルを読めて、発行で 1 本に収束することを確認。
- **UI 配置**: 接続編集画面ではなくアカウントに置いた（トークンはユーザーの資格情報でコネクション単位ではない。
  編集画面に置くと誤解と誤失効を招く）。失効警告は発行**前**に常時表示。

### 指摘
- [nit] `POST /api/me/token` は CSRF 対策を明示していない。Cookie 認証で状態変更を行うため理屈上は対象だが、
  既存の `/api/profiles` 等の変更系と同条件であり、本 PR だけで扱う話ではない。**未対応（別課題）**。
- [nit] 有効期限は今回対象外（ユーザー確認済み）。漏洩時は再発行で失効させる運用。

### 判定
must=0 / should=0 / nit=2。deliver へ進む。
