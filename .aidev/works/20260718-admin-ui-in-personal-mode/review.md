# レビュー: 個人利用モードでも管理画面（セッション・ログ）

## ラウンド 1

### 観点別
- **要件適合**: 認証オフで sessions/logs が 200、users は未登録。認証オン（admin 3 つ / 一般 403 / 未ログイン 401）は回帰テストで担保。
- **規則の一貫性**: `requireAdmin(authEnabled)` は「認証オフ＝全通過」で `assertOwner` / `assertProfileAccess` と揃えた。
  引数の既定値を `true` にしたため、**呼び忘れても安全側（403）に倒れる**。
- **露出の縮小**: ユーザー管理はルートごと登録しない。認証オフでは `users` が undefined になり、
  `registerUserRoutes` 自体を呼ばない構造にした（ハンドラ内で分岐すると漏れやすい）。
- **監査ログの常時収集**: `installAuditBuffer` を無条件にした。個人利用でもログが見えるようになる副作用は意図的。

### 実挙動の確認で判明したこと
`--web-root` 付きで起動すると `/api/admin/users` が **200** を返した。調べると **SPA フォールバックの HTML**
（`content-type: text/html`）で、API は未登録のまま。`--web-root` 無しでは 404。ユーザー情報は漏れていない。
テスト（web-root 無し）の 404 と実機（web-root 有り）の 200 は**どちらも正しい**。

- UI は認証オフでユーザー管理ボタンを出さないため、この経路を呼ぶことはない。
- 仮に呼んでも HTML が返るだけで、ユーザー一覧は取得できない。

### 指摘
- [nit] SPA フォールバックが `/api/*` の未登録パスまで拾うのは一般論としては行儀が悪い（404 が欲しい）。
  ただし本 PR の変更点ではなく、既存の全 API に共通する挙動。別課題とする。
