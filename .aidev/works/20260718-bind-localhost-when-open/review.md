# レビュー: 認証オフの HTTP を localhost に束縛する

## ラウンド 1

### 実挙動の確認（単体テストだけに頼らない）
バインドは「テストが通る」だけでは担保できないため、実際に起動して LAN IP から到達性を確認した。

| 起動 | localhost | LAN IP (172.17.139.237) |
|---|---|---|
| 認証オフ（既定） | 200 | **000＝到達不可** |
| 認証オン `--users` | - | 200（従来どおり） |
| 認証オフ ＋ `--host 0.0.0.0` | - | 200（警告ログ付き） |

起動ログにバインド先と `auth` の有無が出ることも確認。

### 観点別
- **後方互換**: 認証オンの挙動は不変。Electron（`127.0.0.1` へ接続）・start.sh（localhost 案内）とも影響なし。
- **禁止ではなく警告**: リバースプロキシ配下など認証を外側で持つ正当な構成があるため、`--host` での公開は
  許可し警告に留めた。
- **判定は純関数**に切り出し、main.ts（CLI・テスト困難）から分離した。

### 指摘
- [nit] IPv6 の `::`（全インターフェース）は `isLoopback` で false になり、認証オフでの明示指定時は警告対象。
  意図どおりだが、`::1` との取り違えに注意が必要。テストで両方を固定した。
- [nit] `--host` の値そのものの妥当性（解決可能なアドレスか）は検証していない。誤った値は `serve` が
  起動時に失敗するため、二重に検証する必要はないと判断。

### 判定
must=0 / should=0 / nit=2。deliver へ進む。
