# 要件: 認証オフの HTTP サーバーを localhost に束縛する

## 背景 / 課題
`main.ts:140` の `serve({ fetch, port, websocket })` は **`hostname` を指定しておらず**、
`@hono/node-server` の既定で **全インターフェース（0.0.0.0）** を待ち受ける。

認証オフ（`--users` 無し）はコード全体で「**単一の信頼ユーザー**」を前提にしており、
`canEditProfiles` は全通過、`assertProfileAccess` も素通しになる。その状態が LAN に露出すると、
同一ネットワークの誰でも次を実行できる。

- サーバー設定の閲覧・編集
- `autoPdfDir` を任意パスに設定 → **サーバーのファイルシステムへ書き込ませる**
- `autoPrint` に任意名 → **`lp` を実行させる**
- 保存済み資格情報を使った IBM i への接続

信頼境界の前提がネットワーク的に担保されていない。**実在の脆弱性**である。

## 目的 / ゴール
認証オフなら既定で **127.0.0.1 のみ**を待ち受ける。公開したい場合は明示操作を要求する。

## スコープ
### 対象
- 認証オフ時のバインド先を既定で `127.0.0.1` にする。
- 認証オン（`--users` 指定）時は従来どおり全インターフェース。
- `--host <addr>` を追加し、明示指定を可能にする。
- 認証オフのまま非ループバックへ明示公開した場合は**警告ログ**を出す（禁止はしない）。
- 起動ログに実際のバインド先を出す。

### 対象外
- stdio モード（ネットワークを開かない）。
- 認証方式・認可規則の変更。
- TLS 終端やリバースプロキシ構成の追加。

## 機能要件
- 認証オフ・`--host` 無し → `127.0.0.1`
- 認証オン・`--host` 無し → `0.0.0.0`（従来どおり）
- `--host` 指定時はその値を使う（認証状態に関わらず）
- 認証オフ ＋ 非ループバック指定 → 警告を出したうえで起動する
- 判定は**純関数**に切り出し、単体テストできること
