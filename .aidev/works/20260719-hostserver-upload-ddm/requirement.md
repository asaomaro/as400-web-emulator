# 要件: 物理ファイルへの書き込み（DDM レコードレベルアクセス）

## 背景 / 課題

backlog `hostserver.md` の「**アップロード** — 物理ファイルへの取り込み
（`jtopenlite/ddm/` 相当のレコードレベルアクセス）」。

ここまでの 3 作業で、SQL の**読み取り**は MCP からもブラウザからも使えるようになった。
しかし **IBM i 側へデータを入れる手段が無い**。`query` は SELECT 専用で、
`host_command` の `RUNSQL` は 1 文ずつしか流せない。ACS の「データ転送」は
双方向（ダウンロード／アップロード）であり、片道のままでは代替にならない。

前作業（`20260719-hostserver-web-ui`）で **CSV アップロードの UI を意図的に見送った**のも、
この土台が無いためである（「無い機能の上に UI を作らない」）。本作業がその土台にあたる。

### 直読して分かった重要な事実

**DDM はこれまでのホストサーバーと握手が違う。** signon / database / command /
netprint / IFS はいずれも `0x7001`→`0x7002` の枠（`server-connect.ts` に共通化済み）だが、
DDM は **EXCSAT → ACCSEC → SECCHK** という DRDA 系の独自手順で、
**ポートも 446**（DRDA 標準。8471 系ではない）。

つまり**既存の `startHostServer()` は使えない**。ここが本作業の主要な難所であり、
「認証基盤の上に載るだけ」という他の機能とは前提が異なる。

> なお「DDM の既定ポートは 446」は `20260718-acs-data-transfer` の retro にも
> **実機で踏んだ教訓**として記録されている（8471 系だと思い込んで失敗した）。

## 目的 / ゴール

**既存の物理ファイルにレコードを追記できる**状態にする（`packages/core` のライブラリとして）。

## スコープ

### 対象

- DDM サーバー（446 / TLS）への接続と認証（EXCSAT / ACCSEC / SECCHK）
- **既存ファイルのレコード形式の取得**（列名・型・長さ・オフセット）。
  これが無いとレコードのバイト列を組み立てられない
- **ファイルを書き込みで開く → レコードを書く → 閉じる**
- 数値・文字の主要な型のエンコード（パック 10 進・ゾーン 10 進・文字・可変長）
- CLI（`tools/hostserver-check`）で実機検証できること

### 対象外（この作業ではやらない）

- **読み取り系一式**（`readNext` / キー読み / 位置付け / スレッド読み）。
  原典には豊富にあるが、**SQL で読める**ため動機が弱い。書き込みに絞る
- **ファイルの作成**（`CRTPF` 等）。`host_command` の CL で足りる
- CSV のパースと UI／MCP 公開。**土台ができてから別作業**
  （前作業と同じ理由——無い機能の上に UI を作らない）
- キー付きファイルの更新・削除

## 要検討（spec で決める）

### 1. 型変換の範囲 ★見積もりを厚めに見ること

SQL 実装の retro が「**真の難所はバイト列の組み立てではなくデータ型変換**
（パック/ゾーン10進数・日付時刻・可変長・DBCS混在・NULL指標マップ）」と記録している。
**読み取り側で一度通った道だが、書き込みは逆変換で別の実装**になる。

どの型まで対応するかを spec で決め、**対応しない型は明示的に失敗させる**
（黙って壊れたデータを書かない）。

### 2. 失敗時の安全性 ★最重要

**これは初めての「IBM i にデータを書く」機能である。** これまでは読み取りと
限定的な CL 実行だけだった。途中で失敗したときに中途半端なレコードが残る可能性がある。

- 部分的に書けた状態をどう扱うか（ロールバックできるのか、できないなら何を保証するか）
- 検証は**壊しても構わないファイル**に対してのみ行う

### 3. 既存の transport との関係

`transport/host-connection.ts` は 20 バイトヘッダー＋LL/CP を前提にしている。
DDM のフレーム構造が異なるなら、共有できるのはソケットの層だけかもしれない。
**無理に共通化しない**（前作業で「早すぎる共通化で両者を結びつけない」と判断したのと同じ）。
