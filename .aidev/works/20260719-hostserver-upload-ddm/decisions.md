# 決定記録

## D1. 列レイアウトは SQL から得る（`DDMField` 1,220 行を移植しない）

原典は `DDMRecordFormatReader` ＋ `DDMField`（**1,220 行**・全型の読み書き変換）で列情報を扱うが、
このプロジェクトには**実機で検証済みの SQL** がある。`QSYS2.SYSCOLUMNS` から
列の順序・型・長さ・精度・位取りを取り、そこからレコードバッファの配置を計算する。

**根拠**: (1) 移植量が桁で違う (2) 既に検証済みの経路を使う
(3) **検証が別経路になる**——DDM で書いて SQL で読み返すので、配置計算が誤っていれば露見する。

**これは仮説だった**が、実機で確認できた——ホストが `S38OPNFB` で申告する
`recordLength` と SQL 由来で計算した値が一致し、書いた値が正しく読み返せた
（負数・小数・ゼロ・空文字・NULL を含む）。

## D2. 対応する型を絞り、対応外は明示的に失敗させる

CHAR / DECIMAL / NUMERIC / SMALLINT / INTEGER / BIGINT のみ。
`VARCHAR`（2 バイトの長さ接頭辞を持つ）・日付時刻・DBCS 列は
**書き込み前に `HOST_SERVER_UNSUPPORTED`** にする。

**黙って壊れた値を書かない。** ——なお、この原則を**自分で破っていた**のを
実機検証が見つけた（下記 D5）。

## D3. スコープは「追記」だけ

読み取り・キー付きアクセス・更新・削除・ファイル作成をやらない。
読み取りは SQL がある。ファイル作成は CL で足りる。
plan の撤退ラインどおり「**まず動いて検証済みのもの**」を出す。

## D4. 既存の transport を共有しない

DDM のフレームは **2 バイト長**の 6 バイトヘッダーで、
`host-connection.ts` の **4 バイト長**前提と違う。無理に共通化せず
`transport/ddm-transport.ts` を別に持つ。

また DDM は 1 要求に**チェインされた複数フレーム**を返すため、
「1 要求 = 1 応答」の `request()` ではなく **send / receive を分けた**形にした。

ただし**資格情報のバイト化とパスワード置換値は既存を再利用する**——
原典自身が `// Copied from HostServerConnection.` と書いており、共通と確認できている。

## D5. 置換文字を書き込まない（実機で見つけた自分の違反）

`encodeChar` がコーデックの返す `substituted` を**無視していた**。
CCSID 37 のコーデックに日本語を渡すと置換文字になり、実機で `` が 3 つ書き込まれた。

**D2 の「黙って壊れた値を書かない」に自分で違反していた。**
置換が 1 件でもあれば例外にするよう直した。

**単体テストでは見つからなかった**（SBCS の範囲しか試していなかった）。
**書いた経路と別の経路（SQL）で読み返す**という検証設計が効いた。

## D6. NULL 指標マップを API と一致させる（review 指摘）

`buildDdmRecord` は `null` を受け付け `nullable` を検査するのに、
`buildS38Buf` は指標マップを**常に非 NULL（0xF0）で埋めていた**——
`null` を渡しても「空白/ゼロ」が書かれる、**API が嘘をつく**状態だった。

`buildDdmRecord` が NULL 位置を返し `write` が指標マップに反映するよう直し、
**実機で `null` と空文字が区別されることを確認**した。

## D7. EXCSAT の識別名を原典のまま残す

`EXTNAM` が EBCDIC "TBOX2"（原典の値）で、本来はこのプロジェクトの名前にすべき。
だが**識別名を変えてホストの挙動が変わらない保証が無く、実機で確かめずに変えるのは
「推測で直さない」に反する**。原典の値のまま残し、backlog に積んだ。
