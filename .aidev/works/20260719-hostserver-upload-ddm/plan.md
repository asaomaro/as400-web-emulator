# 計画: DDM 書き込み

## 分割の判定

**subtask にしない。** 1 つの接続クラスとその周辺で、割ると結合検証が分散する。
ただし**純関数を先に固める**ことで、実機依存の部分を最小化する。

## 順序（純関数 → プロトコル → 実機）

1. `encode.ts` / `record-layout.ts`（純関数・単体テストで固める）
2. `ddm-datastream.ts`（フレームの組み立て・解析）
3. `ddm-connection.ts`（握手 → open → write → close）
4. CLI 追加 → 実機検証（書いて **SQL で読み返す**）

**純関数を最初にやる**のは、ここが「真の難所」（型変換）であり、
実機に出る前に固められる唯一の部分だから。

## リスク

| リスク | 兆候 | 対応 |
|---|---|---|
| **SQL メタデータからのレイアウト計算が実際と合わない**（D1 の仮説） | 読み返しが不一致 | **実測を記録する**。推測で直さない。合わなければ事実を残して範囲を狭める |
| PUB400 で `CRTPF` の権限が無い | 実機 | 既存ファイルで代替できるか探す。無理なら未検証として明記 |
| DDM サーバーが PUB400 で無効 | 接続失敗 | 446 は開いていることを確認済み（research）。認証まで通るかは未確認 |
| フレーム長の計算違い | 実機で応答が来ない | 原典の定数をそのまま使い、決め打ちを避ける |

## 撤退ライン

requirement に書いたとおり、**見積もりが外れたらスコープを削って
「動いて検証済みのもの」を出す**。優先順位:

1. 接続・認証（これが通らなければ何も出せない）
2. open してレコード長を取れる
3. 1 レコード書けて SQL で読み返せる
4. 複数レコード・NULL・型の網羅

3 まで到達できれば PR にする価値がある。4 が未達なら未検証として明記する。
