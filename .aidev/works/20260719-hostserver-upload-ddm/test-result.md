
新規は `encode.ts` / `record-layout.ts` の純関数テスト。
**実機に出る前に固められる唯一の部分**なので厚く書いた（SQL 実装の retro が
「真の難所は型変換」と記録している）。

固定した不変条件:

- パック 10 進: 正 `0xF` / 負 `0xD` の符号ニブル、バイト数 `floor(precision/2)+1`
- ゾーン 10 進: 1 バイト 1 桁、負数は最終バイト上位ニブル `0xD`
- 整数: ビッグエンディアン、2 の補数、bigint 対応、範囲外を拒否
- **切り捨て・切り詰めを黙ってしない**（位取り超過・桁あふれ・文字長超過はすべて例外）
- レイアウト: 宣言順に連続配置、対応外の型は**レコード全体を拒否**

### 実装中に見つけた自分の誤り

**パック 10 進のニブル位置が 1 つずれていた。** `12.34` が `00 21 4f` になった
（正しくは `01 23 4f`）。符号ニブルのぶんを「先頭にずらす」と誤解していたが、
実際は**末尾の 1 ニブルが符号**で、桁は先頭から素直に詰める。
単体テストが即座に捕まえた——実機に出ていたら壊れたデータを書いていた。

## 実機検証（PUB400・TLS）

**書いた経路と確かめる経路を分けた**——DDM で書き、**SQL で読み返した**。
同じ経路で確認しても、レコード配置の計算が間違っていることに気づけない。

```
1. MARO1/ZZDDM を作成
2. レイアウト（SQL 由来）: NAME:char@0+10 QTY:packed@10+3 AMT:zoned@13+7 SEQ:int@20+4 = 24 バイト
3. open: recordLength=24 increment=28 nullMapOffset=24
   ✅ SQL 由来のレコード長がホストの申告と一致
4. 3 件を書き込み
5. SQL で読み返し: 3 件
   {"NAME":"ALPHA     ","QTY":"12","AMT":"34.56","SEQ":1}
   {"NAME":"BETA      ","QTY":"-7","AMT":"-0.01","SEQ":2}
   {"NAME":"          ","QTY":"0","AMT":"0.00","SEQ":3}
6. 後片付け: 削除しました
```

### NULL 指標マップの検証（review 指摘の修正後）

`NOTE CHAR(5)`（NULL 許可）を足して再実行した:

```
2. レイアウト: … NOTE:char@24+5 = 29 バイト
3. open: recordLength=29 increment=34 nullMapOffset=29
5. SQL で読み返し:
   {"NAME":"ALPHA     ",…,"NOTE":"ok   "}
   {"NAME":"BETA      ",…,"NOTE":null}      ← NULL が NULL として返る
   {"NAME":"          ",…,"NOTE":"     "}   ← 空文字は NULL ではない
```

**`null` と空文字が正しく区別されている**（指標マップが効いている）。
`increment - recordLength = 5` が 5 列ぶんの指標マップで、`nullMapOffset=29` と整合する。

**spec D1 の仮説が実機で確認できた**——「SQL のメタデータから物理レコードの配置を
計算できる」。ホストが申告する `recordLength=24` と、SQL 由来で計算した 24 が一致し、
**書いた値が SQL で正しく読み返せた**（負数・小数・ゼロ・空文字を含む）。

`recordIncrement=28` と `recordLength=24` の差 4 が NULL 指標マップ
（4 列ぶん）で、`nullMapOffset=24` と整合する。

### 実機で見つけた欠陥（修正済み）★

**DBCS が黙って壊れて書き込まれた。** 最初の実行で `"ベータ"` が
``（置換文字 3 つ）として保存された。

原因: `encodeChar` がコーデックの返す **`substituted`（置換された文字数）を無視**していた。
CCSID 37（SBCS）のコーデックに日本語を渡すと置換文字になるが、それを検査していなかった。

**spec の「黙って壊れた値を書かない」に自分で違反していた。**
置換が 1 文字でもあれば例外にするよう直し、回帰テストを追加した
（SBCS では拒否・DBCS 対応 CCSID なら通る）。

修正後の再実行で `BETA` が正しく往復することを確認した。

### 副産物: CL 経由 RUNSQL の落とし穴

テスト用の表を作る過程で見つけた。**`host_command` の RUNSQL では
`DECIMAL(5,0)` が通らず、`DECIMAL(5, 0)`（カンマの後に空白）なら通る。**

```
NG  CREATE TABLE MARO1.ZZT4 (B DECIMAL(5,0))    → SQL0104 Token ( was not valid
OK  CREATE TABLE MARO1.ZZT6 (B DECIMAL(5, 0))   → SQL7905（作成成功）
OK  CREATE TABLE MARO1.ZZT3 (A CHAR(10), B INTEGER)  → 成功
```

`CHAR(10)` は通るので括弧自体は問題なく、**括弧内のカンマ直後に数字が続く形**が
CL の解釈で壊れる。DDM とは無関係だが、RUNSQL を使う人が確実に踏むので記録する。

### 実機の後片付け

検証で作った `MARO1/ZZDDM` と、切り分け用の `ZZT1` / `ZZT3` / `ZZT5` / `ZZT6` /
`ZZT7` / `ZZT8` は**すべて削除済み**（`ZZT2` / `ZZT4` は作成に失敗したので存在しない）。
