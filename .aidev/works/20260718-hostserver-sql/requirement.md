# 要件: ホストサーバー経由の SQL 実行（ダウンロード）

## 背景 / 課題

ACS の「データ転送」相当を実現する第 2 段階。第 1 段階（`20260718-acs-data-transfer`）で
IBM i ホストサーバーの **signon 認証を貫通済み**であり、その基盤の上に SQL 実行を載せる。

前段の調査（`.aidev/works/20260718-acs-data-transfer/research.md`）で判明している事実:

- 認証部分（パスワード置換値・資格情報のバイト化）は **signon 専用ではなく共通**として実装済み。
  database サーバーは `0x7001`（乱数シード交換）→ `0x7002`（サーバー開始）と**枠が違うだけ**で、
  認証の作り方は同一
- 移植の参照元は JTOpen 本体ではなく `archived/jtopenlite/`。そのうち実装の本体は
  **`DatabaseConnection.java`（4,852 行）** で、`jdbc/`（14,880 行）は JDBC API の皮であり
  TypeScript には不要
- JTOpen は IBM Public License 1.0。**事実に基づく書き起こし**で進める（逐語移植しない）

動機は JVM の排除。`node-jt400` は JVM を起こすため起動が重く、MCP ツールとして
短時間に叩かれる用途に合わない。

## 目的 / ゴール

**database サーバーに接続し、SELECT の結果を実用的なデータ型で取得できる**状態にする。

「実用的な型を一通り」とは、実際の業務テーブルを読むのに必要な範囲を指す（下記機能要件）。
文字列と整数だけでは実テーブルがほぼ読めないため、そこで止めない。

## スコープ

### 対象

- database サーバー（8471 / TLS 9471）への接続と認証（既存の認証基盤を再利用）
- SQL 文の実行と結果セットの取得
- **データ型の変換**（この作業の実質的な本体）
  - 文字（固定長 / 可変長）
  - **DBCS を含む文字**（日本語データ。初回から対象に含める）
  - 数値（パック 10 進数 / ゾーン 10 進数 / 2・4・8 バイト整数 / 浮動小数）
  - 日付 / 時刻 / タイムスタンプ
  - **NULL**（指標の解釈）
- 結果セットのメタデータ（列名・型・桁・小数点以下桁数・CCSID）
- 検証用テーブルを PUB400 の **MARO1 ライブラリに新規作成**し、型を網羅して実機確認する
- `packages/core` のライブラリとして実装し、CLI で検証できること

### 対象外（この作業ではやらない）

- INSERT / UPDATE / DELETE（＝アップロード。DDM レコードレベルアクセスを含む）
- CSV などへのファイル出力
- MCP ツールとしての公開
- Web UI からの操作
- LOB（BLOB / CLOB / DBCLOB）
- ストアドプロシージャの呼び出し、結果セットの複数返却
- トランザクション制御（コミット / ロールバック）の作り込み
  ※ 接続を維持するために必要な最小限の扱いは対象に含めうる（research で判断する）

## 機能要件

- ホスト・ユーザー・パスワード・TLS 可否を与えて database サーバーに接続できる
- 任意の SELECT 文を実行し、全行を取得できる
- 各列の**型に応じた JavaScript 値**を返す（文字列 / 数値 / 日付 / null）
  - 10 進数は**精度を落とさない**扱いにする（詳細は spec で決める。要検討事項参照）
- 列のメタデータ（名前・型・桁・小数点以下桁数）を取得できる
- 日本語（DBCS）を含む文字列が正しく読める
- NULL と空文字を区別できる
- SQL エラー（構文誤り・存在しない表・権限不足）を、原因が分かる形で返す
- 大きな結果セットでも一度に全件をメモリへ載せない選択肢がある
  （逐次取得。詳細は spec で決める）
- 通信内容をトレースできる。**資格情報を平文で出さない**（既存方針を踏襲）

## 非機能要件 / 制約

- 既存の TN5250 実装・signon 実装に影響を与えない
- `packages/core` のピュアロジック層は **Node API 非依存**を維持する
  （I/O は `transport/` に隔離。前段の decisions D1 の方針を踏襲）
- **ピュア層で Node のグローバル（`Buffer` 等）も使わない**
  （前段のレビューで lint をすり抜けた実績があるため明示する）
- JTOpen は参照のみ。ソースをリポジトリに取り込まない。各モジュールに参照コメントを残す
- 実機（PUB400）での検証を伴う。パスワードは環境変数経由でのみ扱う

- [ ] 下記の型が期待どおりの JavaScript 値になる（実機で確認）

- [ ] NULL と空文字が区別できる
- [ ] 列のメタデータ（名前・型・桁・小数点以下桁数）が取得できる

## 未確定事項 / 確認したいこと

**すべて research 工程で事実確認する**（推測で spec を書かない。前段でライセンスを
誤記した反省を踏まえる）。

- database サーバーのデータストリーム形式（要求 ID・パラメータの構造）
- 接続確立の手順（`0x7001`→`0x7002` の後、database 固有の初期化があるか）
- SQL 実行の経路。`executeImmediate` で足りるのか、
  `prepare`→`describe`→`open`→`fetch` の流れが必要なのか
- 結果セットのメタデータをどう受け取るか（`describe` の応答形式）
- **各データ型のバイト表現**（パック 10 進数・ゾーン 10 進数・日付時刻の形式）
- **DBCS 列の扱い**。列ごとに CCSID が返るのか、SO/SI が入るのか、
  既存の DBCS コーデック（930/939/1399）がそのまま使えるのか
- NULL 指標の形式（列ごとのバイトか、ビットマップか）
- 逐次取得の仕組み（ブロッキング係数・フェッチバッファサイズ）
- SQL エラーの受け取り方（SQLCODE / SQLSTATE / メッセージ）
- 10 進数の表現方法をどうするか（`number` は 15 桁程度で精度を失う。
  文字列 / `BigInt` / 専用型のいずれにするか。**業務データでは金額の精度が要件になりうる**）
- **PUB400 で MARO が MARO1 に表を作成する権限を持つか**（実機で確認。
  持たない場合は検証方法を変える必要がある）
