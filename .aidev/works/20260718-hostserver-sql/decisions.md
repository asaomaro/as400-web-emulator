# 決定記録

## D1: 応答は「拡張形式」ではなく「元形式」で来る（実機で判明）

- 背景: spec / design では原典 jtopenlite が解析する拡張形式（列定義 CP 0x3812・
  結果データ CP 0x380E）を前提に実装した。しかし実機（IBM i 7.5 / PUB400）は
  **拡張列記述子を要求しても元形式で返す**。
- 決定: 元形式を正として実装する。
  - 列定義: **CP 0x3805**、ヘッダー 8 バイト ＋ **1 列 54 バイト**（原典 DBOriginalDataFormat と一致）
  - 結果データ: **CP 0x3806**、ヘッダー **14 バイト**（拡張形式は 20）
- 理由: 元形式にも型・長さ・位取り・精度・CCSID・列名がすべて含まれ、実用上足りる。
  拡張形式を返させる方法を探すより、実際に返るものを正しく読むほうが確実。
- 影響: 検証は実機の生バイト列をそのままテストに固定した（手で組んだ期待値は誤りやすい。
  実際、最初に手組みした 16 進が長さ不一致で失敗した）。

## D2: 要求側と応答側で同じ CP が別の意味を持つ

- 背景: 応答側の CP を要求側にも流用して実装したところ、prepare が `rcClass=2 rcCode=-102`
  で失敗した。
- 決定: 要求側の CP は原典の書き出し処理から individually 確認して定義し直した。
  - `0x3807` … 要求では **SQL 文テキスト**、応答では **SQLCA**
  - `0x3812` … 要求では **SQL 文種別**、応答では 拡張列定義
  - 文名 `0x3806` / カーソル名 `0x380B` は **CCSID 37 の EBCDIC**、
    SQL 文だけ **CCSID 13488（UTF-16）**
- 理由: 推測した定数がほぼ全滅した。AGENTS.md の「原典をまず確認する」がそのまま当てはまる。

## D3: 日付・時刻の書式は接続時に ISO へ固定する

- 背景: 実機の初回取得で `D_DATE = "26-07-18"` と**年が 2 桁**で返り、世紀が失われていた。
- 決定: `setServerAttributes`(0x1F80) で日付書式 5（*ISO）・時刻書式 1 を明示する。
  実機で 3=*YMD(26-07-18) / 4=*USA(07/18/2026) / 5=*ISO(2026-07-18) / 6=*EUR(18.07.2026)
  を確認して 5 に決めた。
- 理由: 値は書式化済みの文字列として行バッファに入るため、**書式が分からないと解釈できない**。
  ジョブの既定に任せると環境で結果が変わり、しかも**静かに誤ったデータを返す**。
- 影響: design D3「書式を明示的に固定して要求する」の実装。指定しない実装だったら
  気づかないまま世紀の欠けたデータを返し続けていた。

## D4: RPB 作成の要求 ID は 0x1D00

- 背景: 実装当初 `setServerAttributes`(0x1F80) を RPB 作成に使っていた（別物）。
- 決定: `0x1D00`。作る RPB のハンドルは**パラメータではなく template の RPB ハンドル欄**で指定する。
- 理由: 実機に触る前に原典照合で気づけた。

## D5: 10 進数の実機検証は SELECT 式で行った

- 背景: 検証表に DECIMAL / NUMERIC 列を追加できなかった。RUNSQL 経由の
  `CREATE`/`ALTER` がカンマを含む型（`DECIMAL(11,2)`）で
  `Token ( was not valid` となり失敗する。
- 決定: 表を作らず `SELECT CAST(-12345678.91 AS DECIMAL(11,2)) FROM SYSIBM.SYSDUMMY1` で検証した。
- 理由: 自前のクライアント経由なので RUNSQL の制約を受けず、より確実。
- **反省**: 途中「3 つとも個別に成功した」と判断したが誤りだった。5250 のバッチ実行では
  **最後のコマンドのエラーしか画面に出ない**ため、成功と誤認していた（作成したはずの表の
  存在を確認していなかった）。バッチの結果はジョブログで確認すべきだった。
