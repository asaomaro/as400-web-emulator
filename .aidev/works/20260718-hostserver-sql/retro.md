## うまくいった点

### 1. 前回の retro の改善提案が実際に効いた

**must が 2 件 → 0 件になった。** 前回 must だった 2 件（ピュア層での `Buffer`、
`Object.assign` による型外しの後付け）は、今回**着手前に requirement の非機能要件へ明記**し、
review で機械的に再発チェックした結果ゼロだった。

また前回の記録漏れ（C1: requirement の approved 未記録）も再発していない。

### 2. design 工程（3分26秒）が最大の作業削減を生んだ

任意工程を初めて挟んだ。所要は 3 分半だが、そこで**既存 `ibm1399.ts` の DBCS 部が
CCSID 16684 そのもの**だと突き止め、ICU の原典と突き合わせて確証を得た。

結果、spec で見込んでいた「`gen-tables` に DBCS クラスを追加して 300/16684 を生成する」が
**丸ごと不要**になった。3 分の調査で数時間分の作業が消えた計算になる。

### 3. 実機検証が「静かに壊れる」欠陥を捕まえた

とくに**日付書式**。指定しないとジョブの既定になり `26-07-18` と**年が 2 桁**で返る。
例外も警告も出ず、**世紀が欠けたデータを返し続ける**。単体テストでは絶対に出ない。

他にも実機でしか出なかったものが 3 件（応答が元形式・要求 CP の体系差・RPB の要求 ID）。

### 4. テストの固定値を実機の生バイト列にした

手で組んだ 16 進が長さ不一致で失敗したのを機に、実機から採取したものへ切り替えた。
結果、`0x3805` の 116 バイト・`0x3806` の 46 バイトがそのまま回帰資産になっている。
**期待値を人間が組むと、実装より先にテストが間違う。**

## 課題 / 手戻り

### C1【判断ミス】バッチ実行の成否を画面だけで判断した

5250 で `RUNSQL` を複数まとめて実行し、画面にエラーが出ていないことから
「`DECIMAL(11,2)` の作成に 3 つとも成功した」と結論づけた。**誤りだった**。
5250 のコマンド行はバッチの**最後のコマンドの結果しか表示しない**ため、
先行するコマンドの失敗が見えていなかった。作成したはずの表の存在も確認していなかった。

前段で防げたか: **防げた**。「作った」と言うなら「在ることを確かめる」まで含めるべきで、
実際 `SQLTEST1` では `DSPOBJD` で確認していたのに、後続では省いてしまった。

影響: 検証表に 10 進数列を作れず、後で `SELECT CAST(...)` に切り替える回り道が発生した
（結果的にはそちらの方が確実な検証方法だった）。

### C2【前提の誤り】spec/design が「原典が解析する形式」を「サーバーが返す形式」と同一視した

spec と design は、原典 jtopenlite が解析する拡張形式（列定義 `0x3812` /
結果データ `0x380E`）を前提に書いた。しかし実機は**元形式**（`0x3805` / `0x3806`）で返す。
拡張列記述子を要求しても変わらなかった。

前段で防げたか: **完全には防げない**。確かめるには認証と接続が動いている必要があり、
research の時点では実装が無かった。ただし**「原典がそう解析している」は
「サーバーがそう返す」の証拠にならない**と spec に書けたはずで、
断定せず「要実機確認」と印を付けるべきだった。

### C3【推測】要求側の CP を応答側から流用した

応答の CP をそのまま要求にも使えると思い込み、prepare が `rcClass=2 rcCode=-102` で失敗。
実際は **`0x3807` が要求では SQL 文、応答では SQLCA** というように体系が別で、
推測した定数はほぼ全滅だった。原典の書き出し処理を個別に読んで直した。

前段で防げたか: **防げた**。AGENTS.md に追加したばかりの「原典をまず確認する」を、
定数の定義時に適用しきれていなかった。読んだのは解析側だけで、書き出し側を読んでいなかった。

## 改善提案

### 製品 / コード（→ issue 候補）

- **database 接続にトレースを入れる**（signon にはある `traceFrame` 相当）。
  資格情報の漏洩リスクは無いが**障害時の診断手段が無く**、今回の coding では
  使い捨てスクリプトを何度も書く羽目になった。実利のある不足
- `eslint` に `no-restricted-globals` を追加する（**前回の retro でも提案し、未適用**）。
  今回は手作業と review で `Buffer` の混入を防げたが、仕組みで防げていない
- 拡張形式（`0x3812` / `0x380E`）の解析。他バージョン・他構成で返ってきた場合に備える
- 逐次取得（`stream`）をブロッキング係数を跨ぐ規模で検証する

### PJ プロセス / 規約（→ AGENTS.md）

- **5250 でコマンドをまとめて実行したら、画面ではなくジョブログで結果を確認する**。
  コマンド行は最後の 1 件しか表示しないため、先行コマンドの失敗が見えない（C1）。
  作成系のコマンドは `DSPOBJD` 等で**存在を確かめて初めて「作った」と言う**
- **原典を読むときは「解析側」と「書き出し側」の両方を読む**（C3）。
  応答の解釈だけ読んで要求を組み立てると、同じ番号が別の意味を持つ罠にはまる
- **「原典がそう扱っている」は「サーバーがそう返す」の証拠ではない**（C2）。
  実機で確かめていない前提は spec に断定形で書かず、印を付けて coding に送る

### ハーネス自体（→ aidev-* への提案・適用は人間）

- **spec / design に「未検証の前提」を明示する欄があるとよい**。
  requirement には「未確定事項」、research には「実現性 / リスク」があるが、
  spec と design には**推測で書いた箇所を残す場所が無い**。
  C2 はまさにそれで、断定形で書いた前提が coding で覆った。
  テンプレートに「未検証の前提（coding で確かめる）」を足すと、
  後工程が「どこを疑うべきか」を引き継げる
- 前回提案した「`aidev event <工程> start` で前提工程の承認を検査する」は**未適用**。
  今回は記録漏れが起きなかったが、それは人手で気をつけた結果であり仕組みではない
