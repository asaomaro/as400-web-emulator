# 計画: core の負債返済

## 分割の判定

**subtask にしない。** 3 件はいずれも小〜中で、1 PR に収まる。
むしろ **5 件のうち 2 件を落とした**ことがスコープ制御の本体（spec 参照）。

## 順序

D1（lint）→ D3（改名）→ D2（ロガー）の順。

- **D1 を最初に**: 以降の変更が新ルールに従っているかを、作業中ずっと lint が見てくれる
- **D3 を D2 より先に**: 改名は機械的な置換で、ロガー変更と混ざるとレビューしづらい
- **D2 を最後に**: 依存の向きを変えるため影響範囲が最も広い

## リスク

| リスク | 兆候 | 対応 |
|---|---|---|
| lint 追加で既存コードが大量に落ちる | D1 直後 | ピュア層は Buffer を避けている実測（F1）があるので想定は 0 件。落ちたら実態を記録して範囲を見直す |
| **ロガー注入で監査ログが静かに消える** | 気づきにくい | server が自前ロガーを持つ設計（spec D2）で構造的に潰す。**テストで固定**する |
| 改名で `instanceof` が壊れる | テスト | 別名は同一クラスなので `instanceof` は保たれる。テストで固定 |
| pino を server に移して依存が漏れる | ビルド | `npm ls pino` で確認 |
