# 決定記録

## D1. スコープを 5 件から 3 件に絞った

backlog の 5 件のうち **3 件だけ**返した。理由は requirement の「リスクの違うものを 1 つの PR に
混ぜない」に従ったため。

- **やった**: `no-restricted-globals` / ロガー注入 / `As400Error` 改名
- **やらない**: `ErrorCode` 整理（前提が誤り）／CCSID テーブル（パッケージング変更）

**「全部やる」を目的にしない。** 落とした 2 件は実測値つきで backlog に戻した。

## D2. `ErrorCode` の整理はしない — backlog の前提が誤っていた

backlog は「**19 種**に … ホストサーバーに無縁のものが混在」と書いていたが、実測すると
**21 種で未使用は 0 件**。「無縁のものを消す」作業は存在しなかった。

実在する問題は別物: **`CONNECT_FAILED` が server 側で接続と無関係な用途に 11 箇所流用されている**
（限度到達・参照不正・users ファイルが読めない等）。これは core の型ではなく
**server 側の意味を決め直す**作業で、HTTP ステータス写像にも波及する。性質が違うので分けた。

## D3. ロガーは「依存の向きを逆にする」形にした

core を no-op 既定にするだけだと、**server の 6 ファイル（`audit.ts` を含む）が core の
`childLog` に乗っている**ため、`setLogSink` の呼び忘れで**監査証跡が静かに消える**。

そこで **server に自前の pino を持たせ**、core 側（hostserver の debug）だけを `main.ts` が注入する形にした。

**消えて困る度合いが高いほうを、注入に依存させない。** 監査ログは消えたことに気づけない
種類のログなので、これを「呼び忘れ」の射程に置かない。

## D4. `childLog` は呼び出しごとに factory を引く

束縛時ではなく呼び出しごとに解決する。利用側はモジュールのトップレベルで
`const log = childLog({...})` と束縛する形が多く、束縛時に確定させると
**起動時の `setLogSink` が既に取得済みのロガーに効かない**。

割り当てが増えるが、`isDebugEnabled()` が false のときは呼ばれないので実害は小さい。
**性能より正しさを優先した**（テストで固定済み）。

## D5. `setTimeout` は禁止しない — retro の提案をそのまま適用しない

retro は「`Buffer` / `process` / `setTimeout` 等の Node グローバルを禁止」と書いていたが、
**`setTimeout` / `setInterval` はブラウザにも標準の Web API で Node 固有ではない**。

このルールの目的は「ブラウザで動かない依存を防ぐ」ことなので、移植性のあるタイマーを
塞ぐのは目的に合わない。実際そのまま適用したら **`session/` の正当なタイムアウト処理が
11 箇所落ちた**。

**提案が 2 回繰り返されたことは、その提案が正しいことを意味しない。**
未適用の提案を適用するときも、中身を検証してから入れる。理由はルール中のコメントに残した。

## D6. lint はルールを足すだけでなく「効くこと」を確かめる

2 回の retro が繰り返された原因は「提案したが効いているか確かめていない」状態だったので、
**違反コードを実際に書いて lint が落ちること**、**除外範囲（`transport/`）では通ること**を
確認してから完了とした。確認結果は test-result に記録した。
