# 要件: セッションのジョブ情報を「画面を奪わない」取得に置き換える（無理なら廃止）

## 背景 / 課題

セッション情報のポップオーバーに「ジョブ 🔄 取得」ボタンがある。押すと **画面を乗っ取って `DSPJOB` を実行**する:

1. 今の画面のコマンド行を探す（`findCommandField`＝最後の非保護フィールド）
2. そこへ `DSPJOB` を打ち込んで Enter
3. 出た画面のテキストを**英語ラベル前提の正規表現**で走査（`Job: … User: … Number: …`）
4. F3 で元の画面へ戻す

出典: `packages/core/src/session/session.ts:345`（`fetchJobInfo`）、`:92`（`parseJobInfo`）。

この作りには無理がある:

- **利用者の画面を勝手に操作する**。入力途中の内容が飛ぶ・戻り先が元どおりとは限らない
- **コマンド行がある画面でしか動かない**（アプリ画面の途中では `JOB_INFO_UNAVAILABLE`）
- **英語 NLV 前提**。日本語 NLV では「ジョブ： ユーザー： 番号：」でラベルが変わり、正規表現が外れる
- 実行中は他の操作を `JOB_INFO_BUSY` で弾く必要がある（**セッションを占有する**）
- F3 で戻れない画面（確認画面・メニュー階層）では**元の状態に戻らない恐れ**がある

一方、**ホストサーバー経由でジョブを一覧する経路は既にある**（`QGYOLJOB`。`packages/core/src/hostserver/list/job-list.ts`）。
名前・ユーザー・番号・種別で絞り込めるので、**画面に触れずに**ジョブ識別子を得られる可能性が高い。
IBM i の対話ジョブは**ジョブ名＝装置名**なので、装置名が分かれば引ける見込み。

## 目的 / ゴール

ジョブ情報を**画面を奪わずに**（利用者の操作を邪魔せずに）表示する。
それが成り立たないなら、**中途半端な機能として残さずに廃止**する。

## スコープ

### 対象

- 接続時（またはセッション確立直後）に、**画面に触れない方法**でジョブ識別子を取得する
- 取得できたらセッション情報のポップオーバーに表示する
- **`DSPJOB` を画面で実行する経路は撤去する**（core の `fetchJobInfo` / ws の要求 / MCP ツール / UI の取得ボタン）
- 取得できない場合の見せ方（行を出さない／「不明」と出す）を決める

### 対象外

- ジョブの操作（保留・解放・終了）。既存の「ジョブ」パネルの領分
- ジョブの詳細情報（実行時間・CPU・ライブラリーリスト等）。**識別子（番号/ユーザー/名前）だけ**を対象とする
- プリンターセッション（元々ジョブ情報を持たない）
- 5250 の画面から情報を読む方式の改良（NLV 対応など）——**この方式そのものをやめる**のが本件

## 機能要件

1. セッション接続後、**利用者の操作なしに**ジョブ識別子（番号 / ユーザー / 名前）を得る
2. 取得は**画面の状態を一切変えない**（キー送信・フィールド書き込みをしない）
3. 取得は**セッションの操作を止めない**（`JOB_INFO_BUSY` のような占有をしない）
4. 取得に失敗しても**セッションの接続・操作に影響しない**（失敗は静かに諦める）
5. 取得できたらポップオーバーに表示する。**取得ボタンは無くす**
6. 取得できない場合は、**ジョブの行を出さない**（「取得」ボタンや「不明」を残さない）
7. `DSPJOB` 経路（core・ws・MCP・UI）を撤去する

## 非機能要件 / 制約

- **接続を遅くしない**。ジョブ情報の取得は接続完了を待たせない（背後で取り、取れたら表示に足す）
- ホストサーバーへの接続が要る場合、**認証情報が無い環境でも壊れない**（黙って諦める）
- 既存のセッション接続・切断・再接続の挙動を変えない
- MCP のツール構成が変わる場合は README の記述と揃える

## 完了条件 (受け入れ基準)

- [ ] セッションを開くと、**何も押さずに**ジョブ情報が表示される（取得できる環境の場合）
- [ ] ジョブ情報の取得で**画面が変化しない**（DSPJOB が実行されない）
- [ ] 取得できない環境では、ジョブの行が出ない（取得ボタンも出ない）
- [ ] `fetchJobInfo` / `requestJobInfo` / MCP の該当ツール / UI の取得ボタンが無くなっている
- [ ] 実機（PUB400）で、接続直後にジョブ情報が出ることを確認できる
- [ ] 既存のテストが通り、撤去した経路のテストも整理されている

## 未確定事項 / 確認したいこと

- **接続時に装置名が分かるか**。設定で指定した場合は分かるが、**自動割り当て（未指定）のとき**に
  ホストが割り当てた装置名をこちらが知れるかは要調査（RFC 4777 のネゴシエーション応答）
- **ジョブ名＝装置名**が実機で成り立つか（対話ジョブの慣例だが、確認が要る）
- `QGYOLJOB` を**セッションの利用者の資格情報**で呼べるか（自動サインオンが設定されていない場合はどうなるか）
- 取得にかかる時間（接続直後に走らせて体感を損なわないか）
- **どこで取得するか**（サーバー側でセッション確立時に自動取得 / UI がポップオーバーを開いたときに取得）
- 上記が成り立たない場合、**廃止**（機能そのものを消す）で確定してよいか
