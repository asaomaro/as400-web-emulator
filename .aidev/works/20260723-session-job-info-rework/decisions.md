# 決定記録

## D1: 起動応答の「システム名・装置名」は**あれば読む**（無くても応答コードは返す）

- 背景: `parseStartupResponse` を最初は「応答コード＋システム名＋装置名が揃っていなければ undefined」に
  していたが、プリンターの既存テストが落ちた。合成の起動応答レコードが**応答コードまでしか無い**ため。
- 決定: 応答コードだけ読めれば `{ code, system: "", device: "" }` を返す。装置名が要るかは呼び出し側が判断する。
- 理由 / 代替案: プリンターは**接続可否**（応答コード）だけで用が足りる。表示セッション側は
  「装置名が入っているものだけ起動応答として食べる」と決めれば、通常のデータストリームを誤って食べる危険も同時に下がる。
- 影響: `Session5250` は `startup.device !== ""` を条件にレコードを食べる。

## D2: ジョブ照会は `SessionManager` の注入点にする（`lookupJobs`）

- 背景: `CommandConnection` と `listJobs` を直接呼ぶと、「資格情報が無ければ引かない」「1 件のときだけ採用」
  「失敗しても壊れない」という**判断そのもの**が実機なしでテストできない。
- 決定: `SessionManagerOptions.lookupJobs` を任意の注入点として設ける（既定は `QGYOLJOB` 経由）。
- 理由 / 代替案: `host-ifs.ts` の `deps.connect`（テストで偽の接続を差し込む口）と同じ考え方で、
  このリポジトリの既存の作法に揃う。
- 影響: server のテストで 5 通り（起動応答のみ / 1 件 / 複数件 / 0 件 / 例外）を固定できた。

## D3: `jobinfo` メッセージは**サーバー発だけ**残す

- 背景: 旧 `jobinfo` は「クライアントが要求 → サーバーが DSPJOB を実行 → 応答」だった。
- 決定: クライアント→サーバーの要求（`WsJobInfoReq`）を削除し、サーバー→クライアントの通知だけ残す。
  装置名は `opened` に載せ、ユーザー・番号が引けたら `jobinfo` で後追いする。
- 理由 / 代替案: すべて `opened` に載せる案もあるが、照会は接続後に数秒かかるので**接続を待たせる**ことになる。
  「先に出せるものを出し、後から足す」方が要件（接続を遅くしない）に合う。
- 影響: web-ui は `jobinfo` 受信で busy を触らない（要求していないので待ち状態ではない）。

## D4: `get_job_info`（MCP）は消さずに**副作用なしの読み取り**へ変える

- 背景: 要件は「DSPJOB 経路の廃止」。ツールごと消す案もあった。
- 決定: ツール名は残し、**画面に触れずに既に得ている情報を返す**だけにする（`refresh` 引数は削除）。
- 理由 / 代替案: 消すと MCP の利用者から「ジョブが分からない」機能後退になる。
  実装はキャッシュを返すだけで副作用が無く、要件の趣旨（画面を奪わない）は満たす。
- 影響: README のツール表を実態に合わせた。出力の型が変わる（`user`/`number` は任意になる）。
