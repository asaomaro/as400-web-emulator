### 指摘

- [nit] 入力がコメントだけのとき、実行ボタンは押せる（`canRun` は `sql.trim().length > 0` を見ている）が、
  分割の結果が 0 文なので**何も起きない**。
  → 対応: 許容（押しても壊れない。「コメントだけ書いて押す」場面が想像しにくい）

- [nit] CSV のファイル名がタブを区別せず、複数タブを続けて落とすと**同じ名前**になっていた。
  → 対応: **直した**（複数タブのときだけ `-1.csv` `-2.csv` のように文番号を付ける）。回帰テスト 1 本追加

- [nit] タブを切り替えると手で決めた列幅が捨てられる（`selectTab` で `cols.clear()`）。
  → 対応: 許容（列が違うタブに前の幅を当てる方が害が大きい。既存も「列が変わったら捨てる」規律）

### 確認したこと

- **単一文の挙動に退行なし**——既存の `sql-pane.test.ts`（43 件）が無修正で通っている
- 結果セットの手放しが**全タブぶん**になっている（1 本でも残すと接続がプールへ戻らない）
- `loadMore` / CSV / 列幅がすべて**表示中のタブ**を見ている
- サーバー・core の DB 層に**触っていない**（既存 API を 1 文ずつ叩くだけ）

### 良かった点（記録）

- 分割を core の純ロジックに置いたので、境界（文字列・識別子・コメント・閉じ忘れ）を
  UI を通さずに 17 件のテストで固定できている
- 「1 文なら今までどおり」を**タブ帯を出さない**ことで担保し、テストでも固定している
- 実機の画面で見出しの読みにくさ（先頭コメントだけが出る）に気づいて直している
