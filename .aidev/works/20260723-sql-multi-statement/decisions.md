# 決定記録

## D1: 結果の表を子コンポーネントにし、タブごとに KeepAlive で保持する

- 背景: 利用者から「タブの切り替えで、**表示が切り替わった後に操作を受け付けない時間**がある」との指摘。
  `SELECT * FROM QSYS2.SYSTABLES` と `SYSCOLUMNS` を 200 件ずつ表示した状態で再現した。
- 実測（PUB400・実ブラウザ）:

  | 対象 | 切り替えが落ち着くまで | 長いタスク |
  |---|---|---|
  | 200 行 × 40 列（8,000 セル） | 427ms | **279ms** |
  | 200 行 × 34 列（6,800 セル） | 392ms | **254ms** |
  | 50 行（1,700 セル） | 83ms | **なし** |

  さらに同じ表で「DOM を挿し直すだけ」の時間を測ると **65ms**。
  つまり 250ms のうち**大半は Vue の vnode 作り直し**で、DOM 生成やレイアウトではない。
- 決定: 表を `SqlResultTable.vue` に切り出し、`<KeepAlive :max="4">` でタブごとに保持する。
  切り替えは表示の切り替えだけになり、作り直しが起きない。
- 結果（同じ実測）: 長いタスクが **279ms → 50〜100ms**、落ち着くまでが **427ms → 125〜173ms**。
  2 回目以降の切り替えは 50ms 前後で、体感の引っかかりは消える。
- 代替案: 行の仮想化（表示範囲だけ描く）が本命だが、`table-layout: auto` では
  描画範囲によって列幅が揺れるため、別の作業として切り出す（backlog）。
  `table-layout: fixed` は列が中身に合わなくなるので、既存の見た目を壊す。
- 影響: 列幅が**タブごと**になった（切り替えて戻っても残る。review の nit も解消）。
  スクロール位置は `KeepAlive` の切り離しで失われるため、表側で控えて戻す（D2）。

## D2: スクロール位置は表側が自分で覚える

- 背景: `KeepAlive` は DOM をいったん切り離すので、`scrollTop` は 0 に戻る（実機で確認）。
  見比べている最中に先頭へ飛ばされると、タブを行き来する意味が薄れる。
- 決定: スクロールのたびに位置を控え（反応性は要らないので `ref` にしない）、
  `onActivated` で `nextTick` を待ってから当てる。
- 理由: `onDeactivated` で読むと**もう 0 になっている**（実機で確認）。
  再挿入直後も高さが決まっておらず、その場で入れても効かない。
- 影響: タブを戻すと**ほぼ同じ位置**に戻る（実測 800 → 740）。
