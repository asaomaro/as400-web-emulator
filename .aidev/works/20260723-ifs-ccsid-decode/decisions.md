# 決定記録

## D1: 行末（NEL）の判定は「バイト列」ではなく「復号後の文字列」で数える

- 背景: spec「行末の扱い」は 0x15 と 0x25 の個数をバイト列の段階で数えると書いていた。
- 決定: 復号してから U+0085 と U+000A を数える。
- 理由 / 代替案: 混在 DBCS（1399 等）では SO/SI に囲まれた 2 バイト文字の**第 2 バイトに 0x15 が現れうる**。
  バイト列のまま数えると、それを行末と誤って数える。復号は SO/SI の状態機械を通るので取り違えない。
- 影響: 挙動は spec の意図どおり（「多い方を採る」は維持）。数える対象が変わるだけ。

## D2: 「復号できる CCSID」と「符号化できる CCSID」を分ける

- 背景: spec の対応表は復号だけを見ていたが、保存には逆変換が要る。`TextEncoder` は **UTF-8 しか吐けない**。
- 決定: `canDecodeCcsid` と別に `canEncodeCcsid` を設ける。
  UTF-8 / UTF-16BE / EBCDIC 系 / 単バイト系（819・1252）は符号化でき、**Shift_JIS 系（932・943）は読み取り専用**とする。
- 理由 / 代替案: 単バイト系は「全 256 バイトを復号して逆引き表を作る」ことで**表を同梱せずに**戻せる。
  多バイトの Shift_JIS には同じ手が使えず、表を足すには実機の必要性が確認できていない
  （research では 932/943 のタグを 1 件も観測していない）。読めるのに保存だけ断る方が、
  化けたまま保存するより安全。
- 影響: 候補一覧（`TEXT_CCSIDS`）に `writable` を持たせ、UI が保存不可を示せるようにする。
  server は書き込み要求で未対応なら 400 を返す。

## D3: 選択肢の一覧（TEXT_CCSIDS）は表を持たない別モジュールに置く

- 背景: web-ui に候補一覧を出したいが、`ccsid-text.ts` は `codec.js`（DBCS の巨大な表）を引き込む。
- 決定: `codec/ccsid-catalog.ts`（依存ゼロ）に一覧と `LineEnding` 型を置き、`browser.ts` から公開する。
  `ccsid-text.ts` はそれを再エクスポートする。
- 理由 / 代替案: UI 側で一覧を二重定義すると必ずずれる。サーバーに一覧取得の口を足すのは大げさ。
  一覧の `writable` が実装と一致することは core のテストで突き合わせる。
- 影響: ブラウザバンドルに表が入らない（backlog `library-extraction.md:41` の懸念に沿う）。

## D4: BOM も行末と同じく「読んだときの形」を持ち回って書き戻す

- 背景: spec は BOM を判定材料としか書いていなかった。`TextDecoder` は BOM を取り除いて返すため、
  そのまま保存すると **BOM が黙って消える**（Windows で作られたファイルでよく起きる）。
- 決定: 応答に `bom: boolean` を載せ、書き込み要求で受け取ったら先頭に付け直す。
- 理由 / 代替案: `ignoreBOM: true` で BOM を本文に残す手もあるが、編集画面に見えない U+FEFF が
  居座ることになる。行末（`newline`）と同じ「読んだ形を持ち回る」やり方で揃えた。
- 影響: `/read` の応答と `/write` の要求に `bom` が増える。UI はそのまま往復させるだけでよい。

## D5: `writeFile` に任意の `dataCcsid` を足す（既定の挙動は変えない）

- 背景: 実機検証（`hostserver-check`）で EBCDIC タグ付きのファイルを作れないと、
  決定表②（タグに従う）を実機で確かめられない。タグはサーバー既定（実機では 850）で付いてしまう。
- 決定: `IfsConnection.writeFile` に**任意の** `dataCcsid` を足す。既定（未指定＝0）は従来どおり。
- 理由 / 代替案: 通常の保存経路（`/api/host/ifs/write`）は**渡さない**ので製品の挙動は変わらない。
  「書き込み時にタグを正しく付ける」改修は spec の対象外（backlog の別課題）のままで、
  検証手段だけを先に用意した形。
- 影響: `tools/hostserver-check` の `ifs-ccsid --roundtrip` が、作成 → タグ取得 → 復号 → 削除まで通せる。

## D6: プレビューの種別分岐を `kind` だけで判定するよう直す（既存不具合の修正）

- 背景: test 工程の実機確認で、**表示できているテキストの下に「この形式はプレビューできません」**が
  出ているのを画面で見つけた。`main` にもある不具合で、iframe / img / 最後の案内が
  「編集中（保存すると上書きします）」の `v-if` に `v-else` で繋がっていたため、
  テキストが未編集のときに最後の `v-else` が真になっていた。
- 決定: iframe / img / 案内は `kind`（pdf / image / binary）だけで分岐させる。
- 理由 / 代替案: 今回の変更で textarea の条件を触っており、同じ連鎖の中にある。
  そのまま残すと「CCSID 対応で出た不具合」に見える。回帰テストを 2 本（テキストで出ない・
  バイナリで出る）追加した。
- 影響: 表示だけの修正。この work の対象範囲外の既存不具合だが、触っている箇所なので併せて直した。
