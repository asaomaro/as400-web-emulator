## 実機検証（PUB400）

### 1. core の API 経由（`npm run ifs-ops -w @as400web/hostserver-check`）

新設したコマンド。作業フォルダを作って一通り試し、**最後に消している**。

```
--- リネーム
  OK   ファイルを改名 / フォルダを改名
  OK   既存の名前へ改名: ALREADY_EXISTS（期待どおり失敗）
  OK   存在しないものを改名: NOT_FOUND
--- 削除
  OK   中身のあるフォルダを rmdir: NOT_EMPTY
  OK   フォルダをファイル削除で消す: ACCESS_DENIED（種別を取り違えた症状）
  OK   深い順に消す（中身 → 親）
  OK   消えたフォルダをもう一度 rmdir: NOT_FOUND
```

## 差し戻し（1 回）

**実機の HTTP 検証で発見**: **フォルダを「既存のファイル名」に改名**しようとすると、ホストが rc=3（Path not found）を
返すため **404「対象が見つかりません」**になっていた。対象は存在しており、実際は**その名前が既に使われている**。
他の 3 通り（ファイル→既存ファイル / フォルダ→既存フォルダ / ファイル→既存フォルダ）は rc=4 で正しく 409。

対応: rename が `NOT_FOUND` で失敗したときだけ**その名前が使われているかを確かめ**、
使われていれば 409 `ALREADY_EXISTS` に読み替える（成功する経路に往復を足さない）。
回帰テスト 2 本（読み替える / 本当に無いときは 404 のまま）を追加し、**実機で 4 通りすべて 409 になることを再確認**した。

自動テストでは緑のままだった種類の問題で、**実機の応答を見たから見つかった**（前作業の「プレビューできません」と同型）。

## 未検証・既知の制約

- **`zip-writer` の 4 件は環境要因**（このサンドボックスで `unzip` を実行できない `EACCES`）。本変更とは無関係
- **シンボリックリンクの削除は実機未検証**。列挙が対象に含めることは単体テストで固定したが、
  実機の `link.txt` は利用者の資産なので消していない
- **再帰削除の途中失敗**（権限・使用中で止まる場合）は実機で再現していない。
  単体テストでも未カバー——`partial: true` を返す経路は**コードレビューでの確認に留まる**
- **上限の既定値（1,000 件 / 500 フォルダ）は実測に基づかない**。zip の実績値からの当て込み
- 現在地そのものを削除する経路は UI に無い（一覧に出る子フォルダしか選べない）ため、
  plan で挙げた「現在地を消したら親へ移動」は不要だった
