# 要件: IFS ペインの移動と、ファイル・フォルダの削除／リネーム

## 背景 / 課題

IFS ペインは一覧・プレビュー・アップロード・zip 取得まで揃っているが、**日常の操作で詰まる穴**が 3 つ残っている。

| 操作 | 現状 | 根拠 |
|---|---|---|
| 上位フォルダへ戻る | **一覧からは戻れない**。パンくずか左のツリーを使うしかない | `IfsPane.vue` の一覧に上位への行が無い（`.` と `..` は core が落としている。`ifs-connection.ts:241`） |
| ファイルの削除 | できる | `removeSelected`（`IfsPane.vue:345`） |
| **フォルダの削除** | **できない**。選んでも何も起きない | `removeSelected` が `entry.isDirectory` で早期 return（同 `:347`）。core にも rmdir が無い |
| **リネーム** | **できない**（プロトコル層から無い） | `ifs-connection.ts` に rename 系のメソッドが無い |

一覧の中で「1 つ上へ」が出来ないのは、ファイラーとしては基本の欠落で、
深い階層を辿るほど左ツリーへ視線を往復させることになる。
フォルダ削除とリネームは、**アップロードして置き場所を直す**という普通の流れが完結しない
（間違った名前で置いたら消せない・直せない）。

backlog: `.aidev/backlog/hostserver.md`（「IFS のディレクトリ削除（rmdir = 0x000E, CP 0x0001）」）。
リネームは `20260720-ifs-file-browser` の requirement で**対象外**と明記していたものを、今回スコープに入れる。

## 目的 / ゴール

IFS ペインだけで**辿る・整理する**が完結する状態にする。
上位フォルダへ 1 クリックで戻れ、置き間違えたファイル・フォルダを消せて、名前を直せる。

## スコープ

### 対象

- 一覧の **1 行目に「上位フォルダへ」**を置く（ルートでは出さない）
- **フォルダの削除**（rmdir）。**中身があるものは、中身ごと再帰的に削除できる**
- ファイル・フォルダの**リネーム**（同じフォルダ内での改名）

### 対象外

- **別フォルダへの移動**（パスをまたぐ rename）。要求としては同じでも、誤操作で行方不明になりやすいので分ける
- 複数選択しての一括操作（現状の一覧は単一選択）
- ドラッグ＆ドロップでの移動
- ごみ箱・取り消し（IBM i 側に戻す手段が無いため、確認ダイアログで担保する）
- シンボリックリンクそのものの削除・追跡の制御

## 機能要件

### 上位フォルダへ

1. 一覧の**先頭行**に「.. 上位フォルダへ」を表示する
2. クリック / Enter / Space で 1 つ上のフォルダへ移動する（フォルダ行と同じ操作感）
3. **ルート（`/`）では表示しない**（押せない行を残さない）
4. 移動後の一覧・ツリー・パンくずの状態は、既存の移動（フォルダを開く・パンくずを押す）と同じ

### 削除

5. **フォルダを選んで削除できる**（現状はファイルのみ）
6. 空でないフォルダは、**中身ごと削除するかを確認したうえで**削除できる
7. 確認は**何がどれだけ消えるかを示す**（対象の名前と、含まれるファイル数・フォルダ数）
8. 再帰削除には**上限**を設ける。上限を超える対象は削除せず、対象を絞るよう案内する
   （zip の一括取得と同じ考え方。事故の規模を抑える）
9. 削除の実行は**サーバーのログに残す**（誰が・どのパスを・何件消したか）

### リネーム

10. 選択中のファイル / フォルダの名前を変更できる
11. 入力は**現在の名前を初期値**にする。パス区切り（`/`）を含む入力は受け付けない
12. 同じ名前が既にある場合は上書きせず、**その旨を案内**する
13. リネーム後は一覧・ツリー・プレビューの表示が新しい名前に追随する

## 非機能要件 / 制約

- **プロトコルは原典（JTOpen）に照らして実装する**（AGENTS.md「既存プロトコル実装の移植」）。
  推測でレイアウトを組まない
- 実機（PUB400）で往復を確認できること
- 一覧の表示性能を落とさない（先頭行の追加で追加の往復を発生させない）
- 削除・リネームは**ホスト上のデータを壊しうる操作**なので、確認と記録を省かない
- 既存の操作（プレビュー・アップロード・zip・文字コード切替）に退行を出さない

## 完了条件 (受け入れ基準)

- [ ] 一覧の先頭から 1 クリックで上位フォルダへ移動でき、ルートではその行が出ない
- [ ] 空のフォルダを削除できる
- [ ] 中身のあるフォルダを、件数を確認したうえで中身ごと削除できる
- [ ] 上限を超える再帰削除は実行されず、理由が利用者に伝わる
- [ ] ファイル・フォルダをリネームでき、一覧とツリーが新しい名前になる
- [ ] 既存の名前と衝突するリネームは失敗し、その旨が伝わる
- [ ] 実機（PUB400）で上記の削除・リネームが往復する
- [ ] 既存機能（プレビュー・保存・アップロード・zip）に退行が無い

## 未確定事項 / 確認したいこと

- **リネーム要求のプロトコル**（要求 ID・テンプレート長・コードポイント）。JTOpen の該当クラスを直読して確定する必要がある。
  「上書きするか」を指定するフィールドの有無もここで分かる
- **ディレクトリ削除の要求**（backlog は `0x000E` / CP `0x0001` と記録しているが、**実機で未確認**）
- 再帰削除を**どこで実装するか**（server で 1 つずつ削除する／ホスト側にまとめて任せる手があるか）と、
  既存の `ifs-collect.ts`（zip の再帰収集）をどこまで再利用できるか
- 再帰削除の**上限値**の妥当な既定（zip は件数・バイト数・ディレクトリ数で持っている）
