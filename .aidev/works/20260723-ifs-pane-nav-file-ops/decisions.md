# 決定記録

## D1: フォルダは「行末の … ボタン」で開かずに選ぶ

- 背景: spec が**フォルダの選択手段**を決めていなかった。現状 `activate` はフォルダなら移動して
  `selected` を設定しないため、**一覧からフォルダを選べず**、削除・改名の起点が無い（実装中に判明）。
- 決定: フォルダ行の右端に小さな「…」ボタンを置き、押すと**移動せずに選択**する。
  クリック＝開く（移動）は変えない。ユーザーに 3 案（… ボタン / クリック＝選択＋ダブルクリック＝開く /
  ヘッダーで現在地を操作）を提示して選んでもらった。
- 理由 / 代替案: 「クリック＝選択、ダブルクリック＝開く」は一般的だが、**毎日使う移動が 2 クリックになる**。
  ヘッダー案は「今いるフォルダを消す」ことになり、一覧から兄弟フォルダを消せない。
- 影響: spec の「振る舞いの詳細 → 削除」に選択手段の記述を足す必要がある（spec には未記載のまま実装した）。
  フォルダ選択時は**ファイル単体のダウンロードを出さない**（zip はヘッダーの一括に任せる）。

## D2: `deleteFile` は名前を変えずに `recursive` を足す

- 背景: spec の API 一覧は web-ui 側を `deletePath` と書いていた。
- 決定: 既存の `deleteFile(source, path)` を `deleteFile(source, path, { recursive? })` に拡張する。
- 理由 / 代替案: 名前を変えると呼び出し側（`IfsPane`）の差分が増えるだけで、得るものが無い。
  サーバー側のルートは spec どおり（`/delete` に `recursive`、`/delete-plan`、`/rename`）。
- 影響: 無し（新規の `deletePlan` / `renamePath` は spec の名前どおり）。

## D3: ファイル削除が rc=13 でも、種別を引けていれば元のエラーを返す

- 背景: spec は「種別が分からない場合はファイルとして試し、rc=13 ならディレクトリとして解釈し直す」と
  定めた。しかし**本当に権限が無いファイル**に対してディレクトリとして再試行すると、
  rmdir の rc=3（Path not found）→ 404「見つかりません」が返り、原因から遠ざかる。
- 決定: 再解釈は**種別を引けなかったときだけ**行い、**再試行も失敗したら元の ACCESS_DENIED を返す**。
- 理由 / 代替案: 「権限がありません」は正しい案内で、それを別の誤った案内に置き換えてはいけない。
- 影響: `/delete` の実装に `asDirectory()` の包みが入る。挙動は spec の意図どおり。
