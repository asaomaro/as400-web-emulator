### 指摘

- [should] `packages/web-ui/src/components/IfsPane.vue:679` **`role="option"` の `<li>` の中に `<button>` を置いた**。
  ARIA では option の中に操作可能な子孫を置けない（支援技術に「…」が正しく伝わらない）。
  そしてこの「…」は**フォルダを選ぶ唯一の入口**なので、伝わらないとフォルダの削除・改名が
  支援技術の利用者から使えなくなる。
  行は実際には「開く / プレビューする」という**コマンド**で、単一選択リストボックスの意味論とも合っていない。
  → 対応: 差し戻し（一覧を `role="list"` ＋ `listitem` にし、選択は `aria-current` で示す）

- [nit] `packages/server/src/ifs-delete.ts:110` 件数の上限判定が**エントリのループの中だけ**にある。
  空フォルダばかりの木では `targets` にディレクトリだけが積まれ、`maxEntries` の判定を通らずに増える。
  既定では `maxDirectories`（500）が先に効くので実害は無いが、**上限を入れ替えて設定すると穴になる**。
  → 対応: 差し戻しのついでに、ディレクトリを積むときも判定する

- [nit] `.aidev/backlog/hostserver.md` の「CLI 引数を README に追記」の項目に、今回足した
  `--ifs-delete-max-entries` / `--ifs-delete-max-dirs` が含まれていない。
  → 対応: backlog の記述を更新する（この作業では README に書かない方針のまま）

### 良かった点（記録）

- ディレクトリ削除の要求を**ファイル削除のコピーで作らなかった**。テンプレート長 10 と名前位置の違いを
  テストで固定してあり、`buildDeleteRequest` との差分そのものを 1 本のテストで押さえている
- 「数えてから消す」と「上限超過・辿れない場合は 1 件も消さない」が、
  ルートのテストで**削除要求が飛んでいないこと**まで確認されている
- 種別の取り違えの症状（rc=13 →「権限がありません」）を、実機で確かめたうえで
  `asDirectory()` の再解釈に落とし込んでいる。しかも**再解釈に失敗したら元のエラーに戻す**（decisions D3）
- シンボリックリンクを「辿らないが対象に含める」ことを、テスト名で理由ごと残している
  （残すと親が空にならず rmdir が rc=9 で止まる）
