# 決定記録

## D1. 適用範囲を database だけでなく 6 箇所に広げた

backlog は「database 接続に」と書いていたが、**実測すると 4 接続すべてに無かった**。
同じ 20 バイトヘッダー形式を共有しているので、**1 つの共通ヘルパで全部に効く**。
database だけに入れるのは、同じ作業を将来 3 回繰り返すこと。

## D2. マスクだけでは足りない — 応答本文に業務データが載る

backlog は「資格情報の漏洩リスクは無い」で止まっていたが、**問題は CP ではなく本文**だった。
SQL の応答フレームには**取得した行データそのもの**が入る。

対処: (1) 既定では出さない（`isDebugEnabled()` で整形ごと省く）
(2) 値を 64 バイトで切り、**切った量を明示**する (3) README に明記する。

**本文は伏せない**——伏せたら切り分けに使えない。代わりに「debug では業務データが出る」と
はっきり書く。当たり前のことでも、誰も書いていなかったから書く。

## D3. 呼び出しごとではなく接続を包む

`request()` の呼び出しは 5 接続で **12 箇所**。1 箇所書き忘れると
「そこだけ追えない」という気づきにくい穴になる。
`traced(conn, log)` で**接続を作るときに 1 度包む**——書き忘れが構造的に起きない。

## D4. 正体不明の CP `0x111A` は伏せる側に倒す

signon の応答に載る 20 バイト（SHA-1 と同じ長さ）。**正体を特定できなかった**——
原典（jtopenlite の `SignonConnection`）もこの CP を解析せず読み飛ばしている。

**「分からないから出しておく」ではなく「分からないから伏せる」を選んだ。**
損得が非対称である——伏せて失うのは診断項目 1 つ、伏せずに間違えるとトークンをログに残す。

ただし**これは推測に基づく判断**である。そう明記し、正体の特定を backlog に積んだ。

## D5. DDM は含めない

フレーム形式が違う（2 バイト長・6 バイトヘッダー）ため別実装が要る。
**作業 4 の直後に触るリスク**——DDM は同日に書いたばかりで実機検証も最小限——を
取る価値がこの作業では低い。backlog に残す。

## D6. CLI にログ sink を入れる（作業 3 の穴を塞ぐ）

作業 3 でロガーを注入式にしたとき、**`tools/hostserver-check` への注入を忘れていた**。
障害切り分け用の CLI が、いちばん切り分けに使いたい情報を出せない状態だった。

`log-init.ts` を追加。pino は使わず stderr に素で出す
（CLI に依存を増やさない／stdout は結果の出力に使うので汚さない）。

> 作業 3 では「注入し忘れると静かに消える」失敗モードを認識し、
> **server についてはその射程から外した**（自前の pino を持たせた）。
> だが **CLI については考えていなかった**——射程の洗い出しが不完全だった。
