# 振り返り: ホストサーバー認証（ACS データ転送 第1段階）

## サマリ

TN5250 とは別プロトコルの IBM i ホストサーバーに接続し、PUB400 の実機で認証を貫通させた。
当初「実現できなくても構わない」という前提だったが、**research 段階で実機認証に成功**し、
以降は探索ではなく書き起こし作業になった。

| 指標 | 値 |
|---|---|
| リードタイム | 1時間12分（requirement start → deliver approved。承認待ちを含む壁時計） |
| 手戻り回数 | **0**（同一 phase の start が 2 回以上になったものは無い） |
| 差し戻し回数 | **0**（sent_back イベントなし） |
| レビュー指摘 | must 2 / should 1 / nit 3（すべて自己レビューで検出・修正） |
| タスク | 9 計画 / 9 完了 |
| テスト | 709 緑（新規 58） |

工程別の所要（approved − 直近 start）:

```
requirement   約5分   ※ approved 未記録（下記「記録漏れ」）
research     18分20秒  ← 最長の上流工程
spec          4分06秒
plan          2分16秒
coding       26分30秒  ← 最長
test          7分21秒
review        4分46秒
deliver       1分46秒
```

## うまくいった点

### 1. research で実機を叩いたことが、この作業の成否を決めた

認証情報を送らない「交換属性要求」だけを投げる使い捨てスクリプトで、
**パスワードレベル 3 をサーバーから直接取得**できた。これで判明したこと:

- DES 経路（手書き 700 行超）が**丸ごと不要**と判明 → 規模が想定の 1/10 に
- `DSPSYSVAL QPWDLVL` を 5250 で叩く必要すらなかった

さらに research 中に認証成功まで到達したため、**spec 以降は「動くと分かっているものを書く」**
作業になり、手戻り 0・差し戻し 0 に直結した。research 18分の投資が coding 26分を安全にした。

### 2. 実機が、単体テストでは絶対に出ない欠陥を 3 つ捕まえた

| 欠陥 | 発見経路 |
|---|---|
| TLS ではポートマッパーのサービス名に `-s` が要る | 実機（`--tls --resolve-port` の失敗） |
| DDM の既定ポートが 8471 ではなく 446 | 実機（`drda` を照会） |
| VRM の version は上位 16 ビット（4 バイト分割は誤り） | 実機出力の違和感 |

いずれもモックでは再現しない。**プロトコル実装では実機検証が単体テストの代替にならない**。

### 3. 自己レビューが lint とテストの両方をすり抜けた must を 2 件検出した

- **ピュア層での `Buffer`**: lint の `no-restricted-imports` は**import しか見ない**ため、
  グローバル参照は素通りしていた
- **`traceFrame` の長さ未検査**: debug ログ有効時のみ `RangeError` を投げる。
  通常のテスト実行では発現しない

どちらも「テストが緑」「lint がクリーン」の状態で潜んでいた。差分の直読に価値があった実例。

### 4. ユーザーの実績情報が、AI の過剰な保守性を正した

誤パスワードの実機検証を「アカウント無効化リスク」で見送っていたが、
ユーザーから「PUB400 で 1 回失敗しても無効化されなかった」という**実績**を得て実施できた。

結果、未検証の穴が 1 つ埋まったうえ、**存在しないユーザー ID でも `0x0003000B` が返る**
（ユーザー列挙対策）という想定外の発見まで得られた。

なお実施にあたっては、プロトコル自身が持つ安全弁（`0x3000B` と `0x3000C` が別コード）を
根拠にできた。**仕様を読み込んでいたことが、リスク判断の材料になった**。

## 課題 / 手戻り

### C1【記録漏れ】requirement の approved を記録していない

`metrics.yml` に `requirement/start` はあるが `approved` が無く、`state.yml` の
`approved` リストからも requirement が欠けている。

原因: requirement 終了時に「research を推奨」する選択肢を提示し、ユーザーが「researchへ」と
答えた際、`aidev approve requirement` を**実行せずに** `aidev event research start` だけ記録した。
「次工程へ進む」の指示を、承認記録なしで遷移として扱ってしまった。

前段で防げたか: **防げた**。`aidev guard research` を実行していれば前提工程の未承認を
検出できたはずだが、`aidev event ... start` を直接叩いており guard を通していない。

### C2【誤り】requirement に「Apache-2.0」と誤記した

JTOpen のライセンスを確認せずに Apache-2.0 と書き、research（F10）で
**IBM Public License 1.0** と判明して訂正した。

前段で防げたか: **防げなかった/防ぐべきでもない**。requirement は「何を・なぜ」を書く工程で、
未確定事項として research に送るのが正しい扱い。ただし**確認していない事実を断定形で書いた**
点は改めるべきで、「要確認」と明示すべきだった。

結果的にこの誤りは大きな分岐点だった（互恵条項の有無で実装方針が変わる）。
research を挟まず spec に直行していたら、**逐語移植してライセンス違反を犯していた**可能性がある。

### C3【自傷】`cd` の副作用でネストしたディレクトリを作った

前のコマンドで `cd packages/core` した状態が持続しており、`packages/core/packages/core/` に
ファイルを作ってしまった。テスト実行で即座に気づいて修正したが、無駄な往復が発生した。

前段で防げたか: **防げた**。作業ディレクトリを跨ぐ操作では絶対パスを使うか、
各コマンドで作業ディレクトリを明示すればよい。

### C4【設計の見落とし】ピュア層の規約を実装前に確認していなかった

`hostserver/` を実装してから lint に 4 件かかり、I/O を `transport/` へ移す
リファクタが必要になった（decisions D1）。結果的に良い設計に落ちたが、
**spec の段階で既存の `eslint.config.js` を読んでいれば**最初から正しい形で書けた。

前段で防げたか: **防げた**。spec で「既存資産を最大限使う」表を作った際、
lint 規約も確認対象に含めるべきだった。

## 改善提案

### 製品 / コード（→ issue 候補）

- `eslint` に `no-restricted-globals` を追加し、core のピュア層で `Buffer` /
  `process` / `setTimeout` 等の Node グローバルを禁止する。
  現状は `no-restricted-imports` のみで、**グローバル参照という抜け道**が空いている（C3 の must を参照）
- 既に backlog へ積んだもの: ライブラリ切り出しの前提となる負債 4 件
  （ロガー注入・`As400Error` 改名・`ErrorCode` 整理・CCSID テーブルの同梱単位）、
  DES 経路、PUB400 以外での検証

### PJ プロセス / 規約（→ AGENTS.md 等）

- **プロトコル実装では実機検証を単体テストの代替にしない**。今回 3 件の欠陥が実機のみで
  発見された。逆に「単体テストが緑だから正しい」と判断していたら、TLS 経路は壊れたまま出荷していた
- **spec の「既存資産の確認」に lint / アーキテクチャ規約を含める**。
  ライブラリや型だけでなく、`eslint.config.js` の制約も設計入力である（C4）
- **未確認の事実を requirement に断定形で書かない**。「要確認」と明示し research へ送る（C2）

### ハーネス自体（→ aidev-\* への提案・適用は人間）

- **`aidev event <工程> start` の時点で前提工程の承認を検査する**（または警告する）。
  現状 `guard` は別コマンドで、叩き忘れると C1 のような記録漏れが静かに発生する。
  protocol.md「2.」は guard の実行を求めているが、`event start` から強制されていない。
  `event start` が内部で guard 相当の検査を行い、未承認なら警告を出すだけでも防げる
- **`aidev verify` / `doctor` で「start はあるが approved が無い工程」を検出する**。
  retro が `metrics.yml` を読む時点で気づけたが、より早く（次工程の開始時に）気づきたい
- 補足: 今回 `state.yml` の `approved` と `metrics.yml` の齟齬が**同時に**発生した。
  両者は `aidev approve` が一括更新するため、片方だけ欠けることはない。
  逆に言えば **`approve` の実行漏れは両方に等しく現れる**ので、検出は容易なはず
