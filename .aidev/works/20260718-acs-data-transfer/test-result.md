## 誤パスワードの実機検証（追試）

当初はプロファイル無効化（QMAXSIGN）のリスクを避けて未実施としていたが、
ユーザーより「PUB400 で 1 回失敗しても無効化されなかった」実績の情報を得たため実施した。

安全側の根拠:
- プロトコル自体が安全弁を持つ。\`0x0003000B\`(誤り) と \`0x0003000C\`(次に誤ると無効化) が
  別コードなので、**サーバー応答から余裕の有無が分かる**
- 直前に複数回の認証成功があり、失敗カウンタはリセットされている見込みだった
- 手順: 1 回だけ試行 → 結果に関わらず直後に正規パスワードで成功させカウンタをリセット

結果:

\`\`\`
rc=0x0003000b  kind=password-incorrect  retryable=true  code=UNAUTHENTICATED
→ 直後に正規パスワードで認証成功（V7R5M0 / job 665482/QUSER/QZSOSIGN）
\`\`\`

- 一次資料（JTOpen の戻りコード表）と実機の値が一致
- 成功(0) と失敗が実機で区別できることを確認
- \`rc\` / \`kind\` / \`retryable\` が呼び出し側に届き、**文言ではなく値で分岐できる**ことを確認
- \`0x0003000C\` ではなかったため、失敗カウンタに余裕があることも確認できた

### 副次的な発見: 存在しないユーザー ID でも \`0x0003000B\` が返る

\`\`\`
user=NOSUCHUSR → rc=0x0003000b kind=password-incorrect
\`\`\`

\`0x00020001\`(ユーザー ID が不明) **ではない**。IBM i がユーザーの存在有無を漏らさないための
挙動と考えられる（ユーザー列挙対策）。

→ 分類表の \`user-unknown\` 分岐は**実運用ではまず発火しない**。表から消しはしないが
（他バージョン・他構成で返る可能性があるため）、この事実をコードに注記した。
