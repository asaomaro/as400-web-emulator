# 要件: ACS データ転送相当（ホストサーバー接続）— 第1段階: 認証貫通

## 背景 / 課題

IBM i Access Client Solutions（ACS）の「データ転送」は、テーブルを PC に落とし（ダウンロード）、
CSV 等を物理ファイルに取り込む（アップロード）機能である。本エミュレーターにも同等の機能が欲しい。

ただし ACS のデータ転送は **5250（TN5250）とはまったく別のプロトコル**で動く。
既存の `packages/core/src/{telnet,protocol}` は TN5250 専用であり、流用できる層はほぼない
（TLS 化した TCP トランスポート程度）。

必要なのは IBM i の**ホストサーバー**群への接続である。PUB400 で到達性を確認済み:

```
21 OPEN   449 OPEN   8470/8471 OPEN   8476 OPEN   9470/9471/9476 OPEN(TLS)
```

参照実装は **jt400（IBM Toolbox for Java, Apache-2.0）**。本作業はこれを読み解き、
TypeScript へ移植する方針をユーザーが選択した（FTP 等での代替は採らない）。

## 目的 / ゴール

**この作業の第1ゴールは「認証が実際に通ること」**に限定する。
signon サーバー（8476/9476）に接続し、パスワード置換値（password substitute）を正しく生成して
PUB400 の実アカウントで認証が成功する、を達成状態とする。

認証はホストサーバー接続の最初の関門であり、ここが通らなければダウンロードもアップロードも
一切成立しない。逆にここさえ通れば、以降は datastream の実装を積み増す作業になる。

## スコープ

### 対象

- ポートマッパー（449）によるサーバー実ポートの解決
- signon サーバー（8476 / TLS 9476）への接続と認証シーケンス
  - シード交換
  - パスワード置換値の生成（QPWDLVL に応じたアルゴリズム分岐）
  - 認証結果・ユーザー情報の取得
- ホストサーバー共通のデータストリーム基盤（ヘッダー・リクエスト/レスポンスの枠組み）
- TLS / 平文の両対応（既存の TN5250 側と同じ考え方で `tls` を切り替える）
- `packages/core` 内のライブラリとして実装し、**手元で叩ける CLI（またはスクリプト）で検証できる**こと
- 認証情報は**既存の接続設定を再利用**する（`profiles.json` / connections の `signon`）。
  新しい資格情報の保管場所は増やさない

### 対象外（この作業ではやらない）

- database サーバー（8471）への接続、SQL の発行、結果セットの取得（＝ダウンロード本体）
- 物理ファイルへの取り込み（＝アップロード本体）
- MCP ツールとしての公開
- Web UI からの操作（テーブル選択・CSV ダウンロード / ドロップしてアップロード）
- Kerberos / GSSAPI 認証、プロファイルトークン認証
- IFS ファイルサーバー（8473）、コマンドサーバー（8475）

> MCP ツール化と Web UI 組み込みは**最終的には行う**（ユーザーの意向）。
> ただし core が動いてから改めて別作業として起こす。

## 機能要件

- ホスト名・ポート・TLS 可否・ユーザー・パスワードを与えて signon サーバーへ接続できる
- ポートマッパー経由でサーバーポートを解決できる（固定ポート指定も可能）
- パスワード置換値を生成し、認証要求を送って**成功/失敗を判別できる**
- 認証失敗時、原因（ユーザー不明・パスワード不一致・プロファイル無効・パスワード期限切れ等）を
  IBM i が返すエラーコードから**区別できる形で**返す
- 通信内容をトレースできる（既存の `packages/core/src/trace` の考え方に合わせる）

## 非機能要件 / 制約

- 既存の TN5250 実装には影響を与えない（別モジュールとして追加する）
- パスワードを平文でログ・トレースに出さない（既存の秘匿方針を踏襲）
- パスワードを平文でファイルに保存しない（既存の `passwordEnc` / `passwordEnv` を使う）
- jt400 は Apache-2.0。移植にあたり**由来を明示**する（該当クラス名・出典をコード内に残す）
- 実機（PUB400）での検証を伴う。PUB400 のパスワードは環境変数経由でのみ扱う

## 完了条件 (受け入れ基準)

- [ ] ポートマッパー（449）に問い合わせて signon サーバーのポートを解決できる
- [ ] PUB400 の実アカウントで **signon サーバーの認証が成功する**（実機で確認）
- [ ] 誤ったパスワードでは失敗し、**正しいパスワードとは区別できるエラー**が返る
- [ ] TLS（9476）と平文（8476）の双方で接続できる
- [ ] 認証シーケンスの単体テストがある（実機に依存しない形で、既知のシードから期待される
      パスワード置換値が生成されることを検証する）
- [ ] パスワードがトレース出力に平文で現れないことをテストで担保する
- [ ] jt400 のどのクラス／どの仕様に対応するかがコード上で追える

## 未確定事項 / 確認したいこと

以下は **research 工程で事実確認する**（推測で spec を書かない）。

- jt400 のパスワード置換値の生成アルゴリズム（QPWDLVL 0/1 の DES 系と 2/3 の SHA-1 系の分岐条件と手順）
- ホストサーバーのデータストリーム（ヘッダー構造・リクエスト/レスポンスの形式・LL/CP の並び）
- ポートマッパー（449）の要求/応答フォーマット
- signon サーバーの正確なシーケンス（どの要求を何回、どの順で送るか）
- **PUB400 の QPWDLVL の値**（どのアルゴリズムを実装する必要があるか。実機で確認する）
- ユーザー ID / パスワードの文字コード変換（EBCDIC 変換の要否と CCSID。既存の CCSID 273 の
  知見が効くか）
- 移植元 jt400 のソースをどう入手・参照するか（ライセンス表記の扱いを含む）
