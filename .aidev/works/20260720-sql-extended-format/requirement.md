# 要件: SQL の拡張データ形式に対応し、LOB 列を含む結果セットを扱えるようにする

## 背景 / 課題

利用者から報告があった:

> 以下の SQL が STRSQL では実行できますが、web 画面からだと `data format too short: 0 bytes` となります。
> `SELECT * FROM QSYS2.SYSTABLES FETCH FIRST 100 ROWS ONLY`

調査で **2 つの独立した問題**が重なっていたことが分かり、1 つ目
（行数がブロッキング係数の倍数のとき落ちる）は #99 で修正済み。**本作業は 2 つ目**。

### 当初の説明は誤りだった

#99 の時点では「**LOB 列を含む結果セットはプロトコル上未対応**（ロケーターが要る）」と
説明し、そういうエラーメッセージにした。**これは不正確だった。**

原典（JTOpen 本体）の `AS400JDBCConnectionImpl.java:4355-4370` を読むと、
JDBC ドライバーは SQL 属性設定時に次を**セットで**送っている:

- **`setUseExtendedFormatsIndicator(0xF2)`**（CP `0x3821`・1 バイト。V5R4 以降は 0xF2、以前は 0xF1）
- `setLOBFieldThreshold(15728640)`（CP `0x3822`・4 バイト）

実機（PUB400・IBM i 7.5）で `0x3821` を送ってみたところ、**`prepare` が成功した**
（`rcClass=0, code=0`）。つまり:

**LOB が扱えなかったのは、拡張データ形式を有効にしていなかったから。**

有効にすると列定義が **CP `0x3812`（拡張形式）** で返る。これは backlog に
「拡張形式の応答（列定義 CP 0x3812 / 結果 CP 0x380E）の解析」として未実装のまま
積まれていた項目そのもの。当時「実機が元形式でしか返さなかった」と記録されているが、
それは**拡張形式を要求していなかったから**であって、実機の制約ではなかった。

## 目的 / ゴール

拡張データ形式を解析できるようにし、**LOB 列を含む結果セットを取得できる**ようにする。

利用者の提案（「LOB 未対応で SQL 全体をエラーにするのではなく、値なしのカラムとして
受信し表示できないのか」）に対する答えとして、**まず列が取れる状態**を作る。

## スコープ

### 対象

- **拡張データ形式（列定義 CP `0x3812`）の解析**
- **拡張結果データ（CP `0x380E`）の解析**（必要なら）
- `useExtendedFormats`（CP `0x3821`）と `lobFieldThreshold`（CP `0x3822`）の送信
- LOB 列の値の扱い（しきい値以下は inline で取得できるか実機で確認する）

### 対象外

- **ロケーター経由の LOB 取得**（しきい値を超える大きな LOB）。
  必要になったら別作業。まず「列が取れる」ところまで
- LOB の書き込み（DDM 側は別の話）

## 要検討（spec で決める）★最重要

### 1. 既存経路の回帰リスク

**拡張形式を有効にすると、すべてのクエリが新形式で返る。**
既存の元形式パーサー（`parseDataFormat` / `parseResultData`）だけでは**全滅する**。

現在この部分には **core 531 件のテストと実機検証**があり、SQL は MCP・Web UI・
DDM の列レイアウト取得（`SYSCOLUMNS` 参照）から使われている。**壊すと影響が広い。**

取りうる形:

- (a) 両形式を扱えるようにし、**常に拡張形式を要求する**
- (b) 既定は元形式のままで、**必要なときだけ**拡張形式に切り替える
- (c) 常に拡張形式を要求し、元形式のパーサーは互換のため残す

**どれを採るかは spec で決める。** 「新しい方が良い」で選ばない——
既存の動作実績をどう守るかが判断軸。

### 2. LOB の値をどう返すか

しきい値以下なら inline で取れる見込みだが**未確認**。実機で確かめる。

- 取れるなら、そのまま値として返す
- 取れない（ロケーターが返る）なら、**その旨が分かる形**で返す。
  `null` で潰すと「値が無い」のか「取得できない」のか区別できない

### 3. 型の網羅

拡張形式では列メタデータの構造が変わる。**現在対応している全型が引き続き正しく
取得できること**を実機で確認する必要がある（`20260718-hostserver-sql` は
「実機で全型の取得を確認」と記録している。その水準を落とさない）。
