# 決定記録

## D1. 常に超拡張形式を要求する（元形式のパーサーは残す）

接続時に `useExtendedFormats = 0xF2`（CP `0x3821`）を固定で送る。

**根拠**:

1. **切り替え条件を持てない。** 「LOB を含むか」は prepare するまで分からない。
   失敗してから再試行する形は、状態が増えるうえ 1 クエリで 2 往復になる
   （実機で 1 往復 4〜7 秒かかる経路）
2. **原典もそうしている**（接続時に一度立てて以降は常に拡張形式）
3. **元形式のパーサーは後退の余地として残す**。現在 IBM i 7.5 の 1 台でしか
   検証しておらず、`0xF2` を拒むホストが見つかったときに戻せるようにしておく

> レビューで「残してあるが到達しない＝実質デッドコード」と自分で指摘した。
> **消すのは「そういうホストが無い」と確かめてから**にする。

## D2. LOB のしきい値は 0 に固定し、オプションにもしない ★

**実測が決め手**。`SELECT *`（DBCLOB(2M) を含む）2 行の fetch 応答:

| しきい値 | 応答サイズ |
|---|---|
| 15,728,640（原典の上限値） | **8,398,878 バイト** |
| **0（採用）** | **10,270 バイト** |

しきい値以下の LOB は**インラインで丸ごと返る**。100 行なら数百 MB になり、
MCP の応答にも Web UI にも載らない。

**利用者に開放しない**——「大きくすると静かにメモリを食い尽くす」種類の設定だから。
必要になったらロケーター取得を実装するほうが筋が良い。

> **原典の値をそのまま持ってきていたら本番で踏んでいた。**
> 定数は「原典がそう書いているから」ではなく、**自分の用途で測ってから**決める。

## D3. LOB は `null` にせず専用の型で返す

```ts
export interface LobPlaceholder { kind: "lob"; locator: number; maxSize: number; }
export type DbValue = string | number | bigint | null | LobPlaceholder;
```

**根拠**: `null` にすると **SQL の NULL と「取得していない」が区別できなくなる**。
実機で NULL・LOB・空文字の 3 つが同じ表に並ぶことを確認しており、
これらが混ざると利用者は判断できない。

画面では `(LOB)`、CSV でも `(LOB)`（空欄にすると NULL と混ざる）。

これが利用者の提案「値なしのカラムとして受信し表示できないのか」への回答にあたる。

## D4. LOB の判定は型コードで行う（列定義ではない）

当初は列定義の `lobLocator`（基点+17）を見ていたが、**実機ではその値は 0** で、
**ロケーターのハンドルは行データの中に 4 バイト**入り、列の型コードが
`*_LOCATOR`（960 / 964 / 968）になる。

原典に `getFieldLOBLocator` があることに引きずられて「列定義に入っている」と
思い込んだ。**実機が `unsupported type CLOB_LOCATOR (964)` で教えてくれた。**

## D5. `maxSize` が 0 でも位置を動かさない

原典の `getFieldLOBMaxSize`（基点+26）を読んでいるが、実機では常に 0 だった。
値が入る条件が分かっていない。

**推測でオフセットを動かさない。** 0 のまま返し、必要になったら実機で調べる
（`20260718-hostserver-ifs` の「整理に見える変更が退行だった」の教訓）。

## D6. #99 のエラーメッセージを訂正する

#99 で「**LOB 列を含む結果セットは未対応**（ロケーター経由の取得を実装していないため）」
という説明を入れたが、**本作業で誤りと判明した**。実際は拡張形式を要求していなかっただけ。

誤った説明を残さない。汎用の文言に差し替えた。
