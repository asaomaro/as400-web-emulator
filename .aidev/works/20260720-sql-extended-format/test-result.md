## 実機検証（PUB400・IBM i 7.5・TLS）

### 1. 報告されたクエリ ★

`SELECT * FROM QSYS2.SYSTABLES FETCH FIRST 100 ROWS ONLY`

**画面から 100 行 × 33 列**で表示された（所要 11.9 秒）。修正前は
`data format too short: 0 bytes` で全滅していた。

### 2. 既存の全型の回帰（最大のリスク）

拡張形式を有効にすると**すべてのクエリ**が新形式で返るため、
`20260718-hostserver-sql` が「実機で全型の取得を確認」した水準を落としていないか確かめた。

| 型 | 値 |
|---|---|
| CHAR(5) | `"abc  "` |
| VARCHAR(10) | `"xy"` |
| DECIMAL(7,2) | `"-12.34"` |
| NUMERIC(5,0) | `"99"` |
| SMALLINT / INTEGER | `7` / `70000` |
| BIGINT | `9000000000n`（bigint） |
| DATE / TIME / TIMESTAMP | `"2026-07-20"` / `"03:24:27"` / `"2026-07-20-03.24.27.590079"` |

**すべて元形式のときと同じ値**。DBCS を含む `TABLE_TEXT` も正しく取得できた。

### 3. LOB の表現

自作の `CLOB(100)` と、実在の MQT（`TABLE_TYPE='M'`）で確認:

```
{"ID":1,"C":{"kind":"lob","locator":256,"maxSize":0},"TXT":"plain"}
{"ID":2,"C":null,"TXT":null}          ← SQL の NULL
{"ID":3,"C":{"kind":"lob","locator":513,...},"TXT":""}  ← 空文字
```

**LOB・NULL・空文字が明確に区別できている**（利用者の提案「値なしのカラムとして
受信し表示」への回答）。画面では `(LOB)` と淡色表示、CSV でも `(LOB)`:

```
行: 3 / (LOB) セル: 3 / NULL セル: 0
CSV: TABLE_NAME,MQT_DEFINITION,TABLE_TEXT / EXAMPLE_MQT1,(LOB), / MQT1,(LOB),
```

CSV で LOB を空欄にすると NULL と混ざるため `(LOB)` と書いている。

### 4. DDM の回帰（SQL のメタデータに依存している）

`20260719-hostserver-upload-ddm` は列レイアウトを `QSYS2.SYSCOLUMNS` から計算するため、
SQL の形式変更の影響を直接受ける。往復を再実行した:

```
2. レイアウト（SQL 由来）: NAME:char@0+10 QTY:packed@10+3 AMT:zoned@13+7 SEQ:int@20+4 NOTE:char@24+5 = 29 バイト
3. open: recordLength=29 ✅ SQL 由来のレコード長がホストの申告と一致
5. SQL で読み返し: 3 件（NULL と空文字の区別も維持）
```

**無傷。**

### 5. 応答サイズ（しきい値の妥当性）

spec D2 の根拠となった実測を再掲。`SELECT *`（DBCLOB(2M) を含む）2 行で:

| `lobFieldThreshold` | 応答 |
|---|---|
| 15,728,640（JTOpen の上限） | **8,398,878 バイト** |
| **0（採用）** | **10,270 バイト** |

100 行でも実用の範囲に収まることを画面で確認（33 列 100 行が 11.9 秒で表示）。

## 実装中に見つけた自分の誤り

**LOB の検出を列定義の `lobLocator` で行っていた。** 実際にはその値は 0 で、
**ロケーターのハンドルは行データの中に 4 バイト**で入り、列の型コードが
`*_LOCATOR`（960/964/968）になる。型コードで判定するよう直した。

実機で `column "C" has unsupported type CLOB_LOCATOR (964)` が出て気づいた。

## 実機の後片付け

検証で作った `MARO1/ZZUI` / `ZZLOB` / `ZZLOB2` は削除済み。

> 途中、エラーで終了したスクリプトが `DLTF` に到達せず表が残り、
> 次回実行で行が積み増しされて「行が 2 倍に見える」誤解をした。
> **解析の欠陥かと思ったが、テストの後片付け漏れだった**（ヘッダーを実測して切り分けた）。

## 未検証の範囲（引き継ぎ）

- **ロケーター経由の LOB 本体の取得**（`retrieveLOBData`）。本作業の対象外
- `0xF2` を受け付けないホスト（V5R4 未満）。原典は `0xF1` を送る分岐を持つが、
  **こちらは 0xF2 固定**。古いホストに当たるまで分岐を足さない（推測で作らない）
- 元形式のパーサーは残してあるが、**現在どのクエリも通らない経路**になった。
  0xF2 を拒むホストが見つかったときの後退手段として保持している
- 圧縮された結果データ（原典には `dataCompressed_` / `vlfCompressed_` の分岐がある）。
  実機では非圧縮で返っており未対応
