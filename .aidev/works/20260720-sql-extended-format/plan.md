# 計画: 超拡張データ形式

## 分割の判定

**subtask にしない。** 解析器 1 つ＋配線で 1 PR に収まる。

## 順序

1. `db-reply-ext.ts`（純関数。実バイト列を固定値にした単体テストで固める）
2. `DbValue` に `LobPlaceholder` を足す（型が波及するので早めに）
3. `db-connection.ts` で `0x3821` / `0x3822` を送る
4. `query.ts` を両形式対応にする
5. **実機で全型の回帰**（ここが本丸）
6. Web UI / CSV の表示、MCP の JSON
7. #99 のメッセージ訂正 ／ backlog 訂正

**1 を最初に**やるのは、実機に出る前に固められる唯一の部分だから
（DDM のときと同じ理由）。

## リスク

| リスク | 兆候 | 対応 |
|---|---|---|
| **既存の全型が壊れる**（最大のリスク） | 実機で値が変わる | 5 で全型を実機比較。**変わったら止めて記録する**（推測で直さない） |
| 応答が巨大化 | 実機 | しきい値 0 を実測で確認済み。100 行でも測る |
| DDM の列レイアウト取得が壊れる | DDM の実機検証 | DDM の往復テストを再実行する |
| `LobPlaceholder` の型波及 | tsc | web-ui / server まで通す |
