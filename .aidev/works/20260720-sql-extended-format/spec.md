# 仕様: SQL の超拡張データ形式に対応する

## D1. 常に超拡張形式を要求する（元形式のパーサーは残す）

requirement 要検討 1 の (a)/(b)/(c) のうち **(c)** を採る。

- 接続時に `useExtendedFormats = 0xF2`（CP `0x3821`）を送る
- 列定義は `0x3812`、結果データは `0x380E` を解析する
- **元形式（`0x3805` / `0x3806`）のパーサーは消さない**

**根拠**:

1. **形式を切り替える条件を持ちたくない。** (b)（必要なときだけ拡張）は
   「LOB を含むかを事前に知る」必要があるが、**それは prepare するまで分からない**。
   失敗してから再試行する形になり、状態が増えるうえ 1 クエリで 2 往復になる
   （実機で 1 往復 4〜7 秒かかる経路である）
2. **原典もそうしている。** JTOpen は接続時に一度 `0xF2` を立て、以降は常に拡張形式
3. **元形式のパーサーを残すのは後退の余地**。`0xF2` を受け付けないホストが
   見つかったときに戻せる。削除は、そういうホストが**無いと確かめてから**でよい
   （現在 IBM i 7.5 の 1 台でしか検証していない）

## D2. LOB のしきい値は **0**（＝常にロケーター）

**研究 F6 の実測が決め手**:

| `lobFieldThreshold` | 2 行の fetch 応答 |
|---|---|
| 15,728,640（JTOpen の上限） | **8.4 MB** |
| **0** | **10 KB** |

しきい値以下の LOB は**インラインで丸ごと返る**。`DBCLOB(2M)` の表なら
100 行で数百 MB になり、MCP の応答にも Web UI にも載せられない。

**しきい値を利用者に開放しない**（オプションにもしない）。理由は、
「大きくすると静かにメモリを食い尽くす」種類の設定だから。
必要になったらロケーター取得を実装するほうが筋が良い。

## D3. LOB 列は「取得していない」と分かる形で返す

`DbValue` に `null` を入れない。**`null`（SQL の NULL）と区別できなくなる**ため。

```ts
/** LOB 列。値そのものではなくロケーターしか得ていない */
export interface LobPlaceholder {
  kind: "lob";
  /** ロケーターのハンドル。将来 retrieveLOBData で本体を取るのに使う */
  locator: number;
  /** 列定義が申告する最大サイズ */
  maxSize: number;
}
export type DbValue = string | number | bigint | null | LobPlaceholder;
```

利用者の提案（「値なしのカラムとして受信し表示」）への回答がこれ。
**列は取れる・行も取れる・LOB の値だけ来ていないことが型で分かる。**

- MCP / HTTP の JSON では `{"kind":"lob","locator":…,"maxSize":…}` として出す
- Web UI の表では `(LOB)` と淡色表示し、NULL とも空文字とも区別する
- CSV では `(LOB)` と書く（値が無いのに空欄にすると NULL と混ざる）

## D4. 列名は可変長部から取る

超拡張形式は列名を固定部に持たない（research F3）。
`16 + i*48 + <可変長オフセット>` から `LL(4) CP(2) CCSID(2) 名前…` を辿り、
CP `0x3840` を探して `LL - 8` バイトを名前として読む。

**見つからなければ空文字**にする（原典も同じ。名前が無いことを例外にしない——
式や関数の結果列には名前が無いことがある）。

## 2. ファイル構成

```
packages/core/src/hostserver/db/
  db-reply.ts          既存。元形式の解析（残す）
  db-reply-ext.ts      新規。超拡張形式（0x3812）と拡張結果データ（0x380E）の解析
  db-decode.ts         LobPlaceholder を返す分岐を追加
  db-connection.ts     0x3821 / 0x3822 を送る
  query.ts             0x3812 → 0x380E を優先し、無ければ従来の 0x3805 / 0x3806
```

`query.ts` は**両方を見る**:

```ts
const raw = findParam(reply, DB_CP.superExtendedDataFormat) ?? findParam(reply, DB_CP.dataFormat);
```
