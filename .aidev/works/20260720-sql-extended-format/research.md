# 調査: SQL の拡張データ形式

**原典を直読した**（AGENTS.md「知識ベースや類推で書き始めない」）。
今回は `archived/jtopenlite/` ではなく **JTOpen 本体**（`src/main/java/com/ibm/as400/access/`）を
sparse-checkout して読んだ——jtopenlite は LOB について
「LOBs and LOB locators have not been extensively tested」と自ら断っており、
ACS が使うのは本体側だから。

## F1. 3 つの形式がある（元 / 拡張 / 超拡張）

`DBBaseReplyDS.java` の応答分岐から:

| CP | クラス | 意味 |
|---|---|---|
| `0x3805` | `DBOriginalDataFormat`（380 行） | **元形式**。現在の実装が解析しているもの |
| （拡張） | `DBExtendedDataFormat`（390 行） | 拡張形式 |
| **`0x3812`** | **`DBSuperExtendedDataFormat`（457 行）** | **超拡張形式**（V5R4 以降） |
| `0x3806` | 元の結果データ | 現在の実装 |
| **`0x380E`** | `DBExtendedData` | 拡張結果データ |

どれが返るかは **`useExtendedFormatsIndicator`（CP `0x3821`・1 バイト）**で決まる。
`AS400JDBCConnectionImpl.java:4355-4358`:

- V5R4 以降 → **`0xF2`**（super extended）
- それ以前 → `0xF1`（extended）

そして**同じ場所で** `setLOBFieldThreshold`（CP `0x3822`・4 バイト・最大 15,728,640）を送る。

> つまり LOB は「拡張形式を有効にすること」とセットで扱われる。
> **backlog の「実機が元形式でしか返さなかった」は誤り**で、
> 実機の制約ではなく**こちらが要求していなかった**だけだった。

## F2. 実機で確認した事実（PUB400・IBM i 7.5）

| 送ったもの | prepare の結果 | 返った CP |
|---|---|---|
| 現状（`0x3821` なし） | **失敗** `rcClass=7, code=-101` | `0x3805=`（空）`0x3811=`（空） |
| `0x3821=0xF2` ＋ `0x3822=15728640` | **成功** `rcClass=0, code=0` | `0x3807`（SQLCA）`0x3811` **`0x3812`** |

`SELECT * FROM QSYS2.SYSTABLES`（DBCLOB を含む）が **prepare を通り**、
列定義が **`0x3812`（超拡張）** で返ってきた。応答長も 58 → **2,488 バイト**に増えた。

また **LOB は型の問題であってサイズの問題ではない**ことも確認済み——
自分で作った `CLOB(100)`（100 バイト）でも同じエラーになり、`CAST(... AS VARCHAR)` なら通った。

## F3. 超拡張形式のバイト配置（`DBSuperExtendedDataFormat.java`）

固定部（`FIXED_LENGTH_ = 16`）:

| オフセット | 内容 |
|---|---|
| +0 | consistency token（4） |
| **+4** | **フィールド数（4）** |
| +8 / +9 | 日付形式 / 時刻形式（各 1） |
| +10 / +11 | 日付区切り / 時刻区切り（各 1） |
| **+12** | **レコードサイズ（4）** |

繰り返し部（`REPEATED_FIXED_LENGTH_ = 48` バイト／列。基点は `16 + i*48`）:

| 絶対オフセット | 基点からの位置 | 内容 |
|---|---|---|
| +18 | +2 | SQL 型（2） |
| +20 | +4 | 長さ（4） |
| +24 | +8 | 位取り（2） |
| +26 | +10 | 精度（2） |
| +28 | +12 | CCSID（2） |
| +30 | +14 | パラメータ種別（1） |
| **+33** | **+17** | **LOB ロケーター（4）** |
| +37 | +21 | フラグ（4。bit30=配列 / bit27=XML の DBCS） |
| **+42** | **+26** | **LOB 最大サイズ（4）** |
| +48 | +32 | 可変長情報へのオフセット（4） |

**列名は可変長部**にある。`16 + i*48 + <可変長オフセット>` から
`LL(4) CP(2) 列名CCSID(2) 名前…` の形で、CP **`0x3840`** が列名。
名前の長さは `LL - 8`。

→ 元形式（現在の実装）とは**構造が根本的に違う**。既存の `parseDataFormat` は流用できない。

## F4. 回帰リスクの実測

拡張形式を有効にすると**すべてのクエリ**が新形式で返る。現在この経路に乗っているもの:

- `packages/core` の `query` / `stream`
- MCP の `host_sql`
- Web UI の SQL ペイン（`/api/host/sql`）
- **DDM の列レイアウト取得**（`20260719-hostserver-upload-ddm` が `QSYS2.SYSCOLUMNS` を引く）

core のテストは 531 件。SQL 周辺は実機で全型を確認した実績がある
（`20260718-hostserver-sql`「実機で全型の取得を確認」）。**その水準を落とさない**必要がある。

## F5. LOB の値が取れるかは未確認

`getFieldLOBLocator` / `getFieldLOBMaxSize` が列メタデータにあるので、
**LOB 列はロケーターとして記述される**可能性が高い。しきい値以下なら inline で
返るのか、常にロケーターなのかは**実機で確かめる**（推測しない）。

**ロケーターしか得られない場合に何を返すか**を spec で決める必要がある。

## F6. 実機で確かめた「未確認」2 点（spec の前に潰した）

プローブで prepare → open → fetch を直接叩き、返る CP を観測した。

| 段階 | 返った CP |
|---|---|
| prepare | `0x3807`(SQLCA) **`0x3812`**(2,294B) `0x3811`(0B) |
| open | `0x3807` **`0x3812`** |
| fetch | `0x3807` **`0x380e`** |

→ **結果データも `0x3806` から `0x380E` に変わる。** 両方の解析が要る。

### ★ しきい値の設定が致命的に効く

`SELECT * FROM QSYS2.SYSTABLES FETCH FIRST 2 ROWS ONLY`（DBCLOB(2M) を含む）で:

| `lobFieldThreshold` | fetch の応答サイズ |
|---|---|
| 15,728,640（JTOpen の既定上限） | **8,398,878 バイト（8.4 MB / 2 行）** |
| **0** | **10,270 バイト（10 KB / 2 行）** |

しきい値以下の LOB は**インラインで丸ごと返る**。2 行で 8.4 MB ということは、
100 行なら数百 MB になる。**既定を大きくしてはいけない。**

しきい値 0（＝常にロケーター）なら実用的なサイズに収まる。
ロケーターは「値そのものではなくハンドル」なので、本体を取るには別要求が要る（本作業の対象外）。

→ **既定はしきい値 0（ロケーター）**とし、LOB 列は「取得していない」ことが分かる形で返す。
   これが利用者の提案（「値なしのカラムとして受信し表示」）に対する現実的な答えになる。
