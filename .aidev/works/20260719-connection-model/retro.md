## うまくいった点

**解決点の集約が、想定を超えて効いた。** `ConfigResolver` を単一の解決点にし `warn` を必須引数に
したところ、WS の E2E が `secret key not configured (auto-signon skipped)` の警告つきで落ちた。
旧実装は 5 経路中 3 経路で `warn` 未配線だったので、**同じ状況が無言のまま
「なぜかサインオンされない」として現れていた**はず。設計判断がそのまま実地で報われた。

**移行を純関数に切ったのが正解だった。** 実データの期待結果（1 システム + 3 セッション）を
I/O なしでテストに固定でき、後段の前提が最初から安定した。

**「旧ルートから移し替える際に落とした検査は無いか」を review の第一観点にしたのが当たった。**
`20` の M1（更新でパスワードが黙って消える）はこの観点でしか出ない。新機能のバグより、
既存の防御が静かに消えるほうが危険。

**統合 review を独立して置いた価値が出た。** M1（`verify-browser*.mjs` 6 本が実行時に落ちる）は
`20` の担当範囲（サーバー面）からも `30` の担当範囲（UI）からも**自分の担当外に見える位置**にあった。
slice ごとの review だけでは構造的に見つからない。

## 課題 / 手戻り

### C1: 自動テストが 1,000 件緑でも、見た目と操作の欠陥は残る（最大の学び）

subtask の review をすべて通過した後、利用者が動かして **20 件超**の指摘が出た。

| 分類 | 例 |
|---|---|
| 見た目 | フォーム部品にグローバル既定が無く、白い部品がダークテーマに浮いていた |
| 情報設計 | ドロップダウンの「システムを選択」がシステム名に見える |
| 復活漏れ | カード/一覧トグル・詳細ⓘ・セッション種別アイコン |
| 機能不具合 | F1 で待機が解けない・ログ 0 件・D&D の目印・初期取得漏れ |
| 自分の回帰 | フッターがログに覆われた・ログの絞り込みが実 ID と不一致 |

**前段で防げたか**: 一部は防げた。「復活漏れ」3 件は、`ConnectView.vue`（875 行）を削除する前に
**機能の棚卸しをしていれば**気づけた。plan には「必要な部分を `ConfigCard` へ移す。落とすと
入力検証が消える」と書いていたが、**検証ロジックにしか注意が向いておらず、UI 機能の一覧を作らなかった**。

「見た目」「情報設計」は前段では防げない。実物を見るまで判断材料が無い。

### C2: 静的なモックでは見つからない欠陥がある

`ui-proposal.html` を 3 版まで作り込み、レイアウト・情報設計・遷移まで検討した。
それでも「**ランチャーを開くとワークスペースごと隠れて戻り道が無い**」は見つからなかった。

原因は明確で、**画面図には「いまどちらを表示しているか」という状態が無い**。
静的な図は各画面を独立に描くため、画面間の遷移可能性（どこからどこへ戻れるか）を表現できない。

### C3: research の影響範囲リストが不完全だった

`scripts/verify-mcp.mjs` と `verify-ws.mjs` は挙げたが、`verify-browser*.mjs` の 6 本を落とした。
grep の対象を「MCP ツールの引数名」に絞ったため、`ProfileStore` を import しているだけの
スクリプトが漏れた。**「変更する API 名」ではなく「削除する型名」でも探すべきだった。**

### C4: subtask 分割の粒度は妥当だったが、UI の反復は工程外で起きた

3 分割そのものは機能した（依存が一方向、各 slice が単独で検証可能）。
ただし `30-web-ui` の review 通過後に 20 件超の指摘があり、**その修正は工程の外**で行われた。
aidev の工程は「review → deliver」で終わるため、**実機フィードバックによる反復を表現する場所がない**。

## 改善提案

### 製品 / コード（→ issue 候補）

- **PUB400 停止で未確認の実機検証**を、復旧後に通す（`key-done` / 接続エラーの説明文 /
  UI 再設計の後半 / `verify-browser*.mjs` 6 本）
- グローバルなフォーム部品スタイルを入れた副作用として、`.grid-input` の角丸のように
  **打ち消していない属性だけが漏れる**問題が起きた。他にも埋もれている可能性があるので、
  CRT 系コンポーネントを一巡して確認する
- `visibleTabs` は返り値をコピーにしたが、`GroupNode.tabs` 自体は並び替え・分割から直接触られる。
  **タブ操作全体を不変にする**のは別テーマとして残る

### PJ プロセス / 規約（→ AGENTS.md）

- **UI コンポーネントを削除・置換するときは、事前に機能の棚卸し表を作る。**
  「検証ロジックを移す」だけでは不十分で、利用者が使っている機能（表示切替・詳細表示・
  アイコンなど）が静かに消える。C1 の「復活漏れ」3 件はこれで防げた
- **既存の型・クラスを削除するときは、API 名ではなく型名で grep する。**
  C3 の直接原因。`scripts/` のような自動テスト対象外のコードは、実行するまで壊れが分からない
- **グローバルな CSS 既定を足したら、既存コンポーネントへの漏れを確認する。**
  「打ち消していない属性だけが効く」形で出るため、見つけにくい

### ハーネス自体（→ aidev-* への提案・適用は人間）

- **実機・実ブラウザでのフィードバック反復を表現する工程が無い。**
  現状は review → deliver で閉じるが、UI を含む作業では deliver 相当の成果物を
  利用者が触ってから 20 件超の指摘が出た。これを「差し戻し」として扱うと工程が何往復もするし、
  工程外の作業として扱うと記録が残らない。
  **`aidev-70-trial`（試用）のような任意工程**を review と deliver の間に置き、
  指摘と修正を 1 つの記録にまとめられると、今回のような作業の実態を残せる
- **test 工程の「未検証の穴」の扱いを強めたい。** 今回は test.md に「ブラウザでの実クリック確認は
  行っていない」と明記したが、それが review を通過してしまう。**目視・実機の未検証が残る場合は
  deliver のゲートで明示的な確認を求める**形のほうが、実態に合う
- **統合 review の観点として「slice の狭間」を明文化したい。** 今回の M1 は
  「どちらの担当範囲からも自分の外に見える位置」にあった。統合 review の skill に
  「各 slice の担当境界の外側にあるファイル（スクリプト・CI 設定・ドキュメント）を列挙して確認する」
  という観点を足すと、同種の見落としを構造的に潰せる
