# 統合レビュー: 接続設定の分離

subtask 単体の review は各 slice で実施済み（`10` must2/should1、`20` must2/should1、`30` must1/should3/nit1）。
ここでは**結合起因の欠陥**——slice をまたぐ不整合だけを見る。

差分: 58 ファイル変更（+1,683 / −3,435）、新規 22 ファイル。

## must

**M1: `verify-browser*.mjs` 6 本が実行時に落ちる**

`20-server-surfaces` で旧ストアを削除した際、research が挙げた `verify-mcp.mjs` /
`verify-ws.mjs` の 2 本だけを更新していた。**research の影響範囲リストが不完全**で、
`scripts/verify-browser*.mjs` の一群（6 本）が漏れていた。

実行して確認した:

```
SyntaxError: The requested module '@as400web/server'
  does not provide an export named 'ProfileStore'
```

構文チェックでは通るため、**実行するまで気づけない**種類の破損。
自動テストの対象外（実機とブラウザを要するスクリプト）なので CI でも捉えられない。

→ **修正**: 6 本すべてを新モデルへ。旧形式の profiles 配列を受けて `ConfigResolver` を
組み立てる `buildResolver()` を各スクリプトに置いた（装置名を書き換えてから渡す既存の使い方を保つため）。
`scripts/` 全 17 本に旧参照が残っていないことと、必要な export が実在することを確認済み。

**これは slice をまたいだからこそ出た欠陥**である。`20` の担当範囲は「サーバー面」で、
ブラウザ検証スクリプトは `30` の範囲に見える。どちらの slice の review でも自分の担当外として
見落とす位置にあった。統合 review の存在意義がそのまま出た形。

## should

PUB400 停止後に入れた変更（F1 で待ちが解けない不具合の修正）で、
WebSocket に `key-done` を追加したが、ドキュメント更新が `20` の時点で終わっていたため反映漏れ。

→ **修正**: README の WebSocket の節に、画面を返さないキーがあることと、
その場合に `key-done` で完了を通知することを追記。

## 確認したが問題なしと判断した点

- **3 系統の命名整合**: MCP / WebSocket / REST に `profile` / `connection` の残骸なし
  （コメント内の履歴説明を除く）。同一の変更として入っている
- **`describeSocketError` / `withSocketHint`** は 3 経路（ホストサーバー・ポートマッパー・telnet）
  すべてに配線済み。未使用の export ではない
- **信頼境界 5 層**は 5 層それぞれに独立したテストがある。層をまたぐ抜けは見当たらない
- **移行の非破壊性**: 実サーバーで `profiles.local.json` が旧形式のまま残ることを確認済み
- `paneLabels.ts` の共有はタブ帯のみが参照（パンくずは固定名に変えたため不要になった）。
  重複定義は残っていない

## 実機・実ブラウザで出た指摘について（統合の観点）

subtask の review をすべて通過した後、利用者が動かして **20 件以上**の指摘が出た。
内訳は test.md に記載。統合レビューの観点で重要なのは次の 2 点。

**1. 自動テストと HTTP 経路の確認では、見た目と操作の欠陥を捉えられない。**
「フォーム部品にグローバル既定が無く白い部品が浮く」「ドロップダウンの選択肢がシステム名に見える」
「ランチャーから戻れない」——いずれも 1,000 件超のテストが緑のまま存在していた。

**2. 静的なモックでも捉えられないものがある。**
`ui-proposal.html` は 3 版まで作り込んだが、「ランチャーを開くとワークスペースが隠れて
戻り道が無い」は見つからなかった。**画面図には「いまどちらを表示しているか」という状態が無い**ため。

これは retro に送る。
