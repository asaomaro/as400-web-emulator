# 要件: 接続設定の分離（システム / セッション設定）

## 背景 / 課題

現在、1 つの接続設定に**性質の違う 3 つ**が混ざっている。

```
接続先    host / port / tls          ← どこへ
資格情報  signonUser / passwordEnc   ← 誰として
セッション deviceName / screenSize /  ← どう使うか
          sessionType / printer 出力
```

**既に歪みが出ている。** 一覧 UI（#81）を作った際、
「5250 表示セッション（装置 WEBEMU01）」の資格情報をコマンドサーバー接続に流用した。
装置名や画面サイズを持つ設定が、それらと無関係なサーバーへの認証情報の置き場になっている。

同じ問題が今後の機能すべてに出る:

| 機能 | 使うサーバー | 装置名・画面サイズ |
|---|---|---|
| 5250 セッション | telnet 23/992 | 要る |
| プリンター | telnet | 装置名だけ要る |
| SQL・IFS・一覧・スプール | 8471/8473/8475/8474 | **要らない** |
| アップロード（DDM） | 446 | **要らない** |

さらに、資格情報が設定ごとに重複するため、**パスワード変更時に全設定を直す**必要がある。

## 目的 / ゴール

**接続先＋資格情報**（システム）と、**セッション固有の設定**を分離し、
すべての機能がシステムを参照する形にする。

ACS も「システム構成」と「セッション」を分けており、代替を目指す本 PJ の構造として自然。

## スコープ

### 対象

- **システム**の概念を新設（名前 / host / port / tls / ユーザー / パスワード / CCSID）
- **セッション設定**をシステム参照に変更（名前 / システム参照 / deviceName / screenSize / printer 出力）
- 既存データの**自動移行**（1 設定 → システム 1 ＋ セッション設定 1）
- すべての利用箇所の追従（5250 / プリンター / SQL / IFS / 一覧 / MCP / Web UI）
- サーバー設定（`profiles.json`）と個人設定（`connections.json`）の両方

### 対象外

- 1 つのシステムに複数セッション設定をぶら下げる UI の作り込み（構造は許すが UI は最小限）
- アップロード（DDM）— 別作業
- 認証・権限モデルの変更（既存の「サーバー設定は admin のみ」等はそのまま）

## 機能要件

- システムを作成・編集・削除できる（名前・host・port・tls・ユーザー・パスワード・CCSID）
- セッション設定はシステムを選んで作る（装置名・画面サイズ・プリンター出力）
- 5250 / プリンターの接続はセッション設定から、その参照先システムの資格情報で行う
- SQL・IFS・一覧などは**システムだけを選べば使える**（セッション設定を要求しない）
- **既存の設定ファイルを自動で分解して読み込む**（利用者に手作業を求めない）
- パスワードは 1 箇所（システム）だけで管理される

## 非機能要件 / 制約

- **既存の信頼境界を維持する**——printer 出力系（autoPdfDir 等）はユーザー入力から注入させない
- 認可はサーバーで担保（サーバー設定は admin のみ、個人設定は本人のみ）
- パスワードは平文で保存しない（既存の暗号化を踏襲）
- 移行は**読み込み時に行い、書き戻しは明示的な保存時のみ**（勝手にファイルを書き換えない）

## 完了条件 (受け入れ基準)

- [ ] システムの CRUD ができる（API・UI とも）
- [ ] セッション設定がシステムを参照する形になる
- [ ] **旧形式のファイルがそのまま読める**（自動移行。実データで確認）
- [ ] 5250 セッションが従来どおり接続できる（実機）
- [ ] プリンターセッションが従来どおり接続できる（実機）
- [ ] 一覧・SQL がシステム指定だけで動く（実機）
- [ ] MCP の `open_session` / `list_connections` が追従する
- [ ] パスワードが 1 箇所で管理される（システムを直せば全機能に効く）
- [ ] 既存テストが緑（移行で壊れない）

## 未確定事項

- 旧形式から作るシステム名の付け方（設定名をそのまま使うか、host から作るか）
- 同じ host・同じユーザーの設定が複数あるとき、システムを 1 つにまとめるか分けるか
- MCP の `profile` / `connection` 引数をどう変えるか（互換性）
- UI の画面構成（システム一覧とセッション一覧を分けるか、1 画面か）
