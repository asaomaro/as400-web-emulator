# 調査: IFS ファイルの読み書き

## 判明した事実

### F1: サーバー ID 0xE002、認証は既存経路

既定ポート 8473 / TLS 9473。認証は `0x7001`→`0x7002` で、
**サーバー ID を渡すだけ**で既存 `server-connect.ts` が使える。

### F2: 操作と要求 ID

| 操作 | 要求 ID | テンプレート長 |
|---|---|---|
| 交換属性 | 0x0016 | 10 |
| OPEN | 0x0002 | **36** |
| READ | 0x0003 | **22** |
| WRITE | 0x0004 | **18** |
| CLOSE | 0x0009 | 6 |
| DELETE | 0x000C | **8** |

**テンプレート長を誤ると全体の配置がずれ、`rc=17` で失敗する。**
コンパイルでは気づけないため、原典から正確に写す必要がある。

### F3【重要】応答は ReqRep ID で意味が変わる

```
0x8001（エラー）: 連鎖指示(2) の【次】が戻りコード（オフセット 22）
0x8002（OPEN）  : 連鎖指示(2) の次がファイルハンドル（4 バイト）
0x8003（READ）  : 連鎖指示(2) CCSID(2) データ長(4) の後にデータ
```

**オフセット 20 を戻りコードとして読むと常に 0 に見え、失敗を成功と誤認する。**
実際にこれで「書き込み成功・読み取り 0 バイト」という不可解な状態になった。

### F4: 戻りコードの意味（原典 FileConstants）

2=File not found / 3=Path not found / 6=Invalid handle /
13=Access denied / 18=No more files

### F5: 実機で往復に成功

```
書き込み 37 バイト → 読み取り 37 バイト → 内容一致 → 削除成功
```

## spec への申し送り

- テンプレート長は原典から正確に写す（推測しない）
- 応答は ReqRep ID で分岐させる。オフセット 20 を戻りコードにしない
- 戻りコードは意味の分かる文言にする
