# 要件: ジョブ・オブジェクト・ユーザー一覧の Web UI

## 背景 / 課題

ホストサーバー経由の一覧取得（`listJobs` / `listObjects` / `listUsers`）を core に実装したが、
**利用する手段が無い**（CLI と自動テストのみ）。Web UI から使えるようにする。

## 目的 / ゴール

Web UI からジョブ・オブジェクト・ユーザーを一覧し、**操作もできる**状態にする。

## スコープ

### 対象

- **新規ペイン**として実装する（管理画面と同じ `list:*` タブ方式）
- 3 種類の一覧: ジョブ / オブジェクト / ユーザー
- 絞り込み（ユーザー・ライブラリ・種別など）
- **操作**（下記「操作の範囲」）
- サーバー側の API とストア

### 対象外

- スプール一覧の UI（性質が違うため別途相談する）
- 一覧の自動更新（手動で再取得）
- 並べ替え・ページング（件数上限で足りる）

## 誰が使えるか

**接続を持つユーザーなら誰でも**（管理者に限定しない）。

一覧は接続情報を要するが、**見える範囲は IBM i の権限が決める**。
一般ユーザーは自分のジョブとアクセスできるオブジェクトしか見えない
（実機で確認済み: MARO は自分の 1 件のみ）。
アプリ側で追加の制限を掛ける必要はない。

## 操作の範囲

**参照だけでなく操作も可能**にする。ただし**破壊的な操作は確認を挟む**。

| 対象 | 操作 |
|---|---|
| ジョブ | 終了（`ENDJOB`）・保留（`HLDJOB`）・解放（`RLSJOB`） |
| オブジェクト | 削除（`DLTOBJ`） |
| ユーザー | **操作しない**（権限管理はアプリの領分ではない） |

操作は既存の `CommandConnection.run()` で CL コマンドを実行する。
**失敗は IBM i のメッセージ ID とともに表示**する（原因が分かる形で返す仕組みが既にある）。

## 機能要件

- 接続を選んで一覧を取得できる
- 絞り込みを指定できる（ジョブ: ユーザー・種別 / オブジェクト: ライブラリ・種別 / ユーザー: 選択条件）
- 一覧を表形式で表示する
- 行を選んで操作を実行できる（破壊的な操作は確認ダイアログ）
- 操作の結果（成功・失敗とメッセージ）を表示する
- 取得中・エラーの状態が分かる

## 非機能要件 / 制約

- 既存のペイン機構（`admin:*`）と同じ形にする。**新しい概念を増やさない**
- 既存のセッション（5250・プリンター）に影響を与えない
- 認可はサーバー側で担保する（既存方針）
- 接続情報はサーバーに留め、ブラウザへ資格情報を渡さない

## 完了条件 (受け入れ基準)

- [ ] `list:jobs` / `list:objects` / `list:users` がペインとして開ける
- [ ] それぞれ一覧が表示される（実機の接続で確認）
- [ ] 絞り込みが効く
- [ ] ジョブの保留・解放が動く（実機で確認。**元に戻せる操作で検証する**）
- [ ] オブジェクト削除が確認ダイアログを経て動く（**使い捨てオブジェクトで検証**）
- [ ] 操作失敗時にメッセージ ID が表示される
- [ ] コンポーネントテストがある
- [ ] 既存テストが緑のまま
