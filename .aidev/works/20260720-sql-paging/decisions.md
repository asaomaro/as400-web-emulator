# 決定記録

## D1. 画面の経路にだけ状態を持たせる（MCP は単発完結のまま）★

**実測が決め手**。150 件を 50 件ずつ 3 ページ取る場合:

| 方式 | 所要 |
|---|---|
| 単発完結 × OFFSET（現行の接続モデルのまま） | **19,246 ms** |
| 接続を保持 × OFFSET で取り直し | 3,507 ms |
| **カーソルを保持して読み進め** | **2,355 ms** |

1 スクロールあたり 6 秒では遅延ローディングとして成立しない。
**「単発完結を崩さずにページングする」選択肢は無かった。**

`20260719-hostserver-mcp-tools` の D2（単発完結・プールしない）は**覆していない**——
あの根拠は「MCP は対話的に単発で叩かれる用途」であり、ページングは該当しない。
**MCP の `host_sql` は変えていない**（実機で `resultSetId` を返さないことを確認済み）。

## D2. 状態を持つ以上、歯止めを実装とテストで持つ

| 項目 | 値 |
|---|---|
| アイドル | 60 秒で閉じる |
| 1 利用者あたり | 4 つまで（超えたら**最も古いものを閉じる**） |
| プロセス終了時 | `closeAll` で全部閉じる（SIGINT / SIGTERM） |
| 所有 | 既存の `assertOwner` に乗せる |

**掴んだ接続が残ることが最悪の失敗**なので、時刻を注入可能にしてテストで固定した。

## D3. 「続きがあるか」は 1 件多く読んで判断する

`n+1` 件読み、n 件返して 1 件を持ち越す。

**SQLCODE 100 を待つ形にしない**——最後がちょうど n 件のときに余計な往復が出るうえ、
#99 で踏んだ「ちょうど倍数」の経路に近くなる。テストでも固定した。

## D4. OFFSET 方式を採らなかった

技術的には成立した（任意の SQL を副問合せに包める・順序も安定していた）。
だが**接続を保持しないと 6 秒/ページ**で、保持するならカーソルも保持できる。
さらに OFFSET はクエリ全体を毎回実行し直すので、重い結合では効く。

> 順序の安定は PUB400 の SYSTABLES で確かめただけで、
> **ORDER BY の無い SQL の行順は本来 DB2 が保証しない**。
> 採らなかった判断の副次的な理由でもある。

## D5. CSV は「表示中の件数」をボタンに出す

ページングを入れると「どこまで読んだか」で CSV の内容が変わる。
**黙って一部だけ出さない**ため、ボタンに「（表示中の N 件）」と出す。

「全件 CSV」は別の機能（サーバー側で全件を回して流す）であり、本作業では扱わない。

## D6. `openQuery` を core に足した

`query()` は全件読んでカーソルを閉じ、`stream()` は列定義を返さない。
**その中間**——列定義を返し、カーソルの寿命を呼び出し側に委ねる入口——が要った。

⚠ 呼び出し側が「最後まで回すか `return()` する」責任を負う。
JSDoc に明記し、`ResultSetStore` が必ず閉じるようにしてある。
**型では強制できていない**ので、別の場所から使うときは同じ責任を負う。
