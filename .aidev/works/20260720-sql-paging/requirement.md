# 要件: SQL 結果の遅延ローディング（ページング）

## 背景 / 課題

利用者の要望:

> 画面上で最大件数の指定がありますが、**1 度に取得する件数**の扱いに変更し、
> End・PageDown・スクロールで遅延ローディングするようにして。
> 最大件数は手入力ではなく、**50, 200, 500, 1000 など一般的な指定値でドロップダウン**で
> 選べるようにしてください。

現状の「最大行数」は**全体の上限**で、超えると `truncated` として打ち切る。
つまり **1001 行目以降は永久に見られない**。実表を眺める用途には向かない。

## 目的 / ゴール

**表の続きを、必要になった時点で読み足せる**ようにする。
ACS の Run SQL Scripts のように、スクロールしていけば先が見える状態にする。

## スコープ

### 対象

- 「最大行数」を**「1 度に取得する件数」**に読み替える
- ドロップダウン化（50 / 200 / 500 / 1000）
- **End キー・PageDown・スクロール**で次のブロックを読み足す
- 「これ以上ない」ことが分かる表示

### 対象外

- 前方向へのページング（先頭に戻る等）。まず追記方向だけ
- 並べ替え・絞り込みの UI
- CSV へのページング（**要検討 3**）

## 要検討（spec で決める）★

### 1. 続きをどう取るか — 接続モデルとの衝突

**これは前作業（LOB ロケーター）と同じ根の問題である。**

私たちの接続は**単発完結**で、`/api/host/sql` は 1 リクエストごとに
`DbConnection.connect()` → `finally { close() }` する。カーソルも `query()` の中で閉じている。

つまり **「続きを取る」ための状態がどこにも残っていない。**

候補:

- (a) **サーバー側にカーソルを保持する**（結果セットのセッション化）。
      自然だが、単発完結という既存の設計判断（`20260719-hostserver-mcp-tools` D2）に反する。
      アイドルタイムアウト・接続数の上限・per-user の分離が要る
- (b) **SQL を書き換えて毎回取り直す**（`OFFSET n ROWS FETCH NEXT m ROWS ONLY`）。
      ステートレスのままだが、**利用者の SQL を包む**ことになる。
      DB2 for i が `OFFSET` を受けるか、任意の SQL を副問合せに包めるかは**未確認**。
      毎回クエリ全体を実行し直すので重く、**結果が途中で変わりうる**
- (c) クライアントに全件持たせる（**採らない**——上限を無くす意味が無い）

**(a) と (b) は性質がまったく違う。** 実機で (b) が成立するかを確かめてから決める。

### 2. 何をもって「続きがある」とするか

現状の `truncated` は「上限で切った」意味。ページングでは
「**次のブロックがあるか**」を返す必要がある。判定方法を決める
（1 件多く取ってみる／SQLCODE 100 を見る／件数で判断）。

### 3. CSV の扱い

「表示は 50 件ずつ、CSV は全部」なのか「表示中のものだけ」なのか。
**現状は表示中のものだけ**を出している。ページングを入れると
「どこまで読んだか」で内容が変わるので、**利用者に分かる形**にする必要がある。

### 4. スクロール検知の作法

表がペイン内でスクロールする構造なので、どの要素の scroll を見るかを決める。
**キーボード（End / PageDown）とスクロールで同じ経路**を通るようにする。

## 受け入れ基準

- [ ] 件数がドロップダウン（50 / 200 / 500 / 1000）になっている
- [ ] **End / PageDown / スクロール**で次のブロックが読み足される
- [ ] 「これ以上ない」ことが画面で分かる
- [ ] 読み足し中に二重で要求しない
- [ ] **CSV の内容が何を指すか利用者に分かる**
- [ ] 実ブラウザで確認（キーボード・スクロールの両方）
