# テスト結果

## 自動テスト

```
tsc -b / eslint . / web-ui ビルド  通過
vitest core   550（変更なし）
vitest server 284（新規 13）
vitest web-ui 394（既存 2 件を書き換え・新規 1）
```

新規は `ResultSetStore`。**このアプリで唯一「接続を掴み続ける」場所**なので、
歯止めを固定した:

- 指定件数ずつ返し、続きの有無を示す
- **件数がちょうど割り切れても余計なページを作らない**（#99 の「ちょうど倍数」の轍）
- アイドルで閉じ、**接続も閉じる**／使うたびに時計が戻る
- 1 利用者の上限を超えたら**最も古いものを閉じる**／上限は利用者ごと
- **`closeAll` で全部閉じる**（プロセス終了時に接続を残さない）
- `close` はジェネレータも閉じる（カーソルを残さない）
- 所有: 他人のものに触れない／本人・admin・認証オフは取れる

時刻を注入可能にしてあり、テストは実時間に依存しない。

## 実機検証（PUB400）

### 1. 1,250 件を 25 ページ ★

```
1ページ目: 50 件 hasMore=true 列=2
合計: 1250 件 / 25 ページ / ユニーク: 1250 / 続き: true
```

**重複ゼロ**。core のブロッキング係数 100 の境界を 12 回跨いでいる。
`stream` はこれまで実利用が無く「未検証」と backlog にあったが、**これで実証できた**。

### 2. 期限切れと読み切り

| 操作 | 結果 |
|---|---|
| 存在しない ID に `/next` | **404**「この結果セットは期限切れです。もう一度実行してください」 |
| 3 件のクエリ（`hasMore=false`） | 1 ページで完了・結果セットは閉じられる |
| 読み切り後の `/next` | 200・0 件・`hasMore=false`（例外にしない） |

### 3. 既存経路が変わっていないこと

| 経路 | 結果 |
|---|---|
| MCP `host_sql`（`maxRows`） | `rowCount=2` `truncated=true`・**`resultSetId` を返さない** |
| HTTP `maxRows` 指定 | 同上（互換維持） |

**MCP は単発完結のまま。** 状態を持つのは画面の経路だけ（spec D1）。

### 4. 実ブラウザ

| 確認 | 結果 |
|---|---|
| ドロップダウン | **50 件 / 200 件 / 500 件 / 1000 件** |
| 初回 | 50 件 |
| **End キー** | 100 件（+50） |
| **スクロール** | 150 件（+50） |
| CSV ボタン | 「CSV をダウンロード（表示中の 150 件）」 |

### 実ブラウザで見つけた欠陥（修正済み）

**最初の実装で列見出しが画面外に押し出され、表もはみ出していた。**
`.sql-pane` 自体が `overflow: auto` のところに、表領域にも `max-height` を
付けたため**二重スクロール**になっていた。

ペインを縦フレックスにして**表領域だけがスクロール**するようにし、
列見出しを `position: sticky` で残るようにした。
**自動テストでは絶対に気づけない種類の不具合**で、スクリーンショットを見て分かった。

## 未検証の範囲（引き継ぎ）

- **アイドル 60 秒の妥当性**。実際に 60 秒放置してからの読み足しは試していない
  （単体テストでは時刻を注入して確認済み）
- 複数タブ／複数利用者で同時に結果セットを開いたときの挙動
- 認証オンでの所有チェック（単体テストのみ。実機の認証オン環境では未確認）
- 前方向へのページング（対象外）
- 「全件 CSV」（対象外。**表示中の件数をボタンに出す**ことで誤解を避けている）
