# 調査: SQL 結果のページング

requirement の要検討 1（(a) カーソル保持 / (b) OFFSET で取り直し）を、
**実機で測って**決める。

## F1. OFFSET は使える。任意の SQL も包める

| 試した形 | 結果 |
|---|---|
| `… ORDER BY … OFFSET 5 ROWS FETCH NEXT 3 ROWS ONLY` | ✅ |
| `OFFSET 0 ROWS` | ✅ |
| `SELECT * FROM (<利用者の SQL>) X OFFSET 5 ROWS FETCH NEXT 3 ROWS ONLY` | ✅ |
| **元の SQL に `FETCH FIRST` があっても包める** | ✅ |
| ORDER BY 無しでも包める | ✅ |

→ **(b) は技術的に成立する。**

## F2. 順序は安定していた（(b) の最大の懸念）

ORDER BY 無しの SQL を包んで確かめた。

| 検証 | 結果 |
|---|---|
| 一括 30 件 と 10 件 × 3 ページが一致するか | **一致** |
| 分割時の重複 | **0 件** |
| 同じ OFFSET を 2 回 | **同じ** |

→ 少なくとも PUB400 の SYSTABLES では安定している。
   **ただし「安定する保証」ではない**——ORDER BY の無い SQL の行順は
   本来 DB2 が保証しないので、他の表・並行更新下で崩れる可能性は残る。

## F3. コストの実測 ★これが決め手

150 件を 50 件ずつ 3 ページ取る場合:

| 方式 | 所要 |
|---|---|
| **(b) 単発完結 × OFFSET**（現行の接続モデルのまま） | **19,246 ms** |
| (b') 接続を保持 × OFFSET で取り直し | 3,507 ms |
| **(a) カーソルを保持して読み進め** | **2,355 ms** |

**(b) は 1 ページごとに接続を張り直すため 19 秒**かかる。
遅延ローディングは「スクロールしたら続きが出る」体験を狙うものなので、
**1 スクロールあたり 6 秒は要件を満たさない**。

(b') と (a) の差は小さいが、(b') は**クエリ全体を毎回実行し直す**。
今回は小さな表なので差が出なかっただけで、重い結合を含む SQL では
ページごとに全体を再計算することになる。

## F4. つまり接続を保持する必要がある

(b) を採るなら**接続の保持が前提**になる（19 秒は使えない）。
そして接続を保持するなら、**カーソルも保持できる**（(a)）。

→ **「単発完結を崩さずにページングする」選択肢は無い。**
   要検討 1 は「(a) か (b) か」ではなく、
   **「状態を持つことを受け入れるか、ページングを諦めるか」**だった。

## F5. 既存の設計判断との関係

`20260719-hostserver-mcp-tools` の D2 で「接続は単発完結。プールしない」と決めた。
根拠は「MCP ツールは対話的に単発で叩かれる用途」「認可上の難しさを先に払わない」。

**この判断を覆すわけではない。** MCP ツールは単発のままでよい。
ページングが要るのは**画面（Web UI）**だけで、そこは per-user のセッションを
既に持っている（Cookie 認証・`assertOwner`）。

→ **Web UI 用の結果セットセッション**を、既存のセッション管理の枠組みの中に置く。
   MCP の `host_sql` は現状どおり単発完結のまま変えない。
