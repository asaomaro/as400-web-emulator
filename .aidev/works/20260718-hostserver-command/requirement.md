# 要件: コマンドサーバー経由の CL コマンド実行とプログラム呼び出し

## 背景 / 課題

backlog `hostserver.md` の「コマンド実行 / プログラム呼び出し」。
**スプール一覧・MSGW 検出・IFS・各種一覧の 4 件すべての土台**であり、これ無しに先へ進めない。

現在 MCP の `run_steps` 等は 5250 の画面を操作して CL を実行している。
画面レイアウトに依存するため壊れやすく、応答も画面テキストの切り出しになる。
コマンドサーバー経由なら**構造化された応答**が返り、レイアウト変化に影響されない。

前段（`20260718-acs-data-transfer` / `20260718-hostserver-sql`）で確立済みの資産:

- 認証（`0x7001`→`0x7002`）は `server-connect.ts` に**サーバー共通**として切り出し済み
- 20 バイトヘッダー＋LL/CP の読み書き（`hostserver/datastream.ts`）
- ソケットとフレーム分割（`transport/host-connection.ts`）
- CCSID 37 / 273 の EBCDIC 変換

## 目的 / ゴール

**コマンドサーバー（8475 / TLS 9475）に接続し、CL コマンドを実行して結果を構造化して受け取れる**
状態にする。加えて、後続 4 件が必要とする**プログラム呼び出し**の足場を作る。

## スコープ

### 対象

- コマンドサーバーへの接続と認証（既存の `server-connect.ts` を再利用）
- **CL コマンドの実行**と、成否・メッセージの取得
  - メッセージ ID（例 `CPF9801`）・重大度・テキストを構造化して返す
- **プログラム呼び出し**（QSYS の API を呼ぶ足場）
  - 入力/出力/入出力パラメータの受け渡し
  - 後続の `ListSpooledFiles` 等がこの上に載る
- `packages/core` のライブラリとして実装し、CLI で検証できること

### 対象外（この作業ではやらない）

- 具体的な API のラッパ（スプール一覧・MSGW 等）は**後続の別作業**
- MCP ツールとしての公開、Web UI からの操作
- 対話型コマンド（画面を出すもの）。コマンドサーバーは非対話のみ扱う
- サービスプログラムの呼び出し（プロシージャポインタ経由）

## 機能要件

- ホスト・ユーザー・パスワード・TLS 可否を与えてコマンドサーバーに接続できる
- 任意の CL コマンド文字列を実行し、**成功/失敗を判別**できる
- 失敗時にメッセージ ID・重大度・テキストを取得できる（原因が分かる形）
- 成功時でも情報メッセージ（例 `CPC2191`）を取得できる
- プログラムを呼び出し、パラメータを渡して出力を受け取れる
- 通信内容をトレースできる。**資格情報を平文で出さない**

## 非機能要件 / 制約

- 既存の TN5250 / signon / SQL 実装に影響を与えない
- `packages/core` のピュアロジック層は **Node API 非依存**を維持する
  （I/O は `transport/` に隔離）
- **ピュア層で Node のグローバル（`Buffer` 等）も使わない**
  （前段のレビューで lint をすり抜けた実績があるため明示する）
- JTOpen は参照のみ。ソースをリポジトリに取り込まない。参照コメントを残す
- 実機（PUB400）での検証を伴う。パスワードは環境変数経由でのみ扱う

## 完了条件 (受け入れ基準)

- [ ] コマンドサーバーに接続・認証できる（TLS / 平文の双方。実機で確認）
- [ ] 副作用のない CL コマンドが成功する（例 `CHGJOB CCSID(273)` や `DSPLIB` 系ではなく、
      安全な `SNDPGMMSG` 等）。実機で確認
- [ ] **存在しないコマンドが失敗し、メッセージ ID が取れる**（例 `CPD0030`）。実機で確認
- [ ] 権限不足など実行時エラーのメッセージが取れる
- [ ] プログラム呼び出しが成功し、出力パラメータを受け取れる（実機で確認）
- [ ] メッセージ解析の単体テストがある（実機に依存しない固定バイト列から）
- [ ] トレース出力に資格情報が平文で現れないことをテストで担保する
- [ ] JTOpen のどのクラスに対応するかがコード上で追える

## 未確定事項 / 確認したいこと

**research 工程で事実確認する**（推測で spec を書かない。前段で要求側 CP を推測して
全滅させた反省を踏まえ、**解析側と書き出し側の両方**を読む）。

- コマンドサーバーのサーバー ID と要求 ID
- データストリームの形式（signon 系の 20 バイトヘッダーか、database のような template 付きか）
- CL コマンド文字列の送り方（CCSID は何か。37 か、ジョブの CCSID か）
- メッセージの返り方（メッセージ ID・重大度・テキストの位置）
- プログラム呼び出しのパラメータ形式（入力/出力/入出力の区別、長さの指定）
- **実機で安全に試せるコマンドは何か**（副作用が無く、成功と失敗の両方を作れるもの）
- PUB400 でコマンドサーバーが使えるか（権限・ポートの開放状況）
