# 調査: コマンドサーバー経由の CL 実行とプログラム呼び出し

## 調査の問い

- Q1: サーバー ID・要求 ID・データストリーム形式は？
- Q2: CL コマンド文字列の送り方（CCSID）は？
- Q3: メッセージ（ID・重大度・テキスト）の返り方は？
- Q4: 成否はどう判定するか？
- Q5: プログラム呼び出しのパラメータ形式は？
- Q6: PUB400 でコマンドサーバーが使えるか？安全に試せるコマンドは？

## 判明した事実

### F1: サーバー ID は 0xE008。認証は既存実装をそのまま再利用できる

出典: `CommandConnection.getConnection()` / `sendRunCommandRequest()`

- 既定ポート **8475 / TLS 9475**
- 認証は `HostServerConnection.connect()` ＝ **`0x7001`→`0x7002`**。
  前段で `server-connect.ts` に**サーバー共通**として切り出してあるため、
  **サーバー ID を渡すだけで動く**（実機で確認済み。F6）

### F2: データストリームは signon 系と同じ 20 バイトヘッダー（database のような 40 バイトではない）

要求 ID:

| 値 | 意味 | template 長 |
|---|---|---|
| `0x1001` | 交換属性 | 14 |
| `0x1002` | コマンド実行 | 1（メッセージオプション） |
| `0x1003` | プログラム呼び出し | 23 |
| `0x1004` | （接続時のやり取り） | — |

応答は要求 ID の最上位ビットを立てた値（`0x1001` → `0x8001`）。

### F3: コマンド文字列はデータストリームレベルで書式が変わる

出典: `sendRunCommandRequest()`

```
レベル >= 10 : CP 0x1104、CCSID 1200（UTF-16BE）を 4 バイトで前置、messageOption = 4
レベル <  10 : CP 0x1101、CCSID 37 の EBCDIC、messageOption = 0 または 2
```

**PUB400 はレベル 11**（F6）なので UTF-16 経路。ただしレベルは交換属性で得るため、
**接続時に必ず交換属性を送る必要がある**（送らずにコマンドを投げると応答が壊れる。実機で確認）。

### F4: 成否は 2 バイトの戻りコード。0 が成功、0x0400 がメッセージ付き失敗

出典: `runCommand()`

```
オフセット 20（2 バイト）: 戻りコード。0 = 成功 / 0x0400 = 失敗（メッセージあり）
オフセット 22（2 バイト）: メッセージ件数
オフセット 24〜          : メッセージ列（LL/CP 形式）
```

`rc` が `0` でも `0x0400` でもない値は例外扱い。

### F5: メッセージは 2 形式ある。新しい方（CP 0x1106）は全項目が長さ前置

出典: `getMessages()`

**CP 0x1106（レベルが新しいサーバー）**——LL(4)＋CP(2) の後:

```
テキスト CCSID (4)
置換文字列 CCSID (4)
重大度 (2)              ← 符号付き
メッセージ種別長 (4) ＋ 種別 (2) ＋ 残り
メッセージ ID 長 (4) ＋ ID（EBCDIC）
メッセージファイル名 長 (4) ＋ 名前
メッセージファイルライブラリ 長 (4) ＋ 名前
メッセージテキスト 長 (4) ＋ テキスト（EBCDIC）
置換データ 長 (4) ＋ データ
ヘルプ 長 (4) ＋ テキスト
```

**CP 0x1102（古い形式）**——固定長:

```
ID 7 バイト / 種別 2 / 重大度 2 / ファイル名 10 / ライブラリ 10 /
置換データ長 2 / テキスト長 2 / 置換データ / テキスト
```

ID・テキストは **CCSID 37 の EBCDIC**。

### F6: PUB400 で動作を実測（この作業の最大の収穫）

ポートは開いており、ポートマッパーも応答する:

```
as-rmtcmd -> 8475    as-rmtcmd-s -> 9475
```

交換属性の応答（36 バイト・template 16）:

```
rc=0  CCSID=273  NLV="2924"  版数=V7R5M0  データストリームレベル=11
```

**コマンド実行を実際に確認した**（`scratchpad/cmdprobe.mjs`）:

```
cmd="CHGJOB CCSID(273)"          rc=0x0    messages=0
cmd="CHGLIBL LIBL(QGPL QTEMP)"   rc=0x0    messages=1  CPC2101 sev=0 "Library list changed."
cmd="ADDLIBLE LIB(MARO1)"        rc=0x0    messages=1  CPC2196 sev=0 "Library MARO1 added to library list."
cmd="RMVLIBLE LIB(MARO1)"        rc=0x0    messages=1  CPC2197 sev=0 "Library MARO1 removed from library list."
cmd="NOSUCHCMD"                  rc=0x400  messages=2  CPD0030 sev=30 "Command NOSUCHCMD in library *LIBL not found."
                                                       CPF0001 sev=30 "Error found on NOSUCHCMD command."
cmd="DSPLIB LIB(NOSUCHLIB)"      rc=0x400  messages=1  CPF2110 sev=40 "Library NOSUCHLIB not found."
```

- **成功時も情報メッセージ（重大度 0）が返る**。メッセージの有無で成否を判断してはいけない
- 重大度は 0（情報）/ 30（エラー）/ 40（重大）が実際に観測できた
- **`SNDPGMMSG` は使えない**——`CPD0031 "Command SNDPGMMSG not allowed in this setting."`。
  コマンドサーバーは非対話環境なので、対話前提のコマンドは弾かれる

**検証に使える安全なコマンド**（副作用が自ジョブに閉じ、元に戻せる）:

| コマンド | 用途 |
|---|---|
| `CHGJOB CCSID(273)` | 成功・メッセージ 0 件 |
| `ADDLIBLE LIB(MARO1)` → `RMVLIBLE LIB(MARO1)` | 成功・情報メッセージあり。対で元に戻る |
| `NOSUCHCMD` | 失敗・`CPD0030` |
| `DSPLIB LIB(NOSUCHLIB)` | 失敗・`CPF2110` 重大度 40 |

### F7: プログラム呼び出しの構造

出典: `sendCallProgramRequest()`

```
要求 ID 0x1003、template 長 23
  プログラム名   10 バイト（CCSID 37・空白詰め）
  ライブラリ名   10 バイト（同上）
  メッセージオプション 1 バイト
  パラメータ数   2 バイト
パラメータごと:
  LL(4) = 12 + 入力長
  CP(2) = 0x1103
  最大長(4)  ← 入力長と出力長の大きい方
  種別(2)    ← 入力 / 出力 / 入出力 / NULL
  入力データ
```

## 影響範囲

- 新規モジュールとして追加でき、既存には触れない
- **前段の資産がそのまま効く**:
  - `server-connect.ts`（`0x7001`/`0x7002`）— **サーバー ID を変えるだけ**
  - `hostserver/datastream.ts`（20 バイトヘッダー・LL/CP）
  - `transport/host-connection.ts`
  - CCSID 37 の EBCDIC 変換
- 後続 4 件（スプール・MSGW・IFS・各種一覧）がこの上に載る

## 実現性 / リスク

**実現性は実証済み。** research の時点で成功・失敗の両方を実機で確認できた。
前段 2 件で土台が固まっているため、新規に書くのは
「交換属性 → コマンド実行 → メッセージ解析」と「プログラム呼び出し」だけ。

### リスク 1: 交換属性を送り忘れると壊れる

データストリームレベルが分からないとコマンドの書式が決められない。
実際、交換属性を送らずにコマンドを投げたら応答が解釈不能になった。
**接続手順に組み込んで、呼び出し側が忘れられないようにする**。

### リスク 2: 成功でもメッセージが返る

`CPC2101` のような情報メッセージは成功時にも返る。
**メッセージの有無で成否を判断する実装にしない**（戻りコードで判断する）。

### リスク 3: プログラム呼び出しの実機検証先

安全に呼べる QSYS API を選ぶ必要がある。読み取り専用のもの
（例 `QUSRJOBI` ジョブ情報取得）が候補。spec で決める。

## spec への申し送り

- **接続時に交換属性を必ず送る**（データストリームレベルの取得を手順に組み込む）
- コマンド書式はレベルで分岐。**PUB400 は 11 だが、10 未満の経路も実装するか spec で決める**
  （前段の DES 経路のように「未対応」と明示する手もある）
- 成否は**戻りコードで判断**。メッセージの有無ではない
- メッセージは CP 0x1106 / 0x1102 の 2 形式。**両方を実装するか**決める
- 重大度を構造化して返す（0=情報 / 30=エラー / 40=重大 が実測できている）
- 検証は F6 の安全なコマンド 4 種を使う
- 前段の非機能要件を踏襲: ピュア層は Node API 非依存、**グローバルも使わない**
