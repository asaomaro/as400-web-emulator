# 決定記録

## D1. 任意 CL の実行を MCP に許可する

`host_command` は任意の CL 文字列を受け取る。

**根拠**（自分で決めた規則を理由にしない）:

1. **新たな権限を増やさない。** MCP には既に `run_steps` があり、5250 のコマンド行に任意の CL を
   打ち込める（`get_job_info` は実際に DSPJOB をそう実行している）。ホストサーバー経由を塞いでも
   同じことが 5250 経由で可能なまま残る——塞ぐのは security theater になる。
2. **権限境界は IBM i が持つ。** `host-lists.ts:4-6` が既に採っている原則——
   「見える範囲は IBM i の権限が決めるため、アプリ側で追加の制限は掛けない」。
   資格情報の持ち主にできること以上のことはできない。
3. `/api/host/action` が任意 CL を拒むのと矛盾しない。**対象読者が違う**——
   あちらはブラウザ（一般ユーザーを含む）向けで UI のボタンに対応する操作だけを受ける。
   MCP は API トークンを持つ自動化向け。

**残るリスク**: 破壊的 CL（`DLTLIB` 等）が実行可能。ただしこれは 1・2 より**既に成立している状態**で、
本作業で新設される危険ではない。全ツールを `withAudit` に通して証跡を残す。

## D2. 接続は単発完結（プールしない）

ツール呼び出しごとに接続を張り、`finally` で閉じる。

**根拠**: `host-lists.ts` の既存実装（リクエスト単位で開閉）と、`/mcp` の
「ステートレス（接続はリクエスト毎に管理）」宣言（`app.ts:120`）に一致する。

**実測による裏づけ（test 工程）**: PUB400 で 4〜7 秒/呼び出し。1 操作あたり signon＋目的サーバーの
2 接続が張られるためコストは接続確立が支配的で、処理量に比例していない（1 行の SELECT も
5 件の一覧もほぼ同じ時間）。**遅さの主因は設計ではなく回線**（インターネット越しの公開ホスト＋TLS）。

対話的に単発で叩かれる MCP の用途では許容範囲と判断し、プールを導入しない。
プールは「いつ閉じるか」「ユーザーをまたいで再利用しないか」という認可上の難しさを持ち込む——
**実測が許容範囲である以上、複雑さを先に払わない。**

> LAN 内 IBM i なら大幅に短くなると見込まれるが、**これは未検証の見込みであり事実として断定しない**。
> backlog に実測項目として残す。

## D3. 破壊的操作をツール表面に出さない

`deleteFile`（IFS 削除）を専用ツールにしない。

**根拠**: D1 で任意 CL を許すため `RMVLNK` で実行できる（実機で確認済み——`CPCA089 Link removed.`）。
専用ツールを足すと「AI が誤って選びやすい破壊的ボタン」を増やすだけで、能力は増えない。
**能力を増やさない表面は足さない。**

## D4. MSGW は公開しない

core の `retrieveMessage` / `answerMessage` は実装済みだが MCP に出さない。

**根拠**: core のコメント（`netprint-connection.ts:264,319`）に「⚠ 実際の MSGW に対しては未検証」と
明記されている。さらに `answerMessage` は応答文字列だけ可変長で送っており、MSGREPLY が固定長を
要求する場合に隣の値を巻き込む恐れがある（`20260718-hostserver-msgw` review の指摘。未検証のため
実装は変えていない）。**未検証のものを AI から叩けるようにしない。**

backlog の「MSGW が実在する状態での検証」が片付いてから別作業で公開する。

## D5. ホストサーバー系ツールを別ファイルに置く

`mcp-tools.ts` に足さず `host-server-tools.ts` を新設し、登録の合流点を `mcp-server.ts` に置いた。

**根拠**: `mcp-tools.ts` は既に 800 行超で 5250 の関心事で満ちている。経路が違うものを同居させると
「このツールはどちらの経路か」がファイルからは分からなくなる。
また `host-server-tools.ts` が `mcp-tools.ts` から `ToolDeps` / `errorResult` を import するため、
逆向きに登録を呼ぶと**循環参照**になる。合流点を `mcp-server.ts` に置いてこれを避けた。

## D6. `tls` を boolean にキャストしない（既存実装の修正）

移設元の `host-lists.ts:154` は `opts.tls as boolean` としていた。
`ConnectOptions.tls` は `boolean | { rejectUnauthorized?, ca? }` で、core の `HostTlsOptions` と
同じ形をしている。実行時はオブジェクトがそのまま渡って動いていたが、**型が実態と食い違い、
証明書検証の設定を落としているように読める**。`host-connect.ts` では素通しにし、回帰テストで固定した。
