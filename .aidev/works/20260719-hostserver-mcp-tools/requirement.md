# 要件: ホストサーバー機能の MCP ツール公開

## 背景 / 課題

backlog `hostserver.md` の「MCP ツールとして公開」。

2026-07-18 に `packages/core` へホストサーバー機能を一通り実装した（SQL 実行 / CL 実行・プログラム
呼び出し / スプール pull / MSGW / IFS / ジョブ・オブジェクト・ユーザー一覧）。
しかし **外部に露出しているのは一覧系だけ**である。調査で確認した現状:

| 機能 | core 実装 | 唯一の消費者 |
|---|---|---|
| SQL 実行（`hostserver/db/`） | あり | `tools/hostserver-check/src/sql.ts` の手動 CLI のみ |
| CL 実行 / プログラム呼び出し（`hostserver/command/`） | あり | 同 CLI ／ `host-lists.ts`（列挙 4 アクション限定） |
| スプール一覧・取得（`hostserver/spool/`） | あり | **消費者なし** |
| MSGW 検出・応答 | あり | **消費者なし**（かつ実機未検証） |
| IFS 読み書き（`hostserver/ifs/`） | あり | **消費者なし** |
| ジョブ/オブジェクト/ユーザー一覧 | あり | `/api/host/list/:kind` → `HostListPane.vue` ✅ |

MCP ツールは `packages/server/src/mcp-tools.ts` に 19 個登録されているが、**すべて 5250 経由**である。
`list_spools` / `get_spool` / `get_spool_pdf` は名前が紛らわしいが実体は
プリンターセッションで受信済みの帳票（`entry.reports`）で、`listSpooledFiles` は使っていない。
`get_job_info` も DSPJOB の画面操作である。

つまり **実装した価値がまだ何にも届いていない**。これが本作業の動機。

## 目的 / ゴール

ホストサーバー経由の機能を **MCP ツールとして公開**し、AI エージェントから
5250 画面操作を介さず構造化された形で IBM i を扱えるようにする。

5250 経由に対する優位は既存 backlog に記録済み——**構造化された応答が返り、画面レイアウトの
変化に壊されない**。加えて**装置名（デバイス）を必要とせず**、セッションを開かずに単発で叩ける。

## スコープ

### 対象

core に実装済みで**実機検証も済んでいる**機能を MCP に出す。

- **SQL 実行**（`DbConnection` / `query`）— 本命。`node-jt400` と違い JVM を起こさない
- **CL コマンド実行 / プログラム呼び出し**（`CommandConnection`）
- **スプールの一覧・取得（pull 型）**（`listSpooledFiles` / `NetPrintConnection`）
  - 既存の `list_spools`（push 型・プリンターセッション由来）とは**別物**。
    名前が衝突するので、ツール名で経路が分かるようにする
- **IFS ファイルの読み書き**（`IfsConnection`）
- **ジョブ / オブジェクト / ユーザー一覧**（`listJobs` / `listObjects` / `listUsers`）
  - HTTP API はあるが MCP には無い。同じ core 関数を叩くだけなので併せて出す

### 対象外（この作業ではやらない）

- **MSGW（`retrieveMessage` / `answerMessage`）の公開**。実機未検証のまま
  （`netprint-connection.ts:264,319` に「⚠ 実際の MSGW に対しては未検証」と明記されている）。
  さらに `answerMessage` は応答文字列だけ可変長で送っており、固定長要求なら隣の値を巻き込む恐れがある。
  **未検証のものを AI から叩けるようにしない**。backlog の検証項目が片付いてから別作業で出す
- Web UI からの操作（次の作業 `20260719-hostserver-web-ui`）
- 新しいプロトコル層の実装（アップロード/DDM は別作業）

## 要検討（spec で決める。現時点で断定しない）

### 1. 接続の指定方法とライフサイクル ★最重要

5250 の MCP ツールは `open_session` → 操作 → `close_session` というセッションモデルを取る。
ホストサーバーは装置名が要らないため、同じ形にするかは自明でない。候補:

- (a) 単発完結（毎回ツール引数で `system` 参照を受け、内部で接続→実行→切断）
- (b) セッションモデル（`open_host_connection` を足し、`sessionId` を引き回す）
- (c) 接続プール（`system` 参照をキーに接続を再利用し、アイドルで閉じる）

SQL は接続確立のコストが効くうえ、`DbConnection.acquire()` に重複実行ガードがある一方、
コマンドサーバーには共有状態が無い（`20260718-hostserver-command` review の nit）。
**サーバーごとに最適が違う可能性**があるので、実測してから決める。

### 2. 任意 CL 実行を MCP に出してよいか ★セキュリティ判断

既存の HTTP API は **意図的に任意 CL を拒否**している（`host-lists.ts:157`「利用側から任意の
CL は受け取らない」）。サーバー側で `HLDJOB`/`RLSJOB`/`ENDJOB`/`DLTOBJ` を組み立てる設計。

MCP に `run_command(cl)` を出すことは、この方針と**真っ向から矛盾する**。
ただし前提が違う——HTTP API はブラウザ（一般ユーザーを含む）向け、MCP は API トークンを持つ
自動化向けで、既に `run_steps` で 5250 経由の任意 CL 実行を許している。

矛盾を残したまま実装しない。**AGENTS.md「アカウント・権限設計」の 3 層に当てはめて
spec で決め、根拠を decisions.md に残す**こと。「MCP だから何でも許す」は根拠にならない
（それは自分で決めた規則を理由にする循環）。

### 3. 秘密の扱い

ホストサーバー接続には資格情報が要る。AGENTS.md より:
- 平文パスワードをツール引数で受けるのは**避ける**。既存の `signon` MCP ツールの扱いに合わせる
- プロファイル（`srv:` / `own:`）の解決は `ConfigResolver` に集約、`assertProfileAccess` を通す
- ログ・応答に秘密を出さない

### 4. 応答の形と量

SQL の結果セットは大きくなりうる。MCP 応答をトークン浪費で溢れさせない上限・省略の作法を決める
（既定行数、打ち切り時の明示）。`stream` はブロッキング係数を跨ぐ規模が未検証（backlog 記載）なので、
**未検証の経路に依存しない**設計にする。
