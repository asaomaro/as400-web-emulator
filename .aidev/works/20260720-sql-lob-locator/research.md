# 調査: ロケーター経由の LOB 取得

**原典を直読した**（jtopenlite の `DatabaseConnection.retrieveLOBData` /
`sendRetrieveLOBDataRequest` / `parseReply`）。

## F1. 要求（ID `0x1816`）

| 項目 | CP | 幅 |
|---|---|---|
| ロケーターのハンドル | **`0x3818`** | 4 |
| 要求サイズ | **`0x3819`** | 4 |
| 開始オフセット | **`0x381A`** | 4 |
| 圧縮指示 | `0x381B` | 1 |
| 変換指示 | `0x3805` | 1 |
| 現在長を返す指示 | `0x3821` | 1 |

ORS ビットマップは `SEND_REPLY_IMMED | DATA_FORMAT | RESULT_DATA`。

## F2. 応答

| CP | 内容 |
|---|---|
| **`0x3810`** | LOB データ長。先頭 2 バイトが幅（0 / 4 / それ以外＝上下 32 ビットで 8 バイト） |
| **`0x380F`** | 本体。`CCSID(2) + 長さ(4) + データ` |

原典は 64KB のバッファで `newLOBSegment` を繰り返し呼び、**分割して受け取る**。

## F3. 実機で確認した事実 ★ここが設計を決めた

`CLOB(2000)` に `HELLO-LOCATOR-TEST`（18 文字）を入れて確認。

### 取得できる

```
retrieveLOB 応答: 0x3812(0b) 0x3810(6b) 0x380f(24b) | template rc: 0 0
  CCSID: 273  長さ: 18
  先頭: c8c5d3d3d6 60 d3d6c3c1e3d6d9 60 e3c5e2e3  = "HELLO-LOCATOR-TEST"（EBCDIC）
```

**CCSID は 13488（UTF-16）ではなく 273（EBCDIC）だった。** 推測していたら外していた。

### ロケーターの寿命 ★★

| 条件 | 結果 |
|---|---|
| `query()` がカーソルを閉じた後、**同じ接続** | ✅ `rc=0/0`・17 バイト取得 |
| **別の接続**で同じ番号 | ❌ **`rc=2/-815`** |
| 同じ接続で**別のクエリを実行した後** | ✅ `rc=0/0`・取得できる |

→ **ロケーターは接続に紐づく。** カーソルにも、直前のクエリにも紐づかない。

## F4. この事実が設計に与える影響 ★最重要

**私たちの接続は単発完結である**（`20260719-hostserver-mcp-tools` の D2）。
MCP ツールも `/api/host/sql` も、呼び出しごとに `DbConnection.connect()` して
`finally` で閉じている。

つまり:

- 画面で結果を表示 → **接続は既に閉じている**
- 利用者が LOB セルをクリック → **新しい接続**になる → **ロケーターは無効**（`-815`）

**「クリックして展開」は、今の接続モデルでは成立しない。**

成立させるには (1) 接続を跨いで保持する（プール／セッション化）か、
(2) **同じクエリの中で本体まで取る**か、のどちらか。

requirement の要検討 1 は (a)「明示的に呼んだときだけ取る」を候補に挙げていたが、
**F3 の実測により (a) は UI からは使えない**ことが分かった。spec で決め直す。
