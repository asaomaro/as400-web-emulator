## 実機検証（PUB400・IBM i 7.5・TLS）

### 1. ロケーターの寿命 ★設計を決めた実測

`CLOB(2000)` に `HELLO-LOCATOR-TEST` を入れて確認。

| 条件 | 結果 |
|---|---|
| `query()` がカーソルを閉じた後、**同じ接続** | ✅ `rc=0/0` |
| **別の接続**で同じ番号 | ❌ **`rc=2/-815`** |
| 同じ接続で別のクエリを実行した後 | ✅ `rc=0/0` |

→ **ロケーターは接続に紐づく。** カーソルにも直前のクエリにも紐づかない。

この事実により requirement の第一候補（「明示的に呼んだときだけ取る」）が**使えない**と
判明した。私たちの接続は単発完結なので、画面表示後にクリックした時点で接続は閉じている。

**CCSID は 273（EBCDIC）だった**——UTF-16 だと推測していたら外していた。

### 2. 分割受信 ★本丸

**163,840 文字（64KB の 2.5 倍）** の CLOB を作って確認した
（`REPEAT` を使う INSERT は CL の解釈で失敗したため、`C = C || C` の倍加で作った）。

| 指定 | 取得 | 全体 | 状態 | 先頭 / 末尾 |
|---|---|---|---|---|
| 既定（`lob` 無し） | なし | — | `not-requested` | — |
| `lob: {}`（上限 64KB） | **65,536 文字** | 163,840 | `too-large` | `ABCDEFGHIJ` / `GHIJABCDEF` |
| `lob: { maxBytes: 200000 }` | **163,840 文字** | 163,840 | 完全 | `ABCDEFGHIJ` / `ABCDEFGHIJ` |

**64KB 境界を 2 回跨いで正しく結合できている**（末尾が `ABCDEFGHIJ` で閉じており、
セグメントの継ぎ目でずれていない）。

### 3. 3 つの状態が区別できる

`not-requested`（取りに行っていない）／`too-large`（上限で打ち切り）／完全取得。
**空文字で潰していない**ので、空の LOB とも区別できる。

### 4. 実ブラウザ

| チェックボックス | 表示 |
|---|---|
| OFF（既定） | `(LOB)` |
| ON | `ABCDEFGHIJ…（65,536 文字）…（以降省略）` |

応答も確認: `value 長 = 65536 / 状態 = too-large`。

> **最初のブラウザ確認は誤った結果を出した**——チェックを入れて実行した直後に
> 表を読んだが、**実行中は前の結果が残る**ため古い `(LOB)` を読んでいた。
> 応答を待ってから読むよう直した。**待ち方を間違えると「直っていない」と誤判定する。**

## 未検証の範囲（引き継ぎ）

- **BLOB（バイナリ）**。CLOB でしか試していない。CCSID の扱いが変わる可能性がある
- **DBCLOB**（純 DBCS）。`QSYS2.SYSTABLES.MQT_DEFINITION` は実データが NULL のものが多く、
  中身のある DBCLOB を用意できなかった
- ロケーターの**明示的な解放**。接続を閉じれば消えると見込んでいるが確かめていない
- 1 MB を超える LOB（API の上限で弾いている）
- 複数 LOB 列を含む結果セットでの取得（1 列でしか試していない）
