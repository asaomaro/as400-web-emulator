# 決定記録

## D1. 同じクエリの中で取り切る（requirement の第一候補は使えなかった）★

requirement は「明示的に呼んだときだけ取る」を第一候補にしていたが、
**実機の実測で否定された**。

| ロケーターの寿命 | 結果 |
|---|---|
| カーソルを閉じた後、同じ接続 | 有効 |
| **別の接続** | **無効（rcClass=2, code=-815）** |

私たちの接続は**単発完結**（`20260719-hostserver-mcp-tools` の D2）なので、
画面に結果を出した時点で接続は閉じている。利用者が LOB セルをクリックしても、
**そのロケーターは既に無効**。

→ `query()` にオプションを足し、**同じ接続の中で取り切る**形にした。

**この判断は spec の前に実機で確かめたから下せた。** 推測で spec を書いていたら
「クリックで展開」を実装してから矛盾に気づいていた。

## D2. 既定では取りに行かない

`lob` オプション未指定なら従来どおりロケーターのみ。

**根拠**: 前作業で「LOB をインライン化すると 2 行で 8.4MB」を実測している。
ロケーターを自動で辿れば**同じことが起きる**（100 行 × 2MB = 200MB）。
上限は 1 セル 64KB（原典のバッファサイズに合わせた）、API の受け付け上限は 1MB。

## D3. 3 つの状態を区別する（`null` や空文字で潰さない）

```
not-requested … 取りに行っていない（既定）
too-large     … 上限で打ち切った（先頭だけ取れている）
（なし）        … 完全に取れた
```

**空文字で埋めない**——空の LOB と「取っていない」が区別できなくなる。
画面では `(LOB)` / `中身…（以降省略）` / `中身` と出し分ける。

## D4. 文字コードはホストの申告に従う

実機の CLOB は **CCSID 273（EBCDIC）**だった。UTF-16 だと推測していたら外していた。
応答の `0x380F` が申告する CCSID でデコードし、**未知なら `Uint8Array` のまま返す**
（壊れた文字列にしない）。

## D5. 分割受信は「進まなくなったら打ち切る」

`0x3810` が申告する全体長まで開始オフセットを進めて繰り返すが、
**受信量が要求より少なければ打ち切る**（無限ループにしない）。
実機で 163,840 文字（64KB 境界を 2 回跨ぐ）を正しく結合できることを確認した。

## D6. セルごとの往復を最適化しない（今は）

LOB セルの数だけ往復する。100 行なら 100 往復で、実機の 4〜7 秒/往復では重い。

**まとめて取る要求形式が原典にあるか未確認**なので、
「無いと決めつけて自前で工夫する」ことはしなかった。
既定では取りに行かないため既定利用では 0 往復である点も踏まえ、backlog に積んだ。
