# 要件: 接続設定のサーバー一元管理

## 背景 / 課題
現在、接続設定は 2 系統に分かれている。

- **サーバープロファイル**（`--profiles <file>` の `profiles.json`）: 全ユーザー共通・読み取り専用。
  運用者が管理し、信頼設定（`printer.autoPdfDir` / `autoPrint` / `pdfFontPath` 等）や自動サインオン
  （`passwordEnv`）を持てる。
- **ブラウザ保存接続**（`stores/settings.ts`, localStorage キー `as400.connections`）: この端末の
  このブラウザだけに存在。ユーザーが CRUD できるが、端末をまたいで共有されず、自動サインオンの
  資格情報が各端末に平文で散在する。

この二重管理により、(1) 同じユーザーでも端末が変わると接続が引き継がれない、(2)「サーバー設定 vs
ブラウザ設定」という利用者に分かりにくい区別が生じる、(3) 資格情報が端末ローカルに平文で残る、という
問題がある。

## 目的 / ゴール
ユーザーが作成・編集する接続設定を**サーバー側に一元管理**し、どの端末からでも同じ設定を使えるようにする。
表示範囲を認証状態に連動させる:

- **ユーザー管理なし（認証オフ）**: サーバーに保存された全接続設定を表示する。
- **ユーザー管理あり（認証オン）**: ログインユーザー自身の接続設定のみ表示する（既存の per-user
  `owner` 分離モデルに一致させる）。

資格情報は**平文でサーバーに保存しない**（axis: `passwordEnv` 方式に統一）。

## スコープ
### 対象
- サーバー側の**ユーザー接続設定ストア**（永続化。`users.json` と同様の atomic 保存を想定）。
- 接続設定の CRUD API（一覧 / 作成 / 更新 / 削除）。認証オン時は `owner` スコープで自分の設定のみ操作可。
- web-ui の接続画面を localStorage 依存から**サーバー API 経由**に切り替える（`stores/settings.ts` の置換）。
- 「共有プロファイル（読み取り専用）」と「自分の接続（編集可）」の 2 系統を UI で提示する
  （先の変更で入れた種別チップ／カード・一覧トグルはこの新モデル上に載せ替える）。

### 対象外
- 運用者管理の `profiles.json` の廃止（信頼設定を担う共有プロファイルとして**存続**させる。Q1 決定済み）。
- 管理者画面からの他ユーザー接続の編集（閲覧/管理まで踏み込むかは未確定 Q6。まずは本人 CRUD を主とする）。

## 機能要件
- ユーザーが接続設定（名称・ホスト・ポート・CCSID・画面サイズ・デバイス名・TLS・セッション種別）を
  作成・編集・削除でき、サーバーに永続化される。
- **認証オフ時は全接続を誰でも CRUD 可**（単一の暗黙ユーザー扱い。Q2 決定済み）。**認証オン時は自分の接続のみ**を
  一覧・操作・接続に使える。
- ユーザー作成接続も**自動サインオンを持てる**が、資格情報は**ユーザー別に分離された `passwordEnv`**（名前付き
  シークレット）で解決する。ユーザーは**自分の `passwordEnv` のみ**を作成・参照でき、**他ユーザーやサーバーの
  任意環境変数を参照できない**（Q3 決定済み。分離機構の詳細は Q5→research）。
- 接続設定に**信頼設定フィールド（printer 出力系）を含めない**（後述の非機能・安全要件）。
- 既存の共有プロファイル（`profiles.json`）は従来どおり一覧に表示され、接続に使える（読み取り専用）。

## 非機能要件 / 制約
- **信頼境界の維持（必須）**: `printer.autoPdfDir` / `autoPrint` / `pdfFontPath` などサーバー上の
  パス書き込み・`lp` 実行につながる設定は、**運用者管理の `profiles.json` のみ**が持てる。ユーザー作成の
  接続設定スキーマにこれらを含めてはならない（任意パス書き込み・コマンド実行の防止。現行の安全不変条件）。
- **資格情報を平文でサーバーに保存しない**: 自動サインオンは `passwordEnv`（名前付きシークレット）による解決に
  統一する。接続レコード自体にパスワード平文を持たない。
- **passwordEnv のユーザー別分離（必須）**: ユーザーが指定できる `passwordEnv` は**自分専用の名前空間**に限定し、
  他ユーザーのシークレットやサーバープロセスの任意環境変数を読み出せない設計とする（環境変数参照口としての悪用防止）。
- 依存追加なし（Node 標準・既存スタックの範囲で実装）。
- 既存の認証オン／オフ切り替え、`owner` 分離、監査ログの枠組みと整合させる。

## 完了条件 (受け入れ基準)
- [ ] ユーザーが作成した接続設定が localStorage ではなくサーバーに保存され、別ブラウザ／別端末から
      ログインしても同じ設定が見える（認証オン時）。
- [ ] 認証オフ時はサーバー保存の全接続が表示され、認証オン時は自分の接続のみが表示される。
- [ ] ユーザー作成の接続設定スキーマに printer 出力系の信頼フィールドが**存在しない**（型・API 双方で拒否）。
- [ ] 自動サインオンのパスワードがサーバー上に平文で保存されない（`passwordEnv` 経由のみ）。
- [ ] ユーザーは自分の `passwordEnv` のみ参照でき、他ユーザーのシークレットやサーバーの任意環境変数を
      参照できない（分離が型・実装で保証される）。
- [ ] 既存の共有プロファイル（`profiles.json`）は引き続き読み取り専用で一覧・接続に使える。
- [ ] localStorage への接続設定の読み書きが web-ui から無くなる（設定の単一の真実がサーバーになる）。

## 決定事項（requirement ゲートで確定）
- **Q1（profiles.json との住み分け）**: **存続**。共有・読み取り専用・信頼設定の受け皿として残し、ユーザー作成分は
  別ストアにする。
- **Q2（認証オフ時の書き込み権限）**: **誰でも全接続を CRUD 可**（認証オフ＝単一の暗黙ユーザー扱い、ローカル用途前提）。
- **Q3（自動サインオンの扱い）**: ユーザー作成接続も**自動サインオン可**。ただし `passwordEnv` は**ユーザーごとに作成・
  分離**し、他ユーザー／サーバーの任意環境変数を参照できないようにする（名前空間分離）。平文は接続レコードに持たない。
- **Q4（既存 localStorage 接続の移行）**: **破棄**（移行処理は作らない。localStorage の接続読み書きを廃止する）。

## 未確定事項 / 確認したいこと（research/design で解消）
- **Q5（passwordEnv 分離機構）**【要 research】: ユーザー別 `passwordEnv` をどう実装・保存するか。候補:
  (a) 命名規約付きの環境変数（例 `AS400_SECRET_<uid>_<name>`、運用者が provision）、
  (b) ユーザー別のシークレットストア（サーバー保存。平文回避のため暗号化 or OS keyring 等）、
  (c) 実行時のみメモリ保持（都度入力＋セッション内キャッシュ）。
  「平文をサーバーに残さない」「他ユーザー・任意 env を読めない」を満たす方式を research で確定する。
- **Q5b（保存先とキー設計）**【要 research/design】: 接続ストアのファイル形式（`connections.json` 等、`users.json`
  同様の atomic 保存）、接続 ID・`owner` の持ち方、認証オフ時の owner 値の扱い。
- **Q6（管理者画面連携）**: 管理者がユーザーの接続を閲覧・管理できるようにするか。まずは本人 CRUD のみとし、
  管理者連携は将来の別作業に回す方向（本作業のスコープ外候補）。
