# 要件: スプールファイルの一覧・取得（pull 型）

## 背景 / 課題

backlog `hostserver.md` の「スプールの一覧・取得」。

**現状との違いが要点。** 今の `PrinterSession` は AS/400 が仮想プリンターに SCS を送ってくる
**push 型**で、次の制約がある:

1. プリンターセッションを開いておく必要がある
2. その装置に出力を向ける必要がある
3. **過去に出たスプールは取れない**

本作業は既存スプールを任意の OUTQ から検索・取得する **pull 型**を追加する。
「過去のスプールを後から PDF 化する」が可能になる。
**既存実装を置き換えるのではなく、別の入り口として補完する。**

## 目的 / ゴール

**既存のスプールファイルを一覧し、その中身をテキストとして取得できる**状態にする。

## スコープ

### 対象

- スプールファイルの**一覧**（OUTQ・ユーザー・状態などで絞り込み）
- スプールファイルの**中身**（1 行 1 レコードのテキスト）の取得
- **一覧は QGY 経路**（`QGYOLSPL`。コマンドサーバー経由）
- **中身はネットワーク印刷サーバー**（`0xE003` / 8474・9474）の OPEN→READ→CLOSE
- SQL でも同じことができる事実は**ドキュメントに残す**（採用はしないが、代替手段として記録）
- `packages/core` のライブラリとして実装し、CLI で検証できること

### 対象外

- PDF 化（既存 `pdf.ts` は `LogicalPage` を受け取る。接続は別作業）
- スプールの削除・保留・解放などの操作（読み取りのみ）
- MCP ツール公開・Web UI（別作業）
- SCS の解釈（取得するのはテキスト。既存 `ScsDecoder` の領分とは別）

## 機能要件

- ユーザー名・OUTQ・状態でスプールを絞り込んで一覧できる
- 一覧から得た識別子（ジョブ名・ファイル名・ファイル番号）で中身を取得できる
- 一覧の項目: ファイル名・ユーザー・OUTQ・状態・ページ数・作成日時・ジョブ名・ファイル番号
- **2 経路を持ち、利用側が明示的に選べる**。既定は自動選択
- 件数が多い場合に全件をメモリへ載せない選択肢がある
- 通信内容をトレースできる。資格情報を平文で出さない

## 非機能要件 / 制約

- 既存の `PrinterSession`（push 型）に影響を与えない
- ピュア層は Node API 非依存。**`Buffer` 等のグローバルも使わない**
- JTOpen は参照のみ。参照コメントを残す
- 実機（PUB400）での検証を伴う

## 完了条件 (受け入れ基準)

- [ ] QGY 経路でスプールを一覧できる（実機で確認）
- [ ] ネットワーク印刷サーバーでスプールの中身を取得できる（実機で確認）
- [ ] 取得した中身が、同じスプールを SQL で読んだ結果と一致することを確認する
      （SQL は採用経路ではないが、**正しさの照合に使う**）
- [ ] 一覧で得た識別子をそのまま中身取得に渡せる（2 つの経路が繋がる）
- [ ] 存在しないスプールを指定したときのエラーが原因の分かる形で返る
- [ ] 解析の単体テストがある（実機に依存しない）
- [ ] トレースに資格情報が平文で出ない
- [ ] jt400 との対応がコード上で追える（QGY 経路）

## 未確定事項 / 確認したいこと

- QGY 経路の API 仕様（`QGYOLSPL` のパラメータ、`OSPL0300` 形式のレコード配置）
- QGY 経路で PUB400 の権限が足りるか（他人のスプールは見えない見込み）
- SQL 経路が使えない条件（古い IBM i でカタログビューが無い場合）
- 自動選択の判定方法（SQL を試して失敗したら QGY か、事前に判定するか）
- 中身取得を QGY 経路でも実装するか（`QGYOLSPL` は一覧のみ。中身は別 API）

> 補足: 一覧・中身とも **SQL 経路は research 前に実機で取得成功を確認済み**。
> QGY 経路が未知数なので、そちらを重点的に調べる。
