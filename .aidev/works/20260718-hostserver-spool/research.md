# 調査: スプールファイルの一覧・取得

## 調査の問い

- Q1: SQL 経路で一覧・中身が取れるか？
- Q2: QGY 経路（`QGYOLSPL`）の API 仕様は？
- Q3: PUB400 で QGY の権限が足りるか？
- Q4: 両経路の結果は一致するか？

## 判明した事実

### F1: SQL 経路で一覧・中身とも取得できる（実機で確認）

**一覧**——`QSYS2.OUTPUT_QUEUE_ENTRIES_BASIC`:

```
SPOOLED_FILE_NAME = "QPJOBLOG"  USER_NAME = "MARO"  STATUS = "READY"
TOTAL_PAGES = 23  CREATE_TIMESTAMP = "2026-07-18-16.46.05.713830"
JOB_NAME = "672961/MARO/WEBEMU01"  FILE_NUMBER = 2
```

利用可能な列: `OUTPUT_QUEUE_NAME` / `OUTPUT_QUEUE_LIBRARY_NAME` / `CREATE_TIMESTAMP` /
`SPOOLED_FILE_NAME` / `USER_NAME` / `USER_DATA` / `STATUS` / `SIZE` / `TOTAL_PAGES` /
`COPIES` / `FORM_TYPE` / `JOB_NAME` / `DEVICE_TYPE` / `OUTPUT_PRIORITY` / `FILE_NUMBER` / `SYSTEM`

**中身**——`SYSTOOLS.SPOOLED_FILE_DATA` 表関数:

```sql
SELECT ORDINAL_POSITION, SPOOLED_DATA
  FROM TABLE(SYSTOOLS.SPOOLED_FILE_DATA(
    JOB_NAME => '672961/MARO/WEBEMU01',
    SPOOLED_FILE_NAME => 'QPJOBLOG',
    SPOOLED_FILE_NUMBER => 2))
```

1 行 1 レコードのテキストが返る（実機でジョブログの中身を取得成功）。

**既に動く SQL 経路だけで完結する**。新規のプロトコル実装は不要。

### F2: QGY 経路も動く（実機で確認）。ただし仕様は原典と読み違えやすい

`QGYOLSPL`（QGY ライブラリ）をコマンドサーバー経由で呼び出して成功した:

```
総件数=3 返却件数=3 レコード長=136
[0] job="WEBEMU01" user="MARO" nbr="627205" file="QPDSPJOB"
[1] job="WEBEMU01" user="MARO" nbr="672961" file="QPJOBLOG"
[2] job="WEBEMU01" user="MARO" nbr="678307" file="QPJOBLOG"
```

パラメータ（10 個）:

| # | 種別 | 内容 |
|---|---|---|
| 0 | out | 受信変数（レコードが並ぶ） |
| 1 | in | 受信変数の長さ（4） |
| 2 | out | リスト情報（80）。総件数・返却件数・ハンドル・レコード長 |
| 3 | in | 返却件数（4） |
| 4 | in | ソート情報（4。0 で無指定） |
| 5 | in | **フィルタ情報**（下記） |
| 6 | in | 修飾ジョブ名（26）。**空白＝すべて**（`*ALL` は不可） |
| 7 | in | 形式名（8）。`OSPL0300` |
| 8 | inout | エラーコード（4） |
| 9 | in | フィルタ形式名（8）。`OSPF0100` |

**OSPF0100 フィルタは「オフセット表」ではなく「件数と配列が交互に並ぶ連続配置」**:

```
ユーザー数(4) + [名前(10)+予約(2)] × n
OUTQ 数(4)    + [名前(10)+ライブラリ(10)] × n
用紙タイプ(10)
ユーザーデータ(10)
状態数(4)     + [状態(10)+予約(2)] × n
装置数(4)     + [装置(10)+予約(2)] × n
```

**各配列は最低 1 件必要**。0 件だと `GUI0011 "0 is not valid for number of user names."` /
`GUI0012 "... output queue names."` で弾かれる。絞り込まない場合は `*ALL` を 1 件入れる。

リスト情報（80 バイト）: 総件数(0) / 返却件数(4) / ハンドル(8) / レコード長(12)。

レコード（OSPL0300・136 バイト）の先頭: ジョブ名(0-10) / ユーザー(10-20) /
ジョブ番号(20-26) / ファイル名(26-36)。

### F3: 両経路の結果が一致する（実機で確認）

同じ 3 件（`QPDSPJOB` / `QPJOBLOG` × 2）が両経路で返った。

### F4: PUB400 の権限では自分のスプールのみ見える

どちらの経路でも他ユーザーのスプールは返らない。MARO の特殊権限が `*NONE` のため。
**これは制約であって不具合ではない**——実運用でも通常は自分の分だけ見えれば足りる。

### F5【手順の反省】推測でバイト配置を組んで 5 往復無駄にした

QGY のフィルタ構造を**原典を読まずに推測で組み**、次の順で失敗を繰り返した:

```
1. オフセット表と仮定 → GUI0011「ユーザー名 0 件は無効」
2. OSPF0200 のオフセット表 → CPF337C「106〜120 は 0 でなければならない」
3. 開始位置を 120 に → CPF337C「106〜130 は…」（範囲が動くだけ）
4. 開始位置を 106 に → CPF337C「106〜116 は…」
5. 件数を 0 に → GUI0011 に戻る
→ ここでようやく原典を読み、【連続配置】と判明して一発で成功
```

AGENTS.md に「**原典をまず確認する**」と自分で追記した直後に、これを守らなかった。
**エラーメッセージから構造を逆算しようとしたのが誤り**で、最初に 1 回読めば済んだ。

## 影響範囲

- 新規モジュールとして追加。既存の `PrinterSession`（push 型）には触れない
- SQL 経路は既存の `hostserver/db/` をそのまま使う
- QGY 経路は前作業の `CommandConnection.call()` に載る
- **前作業で見つけた欠陥がここで顕在化した**——`call()` の戻りコード判定が
  コマンド実行と同じ規則になっており、`QGYOLSPL` が返す `0x0501` で例外になった。
  前作業のブランチ上で修正済み（コマンド実行は 0/0x0400 のみ、呼び出しは 0 以外すべて失敗）

## 実現性 / リスク

**両経路とも実証済み。**

### リスク 1: QGY のレコード配置（OSPL0300・136 バイト）は先頭 36 バイトしか確認していない

ジョブ名・ユーザー・番号・ファイル名までは実機で読めた。
状態・ページ数・作成日時などの位置は**原典を読んで確定させる**（F5 の反省を踏まえ、推測しない）。

### リスク 2: QGY 経路の「中身取得」は別 API

`QGYOLSPL` は一覧のみ。中身は別の API（`QSPGETSP` 等）が要る。
**SQL 経路には中身取得があるため、QGY 経路は一覧のみに留める**のが妥当。

### リスク 3: 自動選択の判定

SQL 経路はカタログビューの有無に依存する。古い IBM i では無い。
「SQL を試して失敗したら QGY」とするか、事前に判定するかを spec で決める。

## spec への申し送り

- **SQL 経路を主、QGY 経路を代替**とする（ユーザー判断: 両方実装）
- **中身取得は SQL 経路のみ**（QGY は別 API が要るうえ、SQL で足りる）
- QGY のレコード配置は**原典を読んで確定**する（F5 の反省）
- フィルタは各配列に最低 1 件。絞り込まない場合は `*ALL`
- 修飾ジョブ名は**空白**（`*ALL` は `CPF3342` で弾かれる）
- 自動選択の判定方法を決める
- 両経路が同じ結果を返すことを実機で確認する（受け入れ基準）
