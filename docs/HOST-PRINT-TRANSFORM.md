# HOST-PRINT-TRANSFORM.md — 印刷経路の実測記録

日本語（DBCS）帳票をプリンターセッションで扱えなかった件の調査記録。**同じ道を再び辿らないため**に、
実測で分かったことと、試して駄目だったことを残す。すべて日本語実機（IBM i 7.3・CCSID 939）での実測。

関連実装: `packages/server/src/spool-rescue.ts` / `packages/core/src/session/printer-session.ts` /
`packages/core/src/protocol/scs.ts`

---

## 1. 発端と根本原因

IGC（DBCS）属性を持つ帳票をプリンターセッションへ送ると、ホストが応答を待ったまま止まる。

```
CPA3303  ファイル DSPFMT の属性は装置 PRT_ASAO でサポートされていない
         エラー・コード 5 ＝ ユニコード/IGC データが装置でサポートされていない
```

仮想プリンター装置は `TYPE(3812) MODEL(1)` の SBCS プリンターとして作られる（`DSPDEVD` で確認）。
IGC を扱えないので書き出しプログラムが処理できない。**待ち行列の先頭で詰まると後続もすべて止まる**——
「DSPFD の結果が送信できない」という症状はこれが原因だった。

---

## 2. 効かなかった対処

| 試したこと | 結果 |
|---|---|
| 端末タイプを `IBM-5553-B01`（DBCS プリンター）で申告 | **8925** 装置作成失敗 |
| ↑ に `IBMIGCFEAT`（`2424J0` / `3232J0` / `2424J1`）を追加 | 同じく 8925 |
| `IBM-5553-B02` / `IBM-5553-01` | ネゴシエーションで拒否 |
| 装置を `TYPE(5553) MODEL(B01) IGCFEAT(2424J1)` で**事前作成**し、その装置名で接続 | 接続でき装置属性も保たれるが、**CPA3303 は変わらず** |

事前作成した装置には接続できる（装置は 5553/IGCFEAT のまま維持される）が、それでも書き出せない。

---

## 3. 効いた対処

### 3.1 取得経路（pull）は装置の制約を受けない

ネットワーク印刷サーバー経由の取得は書き出しプログラムを通さないため、同じ帳票を問題なく読める。
これを使って詰まったスプールを拾い上げる（`spool-rescue.ts`）。

**ただし当初は 0 ページだった**——`0x2B 0xFD`（IGC 制御オーダー）を未知として打ち切っていたため。
構造は `0x2B 0xD2` と同じ長さ前置。英数字だけの帳票には現れないので長く気づかれなかった。
修正後は 2 ページ・日本語 52 行が桁揃えのまま読める。

### 3.2 書き出し経路（push）は HPT でなら通る

装置を `TRANSFORM(*YES) MFRTYPMDL(*HP4)` にすると CPA3303 が出ず、帳票が届く。
ただし**届くのは SCS ではなく PCL** になるので、当アプリでは表示・PDF ができない。
セッション設定「印刷の経路」で選べるようにしてある。

## 4. HPT の出力（実測）

`*HP4` で日本語帳票を変換させた結果。

```
1b 45                        ESC E        リセット
1b 28 73 30 70 31 37 2e 31 30 68 …54      ESC (s…T    フォント選択（固定ピッチ 17.1cpi）
1b 26 61 2b 30 30 32 31 36 48             ESC &a+00216H  水平位置
```

| 項目 | 実測 |
|---|---|
| データストリーム | 標準の **PCL5**（`ESC E` / `ESC (s…T` / `ESC &…`） |
| ラスタ（ビットマップ） | **無い**（`ESC *r` / `ESC *b` が 0 個） |
| フォントダウンロード | **無い**（`ESC )s` が 0 個） |
| シンボルセット指定 | **無い**（`ESC ( <id>` が出ない） |
| 日本語 | **2 バイトの生バイト**で埋め込まれる |

### 4.1 日本語の符号は不明

置換ではなく実在の符号である（同じ文字は同じバイト・違う文字は違うバイト）。

| 文字 | 観測 | Shift-JIS | EUC-JP | JIS | 939 DBCS |
|---|---|---|---|---|---|
| 浅 | `c6 ec` | 90 f3 | c0 f5 | 40 75 | 46 8d |
| 雄 | `a0 9b` | 97 59 | cd ba | 4d 3a | 45 70 |
| 野 | `a0 a5` | 96 ec | cc ee | 4c 6e | 45 69 |
| 田 | `a0 8e` | 93 63 | c5 c4 | 45 44 | 45 63 |
| レ | `84 aa` | 83 8c | a5 ec | 25 6c | 43 ba |

**Python の全コーデックで総当たりしても一致無し。** 並び順も JIS 系ともホスト DBCS とも一致しない
（田 < 雄 < 野 になり、読み順・ホスト順のどちらとも違う）。

## 5. 符号表を探した結果（すべて空振り）

| 試したこと | 結果 |
|---|---|
| IBM Documentation を直接参照 | **HTTP 403**（このネットワークからは読めない） |
| `RTVWSCST` で `*HP4` のテーブルを取得 | **取得できた**（441 行・可読）。ただし **DBCS 定義が無い**——タグ 58 種に DBCS/IGC/漢字系はゼロ、EBCDIC→ASCII 対応 10 組はすべて Latin（1140-1149→1252）、コードページも 437/850/852/1250/1252/1254 のみ |
| `RTVWSCST WSCST(QSYS/QCTXPDFMBD)` | **CPD0043**——`RTVWSCST` は IBM 提供の機種テーブル専用。既存 WSCST オブジェクトは読めない |
| 日本語プリンター機種のテーブル取得（`*IBM5577` / `*5579` / `*5587` / `*ESCP` / `*NECP`） | **この機には 1 つも無い**（取得も装置作成も不可） |
| `MFRTYPMDL(*WSCST)` で自作 | **この機では無効値** |

取得できた機種は `*IBM3812` / `*HP4` / `*IBM4019` のみで、すべて Latin。

**結論: 日本語 2 バイト符号の対応表を得る手段が無い。したがって PCL を解釈しても日本語は読めない。**

## 6. 取得時変換（`OPEN_MODIFIED_SPLF`）も使えない

表示（SCS）と本来の印刷（PCL）を両立させるため、取得時にホストへ変換させる方式
（JTOpen の `PrintObjectTransformedInputStream` 相当）を実装して試した。

- 要求: action `0x001A` ＋ スプール ID ＋ 選択コードポイント（`ATTR_MFGTYPE=0x0041`）
- 結果: **`rc=0x0009` → `CPF6DFB`「変換中にエラーが発生しました」**

| 条件 | 結果 |
|---|---|
| `*HP4` / `*HP5` / `*IBM4019` | CPF6DFB |
| 変換先コードページ指定（CP850 / CP932 / CP437） | CPF6DFB |
| 日本語帳票・英数字だけの単純な帳票 | どちらも CPF6DFB |
| `*WSCST` | 別エラー（CPF6DF7）＝**値はホストに届いている** |

Transform Services（5770TS1）は導入済み。要求の組み立ては通っており、変換処理そのものが失敗している。
**原因は特定できていない。** ここが解ければ表示と本来の印刷を両立できるので、心当たりが出たら再挑戦の余地がある。

実装は残していない（動かないコードを抱えないため）。再挑戦するならこの節の定数から組み直す。

## 7. 現状の落とし所

| したいこと | 経路 |
|---|---|
| 画面で読む・PDF にする | スプール機能、またはプリンターセッションの既定モード（SCS） |
| 実プリンターへ本来の書式で出す | プリンターセッションの「ホスト変換」モード（PCL をそのまま `lp -o raw`） |
| 詰まった帳票を救う | 自動（`spool-rescue.ts`）。取得後は保留か削除を設定で選ぶ |

**PCL の翻訳が要るのは「ホスト変換モードで受けた帳票も画面で見たい」場合だけで、
それはスプール機能から取得すれば済む。** 追う価値は低い。

## 8. 途中で見つけた実バグ（すべて修正済み）

この調査は、プリンター機能の実バグを掘り当てる結果になった。

| 症状 | 原因 |
|---|---|
| 書き出しメッセージに応答できない | MSGREPLY を **NUL 終端**で送っていなかった（JTOpen は `new byte[len+1]`） |
| メッセージを見分けられない | 受信した属性値の **末尾 NUL** を落としていなかった（`trimEnd()` は空白しか落とさない） |
| **日本語帳票がスプール機能でも 0 ページ** | SCS の **`0x2BFD`（IGC 制御）** 未対応で打ち切っていた |
| 救出した帳票が画面に出ない | ws-handler が `session.on("report")` だけを見ていた（救出分はそのイベントに乗らない） |
| エラーの原因が分からない | 応答に載っている **CPF メッセージ**を捨てていた |
